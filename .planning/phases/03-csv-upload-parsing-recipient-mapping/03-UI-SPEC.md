# Phase 3 — UI Design Contract

> Visual and interaction contract for CSV Upload + Parsing + Recipient Mapping. Generated by gsd-ui-researcher (autonomous mode), verified by gsd-ui-checker.

This phase adds ONE new authenticated surface — a **Recipients** page (`app/(app)/recipients/`, per 03-RESEARCH.md A3) — for uploading a CSV, reviewing the parse summary, confirming/overriding the email column, and persisting a per-user recipient set. It inherits the design system, spacing scale, typography, and color contract established and approved in **02-UI-SPEC.md** verbatim. Consistency with the Phase-2 SMTP wizard (react-hook-form + zod, field-anchored errors, Card-based layout, sonner toasts, `text-[28px]` page headings) is the primary design constraint. Only phase-specific additions (copy, new screen states, the email-column control) are newly declared below.

## Color

All values are existing `app/globals.css` tokens (oklch). Reference by semantic name, never raw values.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--background` oklch(1 0 0) | Page background, main content surface |
| Secondary (30%) | `--card` / `--muted` / `--border` / `--sidebar` | Upload card, summary card, preview-table header row, separators, sidebar |
| Accent (10%) | `--primary` oklch(0.205 0 0) | See reserved list below |
| Destructive | `--destructive` oklch(0.577 0.245 27.325) | Blocking upload errors ONLY (Alert `destructive` variant + field-anchored validation messages). See below on why invalid-row COUNT is NOT destructive |
| Success (semantic, minimal) | `--success` oklch(0.627 0.17 149) — existing token, do NOT redeclare | ONLY: the "all rows valid" positive summary indicator (CheckCircle2 icon + text) and the save-success toast icon |

Accent (`--primary`) reserved for:
1. The single primary CTA per screen: "Upload CSV" (empty state / upload card) and "Save recipient list" (confirm step). One accent button visible per view.
2. Active sidebar nav indicator when "Recipients" is the current route (shadcn `isActive` on `SidebarMenuButton` — the only nav accent use).
3. Focus rings (via `--ring`) on the file input, email-column select, and buttons.

Everything else uses `secondary` / `ghost` / `outline` variants: "Choose a different file", "Cancel", the "Edit SMTP settings"-style secondary actions, and the file-picker trigger before a file is chosen. Never style all interactive elements with accent.

**Invalid-row count is informational, not an error.** Some rows lacking a valid email is an expected, non-blocking outcome (those rows are simply skipped at send time). Render the invalid-count as neutral text (`text-muted-foreground`) with a lucide `AlertCircle` — NOT the destructive token and NOT a destructive Alert. Reserve destructive strictly for uploads that cannot proceed (wrong file type, too large, too many rows, unreadable/empty CSV).

## Screen States

Every state must be implemented by the executor and checked by the auditor.

| Screen / Area | States |
|--------|--------|
| Recipients page (no sets, no file) | Empty-state callout dominant: "Upload your first recipient list" + "Upload CSV" CTA |
| Recipients page (existing sets) | List of saved recipient sets (newest first) + the upload card available to add another |
| Upload — default | File input + help text; CTA disabled until a file is selected |
| Upload — client validation error | Field-anchored destructive message (wrong type / too large / too many rows) — clears when the user picks a new file |
| Upload — parsing in-flight | Controls disabled, button "Reading your file…" + spinner |
| Upload — parse error (server) | Destructive `Alert` (unreadable / empty / no header), file cleared; user re-picks |
| Review — email column detected | Summary counts + `Select` prefilled to the detected column + "detected {column}" help + invalid/valid count line + sample table + "Save recipient list" |
| Review — email column NOT detected | `Select` unset (placeholder "Choose the email column"); save disabled until a column is chosen; not-detected help shown |
| Review — override changed | When the user picks a different column, the invalid/valid count re-computes against the newly selected column before save |
| Save — in-flight | "Saving…" + spinner, controls disabled, no double-submit |
| Save — success | Success toast; page returns to the saved-sets list with the new set at top |
| Save — failure | Destructive `Alert`; settings not lost; user can retry |
| All page states | Async fetch of the caller's existing sets uses `Skeleton` (shell pattern) — never layout shift or a blank flash |

Interaction rules:
- Upload and save buttons disable while in flight; no double-submit (mirrors the wizard verify/test-send rule and guards CSV-05 against a duplicate insert).
- Errors never clear silently — a blocking error clears when the user selects a new file; the email-column requirement clears when a valid column is chosen.
- The email-column choice is a hint the human confirms (CSV-03): auto-detection pre-selects, the user can always override, and save persists the CONFIRMED column — never an unconfirmed guess.
- The parse/preview step never persists bytes; the file is written to disk only in the save action, after validation passes (03-RESEARCH.md Pitfall 5 — orphan-file avoidance).
- CSV cell values are shown in the sample preview only; the password/credential redaction concerns of Phase 2 do not apply here, but no CSV cell value is logged.
