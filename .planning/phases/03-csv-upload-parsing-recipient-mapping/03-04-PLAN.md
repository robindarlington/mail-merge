<objective>
Ship the user-facing slice that turns the tested backend into a real capability: a `/recipients` page where a signed-in user uploads a CSV, reviews the parse summary, confirms or overrides the auto-detected email column, and saves a recipient set. This is the last vertical layer of the phase's slice — after it, a real user can do the whole thing in a browser. It reuses the Phase-2 SMTP-wizard UI seam verbatim (react-hook-form + zod, Card layout, field-anchored errors, sonner toasts, `text-[28px]` headings) and adds only the two shadcn components the review UI needs (`select`, `table`).

Purpose: CSV-01 (browser upload UI), CSV-03 (email-column confirm/override control), CSV-04 (invalid-count display that recomputes on override via the server-supplied per-column map), CSV-05 (saved-set list + save trigger).
Output: the `/recipients` route, the `csv-uploader` client component, the two shadcn components, and the sidebar nav slot.
</objective>

## Phase Goal

**As a** signed-in user, **I want to** upload a CSV and confirm which column holds the email addresses, **so that** I get a validated recipient set my campaigns can draw merge fields from.

- `import { parseUploadedCsv, saveRecipientSet } from "@/lib/csv/actions"` (03-03) — both take a `FormData`; ParseResult = `{ ok:true; data:{ columns, detectedEmailColumn, rowCount, invalidCount, invalidCounts } } | { ok:false; error: ActionError }` where `invalidCounts` is a `Record<string, number>` with a server-computed invalid-email count for EVERY column; SaveResult = `{ ok:true; data:{ rowCount, filename, invalidCount } } | { ok:false; error: ActionError }`. Invalid-count on override (CSV-04): The invalid/valid count line is driven by `parseResult.data.invalidCounts[selectedColumn]` — a server-computed count over the WHOLE file for the currently-selected column. When the user changes the Select, look up the new column's entry in that map; do NOT re-parse the file client-side and do NOT derive the count from the sample rows. This keeps papaparse off the client bundle and keeps the server the source of truth. On Save, the returned `SaveResult.data.invalidCount` (also server-computed on the confirmed column) is the authoritative persisted value.

Sample-row preview (cosmetic only): `parseUploadedCsv` returns summary counts + `invalidCounts`, not the parsed rows. The sample TABLE is a visual aid only — render the columns as the table header and up to 5 sample rows read client-side from the chosen File text for the caption "Showing the first {n} rows." These 5 rows feed the preview table ONLY; they never drive the invalid count (which comes from `invalidCounts` above). Do not send bytes twice for preview; the file is re-sent only on Save.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser file input → Server Action FormData | Untrusted; the action (03-03) re-validates and re-derives userId — the client is a convenience layer, not a trust anchor |
| RSC page load → DAL | `auth()`-derived userId scopes the list; an unauthenticated load yields an empty list, never another tenant's sets |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-IDOR | Information disclosure | recipients/page.tsx | mitigate | The RSC re-derives `userId` via `auth()` and lists only via `listRecipientSetsForUser(userId)`; no client-supplied id path (AUTH-02). Enforcement lives server-side (03-02/03-03); the UI cannot widen scope. |
| T-3-XSS | Tampering | csv-uploader.tsx sample preview | mitigate | CSV cell values render as React text children (auto-escaped), never via dangerouslySetInnerHTML; no CSV value is placed into HTML/attribute sinks. |
| T-3-DBLSUBMIT | Tampering | Save button | mitigate | The Save button disables while its action is in flight (mirrors the wizard rule), guarding CSV-05 against a duplicate insert from a double click. |
| T-3-COUNT | Tampering / Integrity | invalid-count on override | mitigate | The displayed invalid count for any chosen column comes from the server-computed `invalidCounts` map (03-03), not a client parse of 5 sample rows — the count the user confirms is the count the server persists. |
| T-3-SC | Tampering (supply chain) | shadcn add select table | accept | `select`/`table` are official shadcn code scaffolds (`registries: {}`), copied into the repo — not npm runtime installs; no registry-legitimacy gate applies. |
</threat_model>
