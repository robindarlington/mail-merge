<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Server Action caller-derived userId → DAL | The DAL trusts only the userId its caller passes (03-03 re-derives it via Clerk `auth()`); every query filters on it |
| stored recipient_sets rows → cross-tenant read attempt | An id supplied by one user must never surface another user's row |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-IDOR | Information disclosure / Elevation of privilege | getRecipientSetForUser / listRecipientSetsForUser | mitigate | Every function is userId-first and filters on it; `getRecipientSetForUser` uses `and(eq(id), eq(userId))` — no fetch-by-id-alone path exists. Two-tenant test asserts USER_B cannot read USER_A's set (AUTH-02, ASVS V4). |
| T-3-TAMPER-OWNER | Tampering | createRecipientSet | mitigate | userId is server-injected via `{ userId, ...values }`; the `Pick<>` value type omits userId so a caller cannot spoof ownership through the values object. |
</threat_model>
