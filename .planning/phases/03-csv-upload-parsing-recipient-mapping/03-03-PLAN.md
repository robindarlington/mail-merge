- `type ParseSummary = { columns: string[]; detectedEmailColumn: string | null; rowCount: number; invalidCount: number; invalidCounts: Record<string, number> }`. `invalidCounts` has ONE entry per column (`countInvalidEmails(rows, col)` for every `col` in `columns`); `invalidCount` is the detected column's count (`detectedEmailColumn ? invalidCounts[detectedEmailColumn] : 0`). The per-column map is what lets the UI (03-04) display the invalid count for a user-overridden column without shipping papaparse to the browser or re-parsing.
- `type ActionError = { kind: "unauthenticated" } | { kind: "validation"; issues: unknown } | { kind: "wrong_type" } | { kind: "too_large" } | { kind: "too_many_rows" } | { kind: "parse_error" } | { kind: "empty" } | { kind: "unknown"; raw: string }`.
- `type ParseResult = { ok: true; data: ParseSummary } | { ok: false; error: ActionError }`.
- `type SaveResult = { ok: true; data: { rowCount: number; filename: string; invalidCount: number } } | { ok: false; error: ActionError }`. `invalidCount` is computed on the CONFIRMED column so the save return value is authoritative for the persisted set.
- `parseUploadedCsvCore(userId: string, formData: FormData): Promise<ParseResult>` â€” guard file (wrong_type/too_large) â†’ parse â†’ if parseErrors non-empty return parse_error â†’ if columns empty return empty â†’ if `rows.length > MAX_ROWS` return too_many_rows â†’ detect â†’ build `invalidCounts` by running `countInvalidEmails(rows, col)` for every column â†’ `invalidCount` = detected column's entry (or 0) â†’ return summary. NEVER writes to disk.
- `saveRecipientSetCore(userId: string, formData: FormData): Promise<SaveResult>` â€” read the confirmed `emailColumn` from FormData, `confirmColumnSchema.safeParse` it, re-guard + re-parse the re-sent file, enforce MAX_ROWS (return too_many_rows), verify the confirmed column is one of `columns` (validation), re-count invalid on the CONFIRMED column via `countInvalidEmails`, `writeUpload(bytes)`, then `createRecipientSet(userId, { filename: file.name, columns_json: JSON.stringify(columns), row_count, storage_path })`, and return `{ ok:true, data:{ rowCount, filename, invalidCount } }`. Write bytes ONLY after all validation passes (orphan avoidance, 03-RESEARCH.md Pitfall 5).
- `"use server"` wrappers `parseUploadedCsv(formData)` / `saveRecipientSet(formData)` in actions.ts: auth() â†’ userId â†’ delegate; type-only re-export the result types.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client FormData (file + emailColumn) â†’ Server Action | Untrusted; server re-derives userId via auth(), re-validates the file and the confirmed column with zod, never trusts a client id |
| uploaded CSV bytes â†’ filesystem | Written only in saveRecipientSet, only after all guards pass, under an opaque uuid name |
| Server Action return value â†’ client | Closed typed union; a `raw` is always a message string, `invalidCounts` is a columnâ†’number map â€” never file bytes, a DB row, or a raw Error |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-3-IDOR | Information disclosure / Elevation | parseUploadedCsv / saveRecipientSet | mitigate | Each re-derives `userId` via Clerk `auth()` and passes it to the userId-scoped DAL; the userId-accepting core seams live in actions-core.ts (no "use server") so a client cannot bypass auth() (AUTH-02, ASVS V4). |
| T-3-DOS | Denial of service | both actions + next.config.ts | mitigate | `bodySizeLimit: "4mb"` caps the request body; the zod guard enforces MAX_UPLOAD_BYTES and MAX_ROWS with clear typed errors before parse/write, and MAX_ROWS is enforced at PARSE time so an oversized file never enters the review cycle (ASVS V5). |

| T-3-CRED | Information disclosure | action return values | mitigate | Results are a closed union of typed shapes; `raw` is a message string only and `invalidCounts` is a columnâ†’count number map â€” no CSV cell value is logged and no bytes/rows cross outward (D-06 discipline carried forward). |
