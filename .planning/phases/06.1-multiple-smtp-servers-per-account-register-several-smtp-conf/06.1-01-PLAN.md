<objective>
Promote `smtp_configs` from one-row-per-user to many-named-rows-per-user at the
data layer, and replace the userId-upsert DAL with an id-scoped, IDOR-safe,
soft-delete-aware CRUD surface. This is the shared data spine every downstream
slice (settings UI, compose picker, send paths, Phase 6 worker) builds on.

Purpose: Remove the single-row invariant safely on a live SQLite `/data` volume
(metadata-only DROP INDEX + additive ALTER ADD), backfill existing accounts to a
named default (SC4 zero-action), and give every consumer a single owner-scoped
by-id lookup so a client-supplied `smtp_config_id` can never cross tenants (SC3).

Output: Migrated schema (0003 generated + 0004 backfill), reworked `lib/data/smtp.ts`,
updated barrel, a rewritten `lib/data/smtp.test.ts`, and a new
`lib/db/migration.test.ts`.

MVP note: This phase is a data-model refactor whose irreducible foundation is the
schema migration + DAL. This plan is the data spine (the one legitimate
horizontal foundation, like a walking skeleton); Plans 02-04 are the vertical
user-facing slices that light up on top of it.
</objective>
