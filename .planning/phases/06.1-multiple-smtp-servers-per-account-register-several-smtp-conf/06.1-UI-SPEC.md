# Phase 06.1 тАФ UI Design Contract

> Visual and interaction contract for turning the single-server SMTP surface into a **multi-server** experience: a per-server settings list (add / edit / verify / set-default / soft-delete), the reused verify-before-save form extended with a required **label** and a **host-change password re-entry** gate (WR-09), a **destructive delete confirmation** with an in-use guard error state, a **no-default "choose a server"** state, and a **compose server picker** that auto-selects when exactly one verified server exists. Generated by gsd-ui-researcher (autonomous mode), verified by gsd-ui-checker.

This phase promotes `smtp_configs` from one-row-per-user to **many named rows per user** (06.1-RESEARCH.md). The UI consequences are concentrated on three existing surfaces plus their readiness copy:

1. **`/settings/smtp`** тАФ was a single-form wizard; becomes a **server list** (each row: label ┬╖ host:port ┬╖ verified badge ┬╖ default marker ┬╖ edit / delete) with an **"Add server"** entry into the existing wizard, reused unchanged per server.
2. **The add/edit form** (the existing `SmtpWizard`) тАФ gains a **required `label`** field and a **host-change тЖТ password re-entry** gate. Blank-keep-password stays allowed for port / TLS-mode / label / sender edits; changing the **host** forces re-entry so a stored credential is never authed against a host it wasn't verified with (CONTEXT.md WR-09 LOCKED, closes the exfil path).
3. **`/compose`** тАФ the `SendCard`'s boolean `hasSmtpConfig` gate becomes a **server picker** (shadcn `Select`) that auto-selects the account default, renders a **zero-click static line** when exactly one verified server exists, and shows a **"Choose a server"** placeholder in the no-default state.

Plus **destructive delete** (soft-delete тАФ new this phase; Phase 6 had none) and its **in-use guard** error, and **dashboard/compose readiness** copy shifting from "an SMTP server" to "at least one verified server."

It inherits the design system, spacing, typography, and color contract established in **02-UI-SPEC.md**, carried verbatim through **03-/04-/05-/06-UI-SPEC.md**. The shell is unchanged (`app/(app)/layout.tsx` тАФ 640px `max-w-2xl` content column, `p-8`, sidebar + top bar + footer). No new palette tokens, sizes, weights, or spacing values are introduced.

## Interaction & State Contract

### Server list (`/settings/smtp`)
- **Ordering:** default server first, then by `created_at` (CONTEXT.md Claude's Discretion тАФ matches the RESEARCH suggestion). The default row carries the "Default" `Star` badge and shows **no** "Make default" action.
- **Every listed server is verified** (verify-before-save is reused per server), so the "Verified" badge is effectively always present; if a row's `verified_at` is somehow null it renders **without** the badge (no false "Verified" claim).
- **Soft-deleted servers never appear** in the list (DAL filters `deleted_at IS NULL`).
- **Set-default** is a transactional swap (one default per user); on success the badge moves and a toast confirms. No page navigation.

### Add / edit form (reused `SmtpWizard`)
- **Create** always requires a password (existing `smtpFormSchema`).
- **Edit** keeps blank-keep for port / TLS-mode / username / label / sender changes (existing `smtpEditFormSchema` behavior).
- **Host-change gate (WR-09 LOCKED):** when the `host` field is dirty **and** the password is blank, the form blocks verify/save and anchors the host-change password message on the password field. Re-entering a password clears the gate. The stored credential is never sent to a changed host. This is the one new client-side rule layered onto the existing wizard; everything else (stepper, verify-then-save, TLS-switch suggestion, field-anchored errors) is inherited unchanged.
- The `label` field is required in both create and edit; case-insensitive uniqueness is validated server-side and surfaced as a field error (copy above).

### Delete (destructive тАФ new this phase)
- Trigger: the row's `Trash2` "Delete" control opens the **`AlertDialog`** confirmation (role `alertdialog`; not dismissable by a stray outside click for a destructive action тАФ Assumption V6).
- The confirm button runs soft-delete. On the **in-use guard** rejection (a `queued`/`running` campaign references the server) the dialog stays open and shows the destructive in-use `Alert`; the server is **not** deleted. On success the dialog closes, the row disappears, and a toast confirms.
- If the deleted server **was** the default, the account enters the **no-default state** (NO auto-promote тАФ CONTEXT.md LOCKED): the list shows the no-default callout and compose stops pre-selecting.

### No-default state
- **Settings:** the no-default `Alert` callout sits above the list; each non-default row offers "Make default".
- **Compose:** with a default present, the picker pre-selects it. With **no default and multiple** verified servers, the picker shows the **"Choose a server"** placeholder (nothing pre-selected) and the send actions stay gated with "Choose which server to send with above." A single remaining verified server is still auto-selected as the only option (zero clicks) even though it is not the account default (CONTEXT.md: auto-select тЙа silent promote).

### Compose picker (`SendCard`)
- **Exactly one verified server:** render the static "Sending over {label} ({from_addr})" line тАФ no dropdown, zero extra clicks тАФ and pass that server's id straight through.
- **Multiple verified servers:** render the shadcn `Select` labeled "Send with", pre-selecting the account default (or the "Choose a server" placeholder in the no-default state).
- **Zero verified servers:** keep the existing disabled gate with the updated "at least one" copy.
- The client only **proposes** a `smtpConfigId`; the server re-resolves it through `getSmtpConfigByIdForUser(userId, id)` (06.1-RESEARCH.md IDOR pattern). The picker never sees ciphertext тАФ options are built from the redacted DTO.

Interaction rules:
- **Verify-before-save is preserved per server** тАФ a server is only ever stored after a clean `transport.verify()` (inherited invariant, now per row).
- **The host is never authed against with a stored credential it wasn't verified with** тАФ the WR-09 host-change gate enforces password re-entry client-side, and the server re-checks (CONTEXT.md LOCKED).
- **Soft-delete keeps history** тАФ a deleted server disappears from the list and picker but its row survives for historical sender-identity rendering; the delete copy states this.
- **No silent default promotion** тАФ deleting the default yields an explicit no-default state; the user must choose (CONTEXT.md LOCKED).
- **One default per user** тАФ set-default is a transactional swap; the "Default" badge appears on exactly zero or one row.
- **The picker proposes; the server decides** тАФ every `smtpConfigId` is owner-re-resolved via `getSmtpConfigByIdForUser` (IDOR-safe); the DTO never carries the encrypted triple.
- **Labels are required and unique per account** (case-insensitive) тАФ surfaced as field errors, never silently truncated.
