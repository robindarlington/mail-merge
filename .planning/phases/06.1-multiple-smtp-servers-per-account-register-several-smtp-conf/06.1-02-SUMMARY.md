# Phase 06.1 Plan 02: Multi-Server SMTP Actions Summary

**Split the retired verify-and-upsert into id-scoped create/update seams with a required, case-insensitively-unique label, closed the WR-09 credential-exfil path (host change forces password re-entry before any decrypt), and gated soft-delete behind the in-use campaign check — exposed as auth-scoped `createServer`/`updateServer`/`setDefaultServer`/`deleteServer` actions plus a per-config test-send.**

## Performance

- **Duration:** ~25 min
- **Tasks:** 2 (+1 Rule 1 deviation fix)
- **Files modified:** 5 (4 planned + 1 deviation)

## Accomplishments

- Added a required `label` field to `smtpFormSchema` using the exact UI-SPEC Copywriting Contract strings (`"Give this server a label."` / `"Keep the label under 60 characters."`); `smtpEditFormSchema` inherits it.
- Reworked `applyVerifiedConfig` to be id-aware: `id === null` inserts a NEW named server via `createSmtpConfig`; a number updates the owned row via `updateSmtpConfigById`. Removed the retired `upsertSmtpConfig` call entirely.
- **WR-09 (LOCKED option 2):** a blank ("keep") password is only merged when the host is unchanged. A blank password with a changed host is rejected with the field-anchored copy BEFORE any decrypt runs — the stored credential is never re-authed against a client-changed host (T-061-05).
- Enforced case-insensitive, per-account label uniqueness (trim + toLowerCase, excluding the row being edited by id) before verify/persist, rejecting with the `"…Pick a different name."` copy on the `label` path.
- First-server auto-default: a create computes `is_default` = (account currently has zero non-deleted servers); later adds never silently promote themselves over an existing default.
- Added `setDefaultConfigCore` (0-length → `not_found`) and `softDeleteConfigCore` (in-use guard via `countActiveSendsForConfig` → `in_use`, else soft-delete, 0-length → `not_found`) seams; extended the `ActionError` union with `not_found` and `in_use`.
- Replaced `verifyAndSave` with `createServer` and added `updateServer`/`setDefaultServer`/`deleteServer` "use server" wrappers — each re-derives `userId` via `auth()` and validates the id before any DAL call. `sendTestEmail` now resolves its config via `getSmtpConfigByIdForUser(userId, id)`.

## Decisions Made

- **Id coercion kept private in actions.ts:** `smtpConfigIdSchema = z.coerce.number().int().positive()` mirrors `campaignIdSchema` but lives module-private in `actions.ts` — a `"use server"` module may only export async actions, and the plan scoped schema.ts edits to the `label` field only.
- **Tests hit the core seams, not the wrappers:** the wrappers call Clerk `auth()`, which the plain node test runner cannot supply; the existing harness pattern already tests `applyVerifiedConfig`/`sendTestVia` directly, so the new set-default / delete / in-use / cross-tenant cases target `setDefaultConfigCore` / `softDeleteConfigCore`.
