## Deviations from Plan

The two auto tasks were executed as written. Three findings surfaced during the Task 3 human-verify walkthrough and were fixed on master, then re-verified by the user ("approved").

### Fixed during human-verify

**1. [Rule 1 - Bug] Label-only server edit didn't save + latent cross-config clobber**
- **Found during:** Task 3 (multi-server walkthrough, step "rename")
- **Issue:** The wizard's from-only fast path called `updateFromFields`, which never persisted the label AND (latent bug) updated by `userId` with no id filter — clobbering `from_addr`/`from_name` across ALL of the user's configs.
- **Fix:** Replaced with an id-scoped `updateSmtpConfigMeta` + `updateServerMeta` action + `smtpMetaFormSchema`; the wizard fast path now saves label + sender fields with the same duplicate-label check; regression tests added.
- **Files modified:** wizard fast path + `lib/smtp` meta action/schema/DAL, tests.
- **Verification:** `npx tsc --noEmit` clean; `npm test` 212/212; user re-verified rename works.
- **Committed in:** `8577916` (fix)

**2. [Rule 1 - Bug] /compose server picker never appeared**
- **Found during:** Task 3 (walkthrough, step 5 — compose picker)
- **Issue:** `SendCard` gated the entire picker behind `templateId` (session-local `savedTemplateId` starts null), so the "Send with" Select was unreachable until a template save.
- **Fix:** The server block always renders; only the test-send/review controls stay template-gated.
- **Files modified:** `components/campaign/send-card.tsx`.
- **Verification:** user confirmed the picker appears pre-selecting the default with two servers.
- **Committed in:** `806a16e` (fix)

**3. [Informational — out of plan scope] Renamed list labels not shown in /compose recipient selector**
- **Found during:** Task 3 walkthrough (cross-stream observation).
- **Issue:** Renamed list labels weren't reflected in the /compose recipient selector.
- **Fix:** Handled under quick task `260715-r8d` (commit `b3dae53`), outside this plan's scope.
