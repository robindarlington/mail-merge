# Phase 06.1: Code Review Report

**Reviewed:** 2026-07-15
**Depth:** standard
**Files Reviewed:** 28
**Status:** issues_found

## Summary

Reviewed the full 06.1 multi-SMTP surface: DAL, action seams, "use server" wrappers, zod schemas, migrations, and the client islands. The core security invariants hold up well under adversarial tracing:

- **Tenant isolation:** every by-id path goes through `and(eq(id), eq(userId), isNull(deleted_at))`; I found no `eq(id)`-alone query. Cross-tenant ids resolve to `undefined`/0-rows in every seam (verified in DAL, actions-core, and campaign seams, and covered by tests).
- **Credential secrecy:** the encrypted triple never crosses to the client. All RSC pages project through `toSmtpConfigDto`, `decrypt()` outputs live only in locals, and every failure surface is a message-only closed union. No leak path found.
- **WR-09 gate:** client (`smtp-wizard.tsx` `hostChangeGate`) and server (`actions-core.ts` blank-password branch) agree — both compare trimmed hosts (schema trims), and the server rejects before any decrypt. Correct.
- **Checkpoint fixes verified:** `updateSmtpConfigMeta` / `updateMetaCore` are properly id+userId scoped (the sibling-clobber regression is tested); the wizard threads the `createServer`-returned id into `StepTestSend`; the SendCard server block renders before the template gate, so the picker is no longer hidden behind a template save.

However, the **one-default-per-user invariant has a real hole**: `setDefaultSmtpConfig` commits its clear-all-defaults step even when the target update matches zero rows, so any stale, deleted, or cross-tenant id silently strips the caller's existing default — directly contradicting the function's own docstring and the DAL test's assumption. That is the one Critical finding. The rest are robustness gaps: unvalidated/unlimited `sendTestEmail`, a TOCTOU window around the multi-second verify dial, wedged client state on rejected server actions (including a user-trapping undismissable modal), a delete-then-enqueue gap, stale picker state, and orphaned draft accumulation.

## Critical Issues

### CR-01: `setDefaultSmtpConfig` clears the caller's default and COMMITS even when the target id matches no row

**File:** `lib/data/smtp.ts:200-219`
**Issue:** Inside the transaction, step 1 unconditionally clears `is_default` on every row for `userId`; step 2 sets it on the target with the owner/not-deleted filter. When the target id is cross-tenant, soft-deleted, or unknown, step 2 matches 0 rows — but the callback returns normally, so drizzle/better-sqlite3 **commits the clear**. The user is left with ZERO default servers. The docstring claims "0 for a cross-tenant / deleted / missing id (**nothing changes**, because the target update matched no row)" — that is false. This is reachable from the wire: `setDefaultServer(anyId)` is a client-invocable action, so a stale UI (server deleted in another tab) or a deliberately forged id silently corrupts the caller's default state, and downstream compose falls into the "Choose a server" no-default state with no explanation. The test at `lib/data/smtp.test.ts:133-138` asserts USER_B's row is untouched but never asserts USER_A's own default survived the failed call — which is exactly what it loses.
**Fix:**
```ts
export function setDefaultSmtpConfig(userId: string, id: number) {
  try {
    return db.transaction((tx) => {
      tx.update(smtp_configs)
        .set({ is_default: false })
        .where(eq(smtp_configs.userId, userId))
        .run();
      const won = tx
        .update(smtp_configs)
        .set({ is_default: true })
        .where(
          and(
            eq(smtp_configs.id, id),
            eq(smtp_configs.userId, userId),
            isNull(smtp_configs.deleted_at),
          ),
        )
        .returning({ id: smtp_configs.id })
        .all();
      // Roll back the clear when the target did not match — otherwise the
      // caller's existing default is silently stripped.
      if (won.length === 0) throw new Error("SET_DEFAULT_TARGET_NOT_FOUND");
      return won;
    });
  } catch (e) {
    if ((e as Error).message === "SET_DEFAULT_TARGET_NOT_FOUND") return [];
    throw e;
  }
}
```
Also add a test asserting the caller's prior default is intact after a cross-tenant/bogus `setDefaultSmtpConfig` call.

## Warnings

### WR-01: `sendTestEmail` never validates `toAddress`

**File:** `lib/smtp/actions.ts:191-214`
**Issue:** Every other untrusted send-address in the codebase is parsed (`testAddressSchema` in campaign seams), but `sendTestEmail`'s `toAddress` flows straight into `sendMail`'s `to:` with no runtime validation. The TypeScript `string | undefined` annotation is not enforcement — this is a wire-callable server action, so the argument can be any JSON value (object, array, garbage string). Nodemailer will error on nonsense, but the input should fail closed as `validation` at the trust boundary, consistent with the rest of the surface.
**Fix:**
```ts
const toParsed = z.email().optional().safeParse(toAddress);
if (!toParsed.success) {
  return { ok: false, error: { kind: "validation", issues: toParsed.error.issues } };
}
let to = toParsed.data;
```

### WR-02: `sendTestEmail` has no rate limit despite dialing verify + sending real mail per call

**File:** `lib/smtp/actions.ts:191-246`
**Issue:** The file header cites T-061-08/T-2-SPAM as the reason `createServer`/`updateServer` are rate-limited, but `sendTestEmail` — which performs a `transport.verify()` dial AND a real send on every invocation — has none. An authenticated client can loop it without bound (arbitrary recipient via WR-01), turning the endpoint into an unmetered relay through the user's own SMTP. The SSRF surface is bounded (host was screened at save time), but the abuse/spam surface is not.
**Fix:** Apply `underVerifyRateLimit(userId)` (or a dedicated per-user send limiter) at the top of `sendTestEmail`, mirroring create/update.

### WR-03: TOCTOU across the verify dial — duplicate labels or an unhandled UNIQUE-constraint throw on concurrent create

**File:** `lib/smtp/actions-core.ts:135-146, 208-218`
**Issue:** The label-uniqueness read and the `isFirstServer` read both happen BEFORE `verifyFn` — a network dial the UI itself says "can take up to 15 seconds". The inline comment justifies the application-level check with "the TOCTOU window ... is negligible", but the window is the full verify duration, not a single-writer microsecond. Two concurrent creates (two tabs, a retried request):
1. both pass the label check → duplicate labels persist (the "LOCKED unique" rule silently broken — no DB constraint backs it), or
2. both see `existingConfigs.length === 0` → both insert `is_default: true` → the second insert throws `SqliteError: UNIQUE constraint failed` on the partial index. That throw is uncaught, so the action REJECTS — violating the "uniform result ... never rejects" `ActionResult` contract and (per WR-04) wedging the wizard client-side.
**Fix:** Wrap the `createSmtpConfig`/`updateSmtpConfigById` persist in try/catch and map a constraint violation to `validationError("label", ...)` / a retryable error; optionally re-run the label check inside the same tick as the insert (post-verify) so the window collapses to the single-writer guarantee the comment assumes.

### WR-04: No try/catch around awaited server actions — a rejected action permanently wedges client state (including trapping the user in the undismissable confirm modal)

**File:** `components/campaign/confirm-send-dialog.tsx:110-134` (worst case), `components/smtp/step-verify.tsx:193-195`, `components/smtp/server-list.tsx:96-107, 120-134`, `components/campaign/test-send-panel.tsx:128-159`
**Issue:** Server actions reject on network failure (and on any uncaught server throw, e.g. WR-03). None of the client handlers guard for it:
- `confirm-send-dialog.tsx` `confirm()`: `setSubmitting(true)` then `await enqueueCampaign(...)` with no catch. On rejection `submitting` stays `true` forever — Cancel is `disabled={submitting}`, escape and outside-click are `preventDefault`ed, and `showCloseButton={false}`. The user is **permanently trapped in the modal** until a full page reload. The load effect's async IIFE (line 86) similarly has no catch → `loading` skeleton forever.
- `step-verify.tsx` `runVerify()`: `onPendingChange(true); await persist(values);` — on rejection `pending` stays true and the entire wizard fieldset is disabled.
- `server-list.tsx` `onMakeDefault`: `busyId` stuck non-null disables every Make-default button; `onConfirmDelete`: `deleting` stuck true and `closeDelete` refuses to close while `deleting`.
- `test-send-panel.tsx` `send()`: `sending` stuck true disables the panel.
**Fix:** Wrap each awaited action in try/finally so the in-flight flag always resets, and surface a generic "network error — try again" failure on catch. E.g.:
```ts
setSubmitting(true);
try {
  const res = await enqueueCampaign({ campaignId });
  // ...existing handling
} catch {
  setEnqueueError("We couldn't reach the server. Check your connection and try again.");
} finally {
  setSubmitting(false);
}
```

### WR-05: Delete-then-enqueue gap — a draft referencing a soft-deleted config can still be enqueued

**File:** `lib/campaign/actions-core.ts:503-516`, `lib/data/smtp.ts:249-265`
**Issue:** The in-use delete guard counts only `queued`/`running` campaigns, so deleting a config that a *draft* references is allowed (correct — drafts are churn, see WR-07). But `enqueueCampaignCore` flips draft→queued on the row count alone; it never re-resolves `campaign.smtp_config_id` through the soft-delete-aware DAL. The UI path is protected only incidentally (`buildConfirmSummary` returns `not_found` first), yet `enqueueCampaign` is an independently wire-callable action: prepare a draft → delete the config → call `enqueueCampaign(campaignId)` directly → a `queued` campaign now references a soft-deleted config, which the Phase-6 worker will pick up and either fail on or send through a server the user believes is gone.
**Fix:** In `enqueueCampaignCore`, before the flip, load the campaign owner-scoped and `getSmtpConfigByIdForUser(userId, campaign.smtp_config_id)`; return `not_found` when it no longer resolves.

### WR-06: Stale `smtpConfigId` state — SendCard can display one server while proposing another (deleted) one

**File:** `components/compose/compose-editor.tsx:128-130`, `components/campaign/send-card.tsx:80-103`
**Issue:** `smtpConfigId` is initialized once via the `useState` initializer from the `configs` prop and is never reconciled when the prop changes. After an RSC refresh (e.g. the selected server was deleted in the settings tab, one server remains): `configs` becomes `[B]`, SendCard's `singleServer` branch renders the static "Sending over B" line — but `smtpConfigId` still holds deleted A's id, which is what test-send and prepare actually propose. The server correctly rejects with `not_found`, but the UI names one server while sending through a request for a different, dead one. In the multi-server case the `Select` shows the placeholder for the stale value, but `smtpConfigId !== null` keeps the send controls ungated with a dead id.
**Fix:** Reconcile in an effect:
```ts
useEffect(() => {
  if (smtpConfigId !== null && !configs.some((c) => c.id === smtpConfigId)) {
    setSmtpConfigId(initialSmtpConfigId(configs));
  }
}, [configs, smtpConfigId]);
```

### WR-07: Every confirm-dialog open creates a new draft campaign row; cancels are never cleaned up

**File:** `components/campaign/confirm-send-dialog.tsx:78-108`, `lib/campaign/actions-core.ts:341-382`
**Issue:** The open-effect calls `prepareCampaign` on every open transition and on every `[recipientSetId, templateId, smtpConfigId]` change while open. Cancel closes the modal without deleting the draft. A user who opens/cancels the review repeatedly (a normal "check the numbers, tweak, re-check" loop) accumulates one orphaned `campaigns` row per open, forever — each holding NOT NULL FKs into recipient_sets/templates/smtp_configs. Beyond table bloat, these drafts are exactly what enables the WR-05 gap and complicate any future "campaign history" surface (which must now filter abandoned drafts).
**Fix:** Either (a) reuse an existing draft for the same `(recipient_set_id, template_id, smtp_config_id)` tuple in `prepareCampaignCore` (upsert-style), or (b) delete the draft on cancel/unmount, or (c) periodically reap `status='draft'` rows older than a day.

### IN-02: Blank-password edit of a not-owned/deleted id masquerades as "Password is required"

**File:** `lib/smtp/actions-core.ts:156-159`
**Issue:** In the blank-password edit branch, a cross-tenant/deleted/unknown id returns `validationError("password", "Password is required")` while the non-blank path for the same id returns `{ kind: "not_found" }`. The user with a stale edit form gets a confusing password error instead of the "server no longer exists" message StepVerify already renders for `not_found`.
**Fix:** Return `{ ok: false, error: { kind: "not_found" } }` when `getSmtpConfigByIdForUser` misses on the edit path.

### IN-03: `verifyAttempts` map grows unboundedly

**File:** `lib/smtp/actions-core.ts:69-87`
**Issue:** One entry per userId is retained forever (timestamps are filtered per-call, but the key is never deleted, and an empty array is re-set). Negligible for the v1 single-process tier, but a slow leak across a long-lived worker/web process.
**Fix:** `verifyAttempts.delete(userId)` when the filtered array is empty; or use an LRU.

### IN-04: Verify rate-limit budget is consumed before input validation

**File:** `lib/smtp/actions.ts:98-101, 121-124`
**Issue:** `underVerifyRateLimit` records the attempt before `applyVerifiedConfig` parses the input, so five schema-invalid submissions (typos that never dial anything) lock the user out of verifying for a minute. The limit exists to cap *dials*, not form mistakes.
**Fix:** Parse first (or record the attempt only when a dial will actually happen).
