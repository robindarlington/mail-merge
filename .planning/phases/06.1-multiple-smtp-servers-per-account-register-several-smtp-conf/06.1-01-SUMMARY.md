# Phase 06.1 Plan 01: Multi-SMTP Data Spine Summary

**smtp_configs promoted to many-named-rows-per-user via an additive SQLite migration + backfill, with an id-scoped, IDOR-safe, soft-delete-aware Drizzle DAL replacing the userId-upsert surface.**

## Performance

- **Duration:** ~20 min
- **Tasks:** 2 (both TDD)
- **Files modified:** 5 modified, 3 created

## Accomplishments
- Removed the one-row-per-user invariant (dropped `smtp_configs_user_uq`) and added a partial unique index `smtp_configs_user_default_uq` that structurally enforces exactly one default per user.
- Added `label`, `is_default`, `deleted_at` columns via a metadata-only additive migration (DROP INDEX + 3x ALTER ADD) — verified against the Pitfall 1 gate to contain NO `CREATE TABLE __new_smtp_configs` rebuild, so the live `/data` volume and the campaigns FK survive intact.
- Backfilled surviving single-server accounts to `label='Default'`, `is_default=1` with zero user action (SC4).
- Replaced the DAL with an id-addressed CRUD surface whose sole by-id path (`getSmtpConfigByIdForUser`) filters `and(eq(id), eq(userId), isNull(deleted_at))` — one filter delivering both IDOR safety (T-061-01) and soft-delete invisibility.
- Added `setDefaultSmtpConfig` (transactional clear-all-then-set-one), `softDeleteSmtpConfig`, and `countActiveSendsForConfig` (in-use guard for the action layer).
- Extended the client-safe DTO with `id/label/is_default` while keeping the password ciphertext absent by construction (T-061-02).

## Task Commits

Each task was committed atomically:

1. **Task 1: [BLOCKING] Schema change + generate/inspect/apply migration + backfill** - `701e2f7` (feat)

_Both are TDD tasks; each commit bundles the failing-then-passing test with its implementation._

## Files Created/Modified
- `lib/db/schema.ts` - Added `uniqueIndex` import; added label/is_default/deleted_at columns; replaced the single-row unique index with the partial `smtp_configs_user_default_uq`.
- `drizzle/0003_pale_wallow.sql` - Generated additive migration (DROP INDEX + 3x ALTER ADD + partial unique index).
- `drizzle/0004_backfill_default_smtp.sql` - Custom backfill: `UPDATE ... SET label=COALESCE(label,'Default'), is_default=1 WHERE deleted_at IS NULL`.
- `drizzle/meta/_journal.json` - Registered 0003 and 0004 (plus their snapshots).
- `lib/db/migration.test.ts` - New integration test: seeds a pre-06.1 row, applies committed 0003/0004, asserts backfill + index shape + multi-row/one-default invariants.
- `lib/data/smtp.ts` - Reworked to id-scoped soft-delete-aware CRUD; retired `getSmtpConfigForUser`/`upsertSmtpConfig`; extended DTO.
- `lib/data/index.ts` - Barrel updated to the new export surface.
- `lib/data/smtp.test.ts` - Rewritten for multi-row create/list ordering, IDOR + soft-delete invisibility, one-default invariant, in-use guard, DTO redaction.

## Decisions Made
- **Backfill split into its own migration (0004):** keeps 0003 a pure drizzle-generated DDL file (the inspection gate only reasons about generated output) and 0004 the hand-authored data step. Order is guaranteed and safe because each account held at most one row pre-migration.
- **`countActiveSendsForConfig` added to the DAL now (not deferred):** it is the in-use guard the plan's `<action>` block specifies the action layer must call before soft-delete; landing it with the DAL keeps the guard tenant-scoped (`inArray(status, ['queued','running'])` + `eq(userId)`).

## Deviations from Plan

None affecting scope — the plan was executed as written. One import beyond the plan's literal `import { and, desc, eq, isNull, sql }` line was required: `inArray` was added to `lib/data/smtp.ts` to express `status IN ('queued','running')` inside `countActiveSendsForConfig` (the plan named the helper but not the operator). This is a mechanical completion of a specified function, not a scope change.
