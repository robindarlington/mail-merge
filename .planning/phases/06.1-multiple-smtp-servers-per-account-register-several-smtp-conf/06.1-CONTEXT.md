# Phase 06.1: Multiple SMTP servers per account - Context

**Gathered:** 2026-07-14
**Status:** Ready for planning
**Source:** Live Q&A with Rob (4 decisions), following 06.1-RESEARCH.md open questions

<domain>
## Phase Boundary

A user can register several named SMTP servers on one account — each verified
(`transport.verify()`) and AES-256-GCM-encrypted independently — and pick which
one a given campaign sends through. Existing single-server accounts migrate
with zero action. Send paths load config by `campaign.smtp_config_id`
(owner-scoped). Executes BEFORE Phase 6's worker is built.

</domain>

<decisions>
## Implementation Decisions

### WR-09 posture — edit flow (LOCKED, resolves the standing WR-09 todo)
- Blank-keep (leave password empty to keep the stored one) remains allowed for
  port / TLS-mode / label edits.
- Changing the HOST requires re-entering the password. The stored credential is
  NEVER authed against a host it wasn't verified with. This closes the
  credential-exfiltration path identified in WR-09 (option 2 of that todo).
- Applies per server row in the multi-server settings UI.

### Deletion model (LOCKED)
- Soft-delete: `deleted_at` column. Soft-deleted servers disappear from
  settings list and compose picker; campaign history stays intact (FK rows
  untouched).
- In-use guard: deleting a server referenced by a queued/running campaign is
  blocked with an actionable error. Completed/failed campaign references do not
  block soft-delete.
- Never hard-DELETE a referenced row (FK `no action` + `foreign_keys=ON`
  already throws — keep that as the backstop).

### Default-server policy (LOCKED — note: differs from research recommendation)
- Explicit `is_default` column with a partial unique index (`WHERE is_default=1`)
  per user; transactional default swap.
- Migration backfill: the surviving pre-migration row gets `label='Default'`,
  `is_default=1` (SC4 zero-action).
- **When the default server is deleted: NO auto-promote.** The account enters a
  no-default state; compose shows a "choose a server" prompt until Rob (the
  user) explicitly sets a new default or picks one for the campaign. A single
  remaining verified server may still be auto-selected in the compose picker as
  the only option, but it does not silently become the account default.

### Labels (LOCKED)
- Required on every server, unique per account (case-insensitive uniqueness
  preferred), max ~60 chars. Compose picker and settings list show the label.

### Claude's Discretion
- Exact zod schema shapes, error copy wording (follow UI-SPEC copy discipline),
  ordering of servers in lists (suggest: default first, then created_at),
  whether label uniqueness is enforced by index or application check (justify),
  migration file naming, test structure.

</decisions>

  existing verify-before-save flow is reused unchanged per server.
