# Phase 06.1: Multiple SMTP servers per account — Research

**Researched:** 2026-07-14
**Domain:** Multi-tenant relational data modelling + SQLite schema migration (Drizzle/better-sqlite3), credential-flow security, Next.js Server Actions + shadcn UI
**Confidence:** HIGH (codebase facts verified by direct read; migration behaviour cross-checked with Drizzle docs)

## Summary

This phase promotes the app's `smtp_configs` table from **one row per user** to **many named rows per user**, lets a user pick which server a campaign sends through, and migrates existing single-server accounts transparently. The data model is already 90% ready: `campaigns.smtp_config_id` is a NOT-NULL FK stamped at draft creation (`lib/campaign/actions-core.ts:343`), so per-campaign server selection is *structurally* in place. The only hard blocker is the single-row invariant, which lives in exactly two places: the standalone unique index `smtp_configs_user_uq` (`lib/db/schema.ts:64`, materialised as `drizzle/0001_shiny_stature.sql`) and `upsertSmtpConfig`'s `onConflictDoUpdate(userId)` (`lib/data/smtp.ts:90`).

The single most de-risking finding: **the unique constraint is a standalone `CREATE UNIQUE INDEX`, not an inline table/column constraint.** That means dropping it generates a plain `DROP INDEX IF EXISTS` — SQLite metadata-only, no 12-step table rebuild, zero row-rewrite risk on the live `/data` volume. Combined with additive `ALTER TABLE ... ADD COLUMN` (which SQLite also does as an O(1) metadata op), the entire migration is safe and non-destructive. This kills the biggest risk in the feasibility sketch (Key Question 1).

Two decisions need explicit resolution before planning locks: (a) **deletion semantics** — the `smtp_config_id` FK is `ON DELETE no action` with `PRAGMA foreign_keys=ON`, so a hard `DELETE` of *any* referenced config already throws `SQLITE_CONSTRAINT_FOREIGNKEY` (even for old completed campaigns). This is *too* strict; the recommended answer is **soft-delete** (`deleted_at`) plus an explicit in-use guard. (b) **WR-09** — the blank-keep-password edit flow's credential-exfiltration surface is *multiplied* per-row by this phase and remains an open product decision; this phase must not silently widen it.

**Primary recommendation:** One additive Drizzle migration (drop `smtp_configs_user_uq`; add `label`, `is_default`, `deleted_at`; partial-unique-index for one-default-per-user; backfill the existing row to `label='Default', is_default=1`). Replace the userId-upsert DAL with id-scoped CRUD (`listSmtpConfigsForUser`, `getSmtpConfigByIdForUser`, `createSmtpConfig`, `updateSmtpConfigById`, `setDefaultSmtpConfig`, `softDeleteSmtpConfig`). Thread a `smtpConfigId` through prepare/test-send inputs and the compose picker (auto-select when one). Add `id`/`label`/`is_default` to the redacted DTO. Ship no new npm packages.

### Pattern 1: Standalone-index drop is metadata-only (the migration lynchpin)
**What:** The one-per-user rule is a *named standalone index*, not an inline constraint.
**Evidence:** `drizzle/0001_shiny_stature.sql` = `CREATE UNIQUE INDEX \`smtp_configs_user_uq\` ON \`smtp_configs\` (\`user_id\`);` — a separate statement, produced because the schema declares it in the table's third-arg array (`(t) => [unique("smtp_configs_user_uq").on(t.userId)]`, `schema.ts:64`).
**When to use:** Removing the `unique(...)` line from the schema and running `drizzle-kit generate` emits `DROP INDEX IF EXISTS "smtp_configs_user_uq";` — no `CREATE TABLE __new / INSERT SELECT / DROP / RENAME` rebuild. `[CITED: orm.drizzle.team — DROP INDEX IF EXISTS pattern]` corroborated by the standalone-index evidence above `[VERIFIED: drizzle/0001 file]`.
**Example (expected generated 0003, verify by inspecting actual output):**
```sql
-- drizzle-kit generate output (INSPECT the real file before applying)
DROP INDEX IF EXISTS `smtp_configs_user_uq`;--> statement-breakpoint
ALTER TABLE `smtp_configs` ADD `label` text;--> statement-breakpoint
ALTER TABLE `smtp_configs` ADD `is_default` integer DEFAULT 0 NOT NULL;--> statement-breakpoint
ALTER TABLE `smtp_configs` ADD `deleted_at` integer;--> statement-breakpoint
CREATE UNIQUE INDEX `smtp_configs_user_default_uq` ON `smtp_configs` (`user_id`) WHERE `is_default` = 1;
```
> ⚠️ **Verification gate:** drizzle-kit *sometimes* chooses a full table rebuild for SQLite when it can't express a change as ALTER (documented behaviour for column drops/type changes). Dropping an index + adding columns should NOT trigger a rebuild, but the plan MUST include a task that runs `npm run db:generate` and **reads the generated `0003_*.sql`** to confirm it is `DROP INDEX` + `ALTER ADD`, not `CREATE TABLE __new_smtp_configs`. If a rebuild appears, that is a red flag on a live `/data` volume (foreign_keys must be OFF during the rebuild, and the FK from `campaigns` must survive).

### Pattern 2: Data backfill for the surviving row (SC4 — zero-action migration)
**What:** Existing accounts have exactly one `smtp_configs` row (enforced until now). Post-column-add, that row has `label=NULL, is_default=0` and would fail the "surviving row becomes default" criterion.
**How:** A backfill statement, run *after* the DDL. Two viable placements:
- **Data migration file (recommended):** add a second migration `0004_backfill_default_smtp.sql` = `UPDATE smtp_configs SET label = COALESCE(label, 'Default'), is_default = 1 WHERE deleted_at IS NULL;`. Because pre-migration there is ≤1 row per user, unconditionally setting `is_default=1` cannot violate the partial-unique index. Drizzle applies custom SQL migrations placed in `drizzle/` and tracked in `_journal.json`.
- Alternative: a guarded step in `scripts/migrate.ts` after `migrate()`. Rejected — keeps migration logic out of the tracked migration chain and is harder to test.
**Example:**
```sql
-- 0004_backfill_default_smtp.sql (custom data migration)
UPDATE `smtp_configs` SET `label` = COALESCE(`label`, 'Default'), `is_default` = 1 WHERE `deleted_at` IS NULL;
```

### Pattern 3: IDOR-safe get-by-id (mirror the campaigns DAL)
**What:** Multi-server introduces a client-supplied `smtp_config_id` — a fresh IDOR surface.
**How:** New DAL fn structurally identical to `getCampaignForUser` (`lib/data/campaigns.ts:68`): `and(eq(id), eq(userId))`, never `eq(id)` alone. There must be NO fetch-by-id-without-owner path.
**Example:**
```typescript
// lib/data/smtp.ts — the ONLY by-id path (IDOR defense, AUTH-02)
export function getSmtpConfigByIdForUser(userId: string, id: number) {
  return db.query.smtp_configs.findFirst({
    where: and(eq(smtp_configs.id, id), eq(smtp_configs.userId, userId),
               isNull(smtp_configs.deleted_at)),  // soft-deleted rows are invisible to picks
  });
}
```

### Pattern 4: Set-default as an atomic transaction
**What:** Exactly one `is_default=1` per user. The partial unique index enforces it structurally, but the *transition* (clear old default, set new) must be atomic or the index throws mid-update.
**How:** better-sqlite3 synchronous transaction: clear then set, in one txn, scoped to userId.
```typescript
export function setDefaultSmtpConfig(userId: string, id: number) {
  return db.transaction((tx) => {
    tx.update(smtp_configs).set({ is_default: false })
      .where(eq(smtp_configs.userId, userId)).run();
    return tx.update(smtp_configs).set({ is_default: true })
      .where(and(eq(smtp_configs.id, id), eq(smtp_configs.userId, userId),
                 isNull(smtp_configs.deleted_at)))
      .returning({ id: smtp_configs.id }).all();  // length 1 = success, 0 = IDOR/not-found
  });
}
```

### Anti-Patterns to Avoid
- **Changing the `campaigns.smtp_config_id` FK to CASCADE or SET NULL** — CASCADE destroys campaign history (violates SC5); SET NULL violates the NOT-NULL column. Leave it `ON DELETE no action`.
- **Hard-DELETE of a config** — blocked by the FK for *any* referencing campaign (including completed ones), so it will surprise users who "just want to remove an old server." Use soft-delete.
- **Keeping `upsertSmtpConfig(onConflictDoUpdate(userId))`** — its conflict target (`smtp_configs.userId`) ceases to exist when the unique index is dropped; the upsert becomes a no-op-or-error. It MUST be replaced by explicit create + update-by-id.
- **Exposing the encrypted triple by adding fields to the DTO carelessly** — when adding `id/label/is_default` to `SmtpConfigDto`, keep the explicit enumeration; never spread the row.
- **Trusting a client `smtpConfigId` without owner re-resolution** — always route through `getSmtpConfigByIdForUser`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| One-default-per-user enforcement | App-only "check then update" | SQLite **partial unique index** `WHERE is_default=1` | DB-level guarantee immune to races/bugs; drizzle-orm supports `.where()` on unique indexes |
| Dropping the single-row constraint | Manual `CREATE TABLE __new…INSERT SELECT` rebuild script | `drizzle-kit generate` → `DROP INDEX` | It's a standalone index; a rebuild would be strictly more dangerous on the live volume |
| IDOR-safe config lookup | Ad-hoc `where id=?` + a manual owner check afterwards | `getSmtpConfigByIdForUser` with `and(eq(id),eq(userId))` | Structural single-path defense (mirrors `getCampaignForUser`) — no forgotten check |
| Server→client redaction | Manually picking safe fields at each call site | Extend the single `toSmtpConfigDto` boundary | One redaction point can't leak the password triple by omission |
| Atomic default-swap | Two separate awaited updates | `db.transaction()` (better-sqlite3 sync txn) | Avoids a window where 0 or 2 defaults exist / partial-index throws |

**Key insight:** Every "new" capability here already has a proven twin in the codebase (campaigns DAL for IDOR, the wizard for verify+encrypt, the DTO for redaction). The phase is mostly *cloning established patterns onto a second cardinality*, not inventing mechanisms.

## Common Pitfalls

### Pitfall 1: drizzle-kit silently choosing a table rebuild
**What goes wrong:** For some SQLite schema edits drizzle-kit emits a full `CREATE TABLE __new_… / INSERT … SELECT / DROP / ALTER RENAME` sequence. On a live `/data` volume this temporarily drops+recreates the table; the `campaigns→smtp_configs` FK and `foreign_keys` pragma must be handled correctly or referential integrity breaks.
**Why it happens:** drizzle-kit can't express column drops/renames/type-changes as `ALTER`, so it rebuilds. Dropping an *index* + *adding* columns should not trigger this, but a mixed diff can surprise.
**How to avoid:** Add a mandatory plan task: run `npm run db:generate`, open `drizzle/0003_*.sql`, assert it contains `DROP INDEX` + `ALTER TABLE … ADD` and **NOT** `CREATE TABLE __new`. Only then apply.
**Warning signs:** `__new_smtp_configs` in the generated SQL; a `PRAGMA foreign_keys=OFF` wrapper.

### Pitfall 2: `upsertSmtpConfig` becomes a silent trap
**What goes wrong:** `onConflictDoUpdate({ target: smtp_configs.userId })` references an index that no longer exists after the migration → either a runtime error or (worse) it inserts a duplicate every save.
**Why it happens:** The conflict target is the dropped index.
**How to avoid:** Delete `upsertSmtpConfig` entirely in the same change; replace with `createSmtpConfig` (insert) + `updateSmtpConfigById` (update-by-id, owner-scoped). Update `lib/data/index.ts` re-exports and the `lib/data/smtp.test.ts` which currently *manually recreates* the index (`CREATE UNIQUE INDEX IF NOT EXISTS smtp_configs_user_uq`) — that test line must be removed/rewritten.

### Pitfall 3: The hidden `getSmtpConfigForUser` call graph
**What goes wrong:** Seven production call sites use `getSmtpConfigForUser(userId)` (single-row). Left unchanged, they'll grab an arbitrary row (or `findFirst`'s first) instead of the campaign-selected one — sending via the wrong server.
**Call sites (verified):** `lib/smtp/actions-core.ts:98` (blank-keep edit), `lib/smtp/actions.ts:114` (sendTestEmail), `lib/campaign/actions-core.ts:167` (test-send), `:339` (prepareCampaign), `:391` (buildConfirmSummary); RSCs `app/(app)/settings/smtp/page.tsx:20`, `app/(app)/compose/page.tsx:34`, `app/(app)/dashboard/page.tsx:35`.
**How to avoid:** Triage each: **edit/blank-keep** → by the row's id; **test-send/prepare** → by input `smtpConfigId`; **confirm-summary** → by `campaign.smtp_config_id`; **settings RSC** → `listSmtpConfigsForUser`; **compose/dashboard readiness** → "≥1 non-deleted verified config." Keep `getSmtpConfigForUser` only if a "get the default" helper is genuinely useful (e.g. auto-select), otherwise retire it.

### Pitfall 4: WR-09 credential-exfil surface multiplies (SECURITY — needs decision)
**What goes wrong:** The blank-keep-password edit (`applyVerifiedConfig`, `actions-core.ts:97`) lets a hijacked session point `host` at an attacker server, leave the password blank, and the server decrypts the stored credential and AUTHs against the attacker host — **exfiltrating plaintext**. Multi-server *multiplies* this (more stored credentials, a per-row edit surface, plus a new create flow).
**Why it happens:** blank = "keep stored password," but the stored password is then used against a *client-changed* host.
**How to avoid:** This phase must NOT silently widen it. Adopt whatever WR-09 posture Rob picks (`.planning/todos/pending/wr-09-blank-keep-host-change-exfil.md`, options 1/2/3). Interim-safest for the *new* code: **option 2** — allow blank-keep only for port/TLS changes; require password re-entry when `host` changes on an existing row (create always requires a password). Flag as an Open Question — do not auto-resolve (it touches locked D-07/D-08).

### Pitfall 5: Deleting a config that history depends on
**What goes wrong:** A hard `DELETE` on a config referenced by *any* campaign throws `SQLITE_CONSTRAINT_FOREIGNKEY` (FK `no action` + `foreign_keys=ON`), so "delete old server" fails confusingly; conversely if you weaken the FK you corrupt history.
**How to avoid:** Soft-delete (`deleted_at`), hide from lists/pickers, keep the row so past campaigns still render sender identity. Add an explicit guard blocking *even soft-delete* when a `queued`/`running` campaign references it (`SELECT count(*) … WHERE smtp_config_id=? AND status IN ('queued','running')`). This satisfies both SC5 clauses.

### Pitfall 6: Partial-unique-index violation during backfill or multi-add
**What goes wrong:** Setting `is_default=1` on two rows for one user throws once the partial unique index exists.
**How to avoid:** Order the migration: add columns → backfill (safe, ≤1 row/user) → create the partial unique index LAST. In app code, always use the transactional `setDefaultSmtpConfig` swap; auto-set `is_default=1` only for the *first* server a user adds.

export function listSmtpConfigsForUser(userId: string) {
  return db.query.smtp_configs.findMany({

export function createSmtpConfig(userId: string, values: PersistableConfig & { label: string }) {
  // verified_at stamped here (only reached after a clean verify), is_default handled by caller/first-add
  return db.insert(smtp_configs)
    .values({ userId, ...values, verified_at: sql`(unixepoch())` })
    .returning();
}

export function updateSmtpConfigById(userId: string, id: number, values: Partial<PersistableConfig> & { label?: string }) {
  return db.update(smtp_configs).set({ ...values, verified_at: sql`(unixepoch())` })
    .where(and(eq(smtp_configs.id, id), eq(smtp_configs.userId, userId), isNull(smtp_configs.deleted_at)))
    .returning({ id: smtp_configs.id });   // length 1 = ok, 0 = IDOR/not-found
}

  return db.update(smtp_configs).set({ deleted_at: sql`(unixepoch())`, is_default: false })

    .returning({ id: smtp_configs.id });
}

## Open Questions

1. **WR-09 posture for the multi-server edit flow** (blocks safe planning of the edit path)
   - What we know: blank-keep + host-change = plaintext exfil on a hijacked session; multi-server multiplies the surface; options 1/2/3 documented.
   - What's unclear: which option Rob accepts; it changes locked D-07/D-08 behaviour.

2. **Deletion model: soft-delete vs restrict-to-unreferenced**
   - What we know: FK `no action` already blocks hard-delete of any referenced config; soft-delete keeps history + allows "removal."
   - What's unclear: does Rob want deleted servers to remain visible anywhere (e.g. greyed in history), or fully hidden?
   - Recommendation: soft-delete, hidden from lists/pickers, still resolvable by id for historical sender-identity rendering.

3. **Default-server UX**
   - What we know: explicit `is_default` is predictable; partial unique index enforces one.
   - What's unclear: auto-promote policy when the default is deleted (most-recently-verified? oldest? force user pick?).
   - Recommendation: auto-promote the most-recently-`verified_at` remaining row; surface a toast so it isn't silent.

4. **Label constraints** — required? unique per user? max length? Recommendation: required, trimmed, 1–60 chars, not necessarily unique (host disambiguates); validate in the shared zod schema.

### Known Threat Patterns for {Next.js Server Actions + BYO-SMTP + SQLite}
| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| IDOR on `smtp_config_id` (picker / delete / set-default / send) | Elevation of Privilege / Information Disclosure | Owner-scoped single-path DAL; client id always re-resolved to a `userId`-owned, non-deleted row |

| Encrypted-triple leak through the enlarged DTO | Information Disclosure | Keep explicit field enumeration in `toSmtpConfigDto`; never spread the row |
| SSRF via user-supplied host on a *new* server add | Information Disclosure / SSRF | Existing `isPrivateHostLiteral` screen in `smtpFormSchema` + per-user verify rate limit (`underVerifyRateLimit`) — reuse per add |
| FK-break corrupting campaign history on delete | Tampering / Repudiation | Soft-delete; never CASCADE/SET NULL; keep FK `no action` |
