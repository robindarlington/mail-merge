# Phase 08.1: Agent access — standalone CLI + MCP server - Context

**Gathered:** 2026-07-18
**Status:** Ready for planning

<domain>
## Phase Boundary

An AI agent or developer can run mail merges without the web UI: an npm-installable
CLI (command `mail-merge`) that reuses lib/core directly, plus a stdio MCP server
(same package) exposing validate-csv / preview-merge / test-send / send as typed
tools. Standalone-local — NO API-token layer, NO web-app coupling beyond lib/core
(decision LOCKED 2026-07-15). Plain-text email only. Mirrors the original
send-credentials.ts run-mode contract (dry-run default, --test ADDR, --send).

</domain>

<decisions>
## Implementation Decisions

### Packaging & npm identity
- **D-01 (amended post-research):** Separate lightweight package at `packages/cli/`
  with its own package.json **and its own lockfile — a STANDALONE sub-package, NOT a
  root npm-workspaces member**. Research finding: a root `workspaces` conversion breaks
  the hardened Dockerfile's `npm ci` (root-manifest-only copy) and hoists the MCP SDK
  tree into the production web image; the standalone layout achieves D-01's actual goal
  (tiny npx install, none of the web app's tree) with ZERO changes to the phase-8
  deploy. lib/core is **esbuild-inlined at bundle time** (same discipline as
  worker.js/migrate.js).
- **D-02:** Published name **`@robindarlington/mail-merge`** (scoped — unscoped
  `mail-merge` assumed taken; scope ties the tool to Rob's brand/portfolio). The bin
  name stays **`mail-merge`**.
- **D-03:** **Publish to the npm registry in this phase** so `npx @robindarlington/mail-merge`
  works for phase 9 collateral. First publish requires Rob's npm login — a QUEUED
  human checkpoint (per autonomy preference: prepare everything, don't block other work).

### Send safety for agents
- **D-04:** MCP live send is gated by a **two-step confirmation token**: the first
  `send` call returns a preview summary (recipient count, subject, from, throttle) plus
  a one-time confirmation token; the send only executes on a second call presenting
  that token. Mirrors the web app's preview→confirm gate.
- **D-05:** **No recipient cap** — the confirm gate and the operator's SMTP provider
  are the limiters. Do not add row caps.
- **D-06:** Default throttle **3000 ms** between sends (the original DELAY_MS),
  overridable (`--delay-ms` / env / MCP param).

### CLI flag/config contract
- **D-07:** **Env for SMTP, flags for files.** SMTP_HOST/PORT/USER/PASS/FROM_ADDR
  (+ optional FROM_NAME) from env or `--env-file`; hidden interactive prompt fallback
  for a missing SMTP_PASS. Password NEVER accepted via argv, never echoed/logged.
- **D-08:** Paths are proper flags: `--csv <file>` and `--template <file>` (no
  hard-coded paths). Template file keeps the original format: first line
  `Subject: ...`, remainder is the body; `{{column}}` placeholders over any CSV column
  (lib/core fillMessage semantics, subject AND body).
- **D-09:** Run modes mirror the original contract: **dry-run default**, `--test ADDR`
  (whole merge redirected to one address), `--send` (live).
- **D-10:** Email column **auto-detected** via lib/core detectEmailColumn with an
  explicit `--email-column` override.

### Send receipts & resume
- **D-11:** Live sends write **JSONL receipts** — one line per recipient:
  `{to, status: "sent"|"failed", messageId|error, timestamp}` — and **`--resume`**
  re-runs a batch skipping addresses already recorded `sent` (file-based analogue of
  the web app's send_records exactly-once idea; no database).
- **D-12:** Receipts are **on by default** for live sends; default path derived from
  the CSV (`mydata.csv` → `mydata.receipts.jsonl`), `--receipts <file>` to relocate,
  `--no-receipts` to disable. MCP `send` uses the same mechanism.

### Claude's Discretion
- MCP tool result shapes / error style, exact flag help text, workspace tooling
  details, test harness layout, npm publish mechanics (docs vs CI), MCP server
  registration snippet format for README.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `lib/core/*` — the entire merge/send engine, pure (nodemailer + papaparse only),
  secret-safe by grep-enforced discipline; built exactly for this reuse.
- `send-credentials.ts` — `promptHidden()` (no-echo password prompt) and the mode
  banner/log-line UX worth mirroring (`[3/42] sent -> addr`, `DRY RUN:`/`TEST mode:`/
  `LIVE mode:`).
- esbuild bundling pattern from `npm run build:worker` (`--bundle --platform=node
  --format=esm --external:...`) — template for the CLI/MCP build.

### Established Patterns
- Secret discipline: password from env/hidden prompt only; never logged; error
  messages carry variable NAMES not values (worker envInt idiom).
- Structured per-recipient outcomes (`{ok,...}`) — receipts (D-11) serialize these.
- TDD with node:test across seams; tests never open real sockets (stub transport,
  as in lib/worker tests + scripts/stub-smtp.ts which can serve as a test sink).

### Integration Points
- Repo root package.json becomes a workspace root (`workspaces: ["packages/*"]`) —
  must NOT disturb the web app's build, the Dockerfile's `npm ci`, or CI; verify the
  Docker build still resolves after the workspace conversion (the Dockerfile copies
  package.json/package-lock.json — workspace hoisting changes lockfile layout).
- Phase 9 consumes this phase's README/examples for the `/agents` page.

</code_context>

## Specific Ideas

- The two-step MCP confirm (D-04) deliberately mirrors the web app's preview →
  confirmation-modal gate: an agent must "see" the preview summary before it can fire.
- Receipts naming derived from the CSV filename so a directory of merges stays
  self-documenting (D-12).
