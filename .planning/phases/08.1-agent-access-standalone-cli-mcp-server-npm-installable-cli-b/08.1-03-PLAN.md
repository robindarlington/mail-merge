<objective>
Third vertical slice: expose the identical merge/send engine to AI agents over a stdio MCP server. Register four tools (validate-csv, preview-merge, test-send, send) on an `McpServer`, gate live `send` behind a two-step confirm token, and dispatch `mail-merge mcp` to a StdioServerTransport. After this slice an agent can discover and drive the whole pipeline programmatically, with the same lib/core paths and the same shared send driver as the CLI.

Purpose: Satisfy SC-3 (MCP tools with typed results reusing lib/core) and the D-04 agent-safety gate, without duplicating any merge/send logic — the tools are thin adapters over lib/core + run.ts.
Output: `mcp.ts` (buildServer factory), the `mcp` subcommand dispatch, and in-process MCP tests (socket-free, InMemoryTransport).
</objective>

## Phase Goal

**As an** AI agent or developer, **I want to** run mail merges from a CLI and an MCP server without the web UI, **so that** I can discover and drive the same lib/core merge-and-send engine locally.

<mcp_input_note>
Per RESEARCH Security Domain: validate-csv and preview-merge take CSV **text** (a `csv` string arg),
NOT a filesystem path — this keeps the stdio server from reaching into the host FS on an agent's
behalf (V4 access control). test-send/send accept the SMTP config + template + CSV text via typed
inputSchema; the SMTP password is supplied through the MCP param object by the operator's own agent
config, never logged, and never echoed back in a tool result.

Receipts on MCP (D-12 adapted to text input): because the CSV arrives as TEXT there is no CSV path
to DERIVE a receipts filename from. So receipts are OPT-IN via an explicit optional `receiptsPath`
string on `send` (and `test-send` for consistency). When `receiptsPath` is provided, runSend writes/
appends the same JSONL receipts and honors the already-sent skip set (resume semantics). When it is
ABSENT, the `send` tool's preview/confirm response AND its final result MUST explicitly carry a
`receiptsWarning` field ("No receipts file will be written for this send.") so the agent knowingly
opts out — D-12's "on by default" intent adapted to the no-path reality of text input. Throttle
(D-06) is likewise an optional `delayMs` param, defaulting to 3000 ms when absent, mirroring the
CLI's `--delay-ms`.
</mcp_input_note>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| agent (stdio JSON-RPC) → MCP server | Untrusted tool arguments cross here; validated by zod inputSchema |
| MCP send tool → user's SMTP | Agent-initiated live delivery — the abuse surface |
| tool params → SMTP password | Password arrives in a tool param object; must never be echoed/logged |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-081-04 | Elevation / abuse | mcp.ts `send` tool | mitigate | Two-step confirm token (D-04): first call returns preview + one-time token, no delivery; delivery only on second call with the token; per-send throttle from run.ts (optional delayMs, default 3000 per D-06) |
| T-081-05 | Input validation | registerTool inputSchema | mitigate | Every tool has a zod inputSchema; malformed args refused by the SDK before the callback runs |
| T-081-02 | Information disclosure | tool results / logs | mitigate | `pass` never in structuredContent, preview, or logs; error `raw` is a string; no console.* of CSV cells/subject/body (actions-core discipline) |
| T-081-FS | Access control | validate-csv / preview-merge / receiptsPath | mitigate | Read tools take CSV TEXT, not a path — the stdio server never reads the host FS on the agent's behalf; the only FS write is the explicit opt-in `receiptsPath` on send/test-send |
| T-081-SDK | Tampering (supply chain) | @modelcontextprotocol/sdk import | mitigate | Pinned 1.29.0 (Plan 01 legitimacy gate); verified 1.29.0 import paths, not the unreleased v2 README |
</threat_model>
