# Phase 08.1 Plan 03: MCP Server (stdio) Summary

**A stdio MCP server (`mail-merge mcp`) exposing the identical lib/core + run.ts engine to AI agents as four typed tools — validate-csv / preview-merge / test-send / send — with live `send` gated behind a two-step confirm token, opt-in JSONL receipts + resume, and the SMTP password structurally absent from every tool result.**

## Performance

- **Duration:** ~30 min
- **Completed:** 2026-07-18
- **Tasks:** 3 (Tasks 1 & 2 TDD RED→GREEN, Task 3 wiring)
- **Files created:** 2 · **modified:** 1

## Accomplishments
- `mcp.ts` `buildServer()` registers four tools on an `McpServer`, all thin adapters over existing code — no merge/send logic duplicated:
  - **validate-csv** — `parseCsv` + `detectEmailColumn` + `countInvalidEmails` → typed `{columns, rowCount, detectedEmailColumn, invalidEmailCount}`.
  - **preview-merge** — per-row `fillMessage` subject/body with an optional row limit → `{rows:[{to,subject,body}]}`.
  - **test-send** — delegates to `runSend({mode:"test"})` against the operator's SMTP; every row to one test address.
  - **send** — delegates to `runSend({mode:"live"})`, gated by a two-step confirm token.
- **Two-step confirm token (D-04):** the first `send` call (no token) returns a preview + a one-time `crypto.randomUUID` token keyed to a sha256 of all send params and delivers NOTHING; the second call with that token consumes it and delivers one-per-row. A replayed, unknown, or param-mismatched token is refused (`isError`, no delivery).
- **D-06 throttle + D-12 receipts:** `test-send`/`send` accept optional `delayMs` (default 3000, threaded into runSend) and optional `receiptsPath`; a supplied path writes the same JSONL receipts and honors the resume skip-set (a re-send with the same path skips already-sent rows), and an absent path surfaces `receiptsWarning:"No receipts file will be written for this send."` in BOTH the preview and the final result.
- **Security:** CSV arrives as TEXT (no host-FS read on the agent's behalf, T-081-FS); every tool has a zod `inputSchema` so malformed args are refused before the callback (T-081-05); the SMTP password is mapped only into runSend's `auth` and never appears in a result/preview/log (T-081-02).
- **`mail-merge mcp`** wired to `StdioServerTransport` with stdout kept clean (verified 0 bytes to stdout on startup); `runSendQuietStdout` reroutes runSend's per-row progress to stderr so a live MCP send cannot corrupt the JSON-RPC stream.
- **Socket-free tests:** in-process `InMemoryTransport` + `Client` harness drives all four tools; send tools proven at the wire against a real spawned `scripts/stub-smtp`.

## Decisions Made
- **SDK malformed-input contract:** `@modelcontextprotocol/sdk@1.29.0` returns an `isError:true` result (with a `validation` message) for a schema-invalid call rather than rejecting the client promise. Confirmed by probing the installed dist; the test asserts `result.isError === true` accordingly.
- **outputSchema only where the shape is fixed:** declared on validate-csv/preview-merge/test-send; omitted on `send` (preview vs. confirmed-send shapes differ). The SDK skips output validation on `isError` results, so error paths are unaffected. `structuredContent` is always returned regardless.
- **Token keyed to params minus the token:** the sha256 covers csv/subject/body/smtp(incl. pass)/from/resolved-delayMs/receiptsPath but not `confirmToken`, so the preview and confirm hashes match and a swapped payload on confirm is rejected.
- **`attempted` = sent+failed** so a full-skip resume reports `attempted:0`.
- **resume-on-receiptsPath:** live `send` sets `resume:true` whenever `receiptsPath` is present (D-12 skip-set parity with the CLI `--resume`).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing critical] Route runSend's stdout progress to stderr inside the MCP send tools**
- **Found during:** Task 2 (implementing test-send/send over run.ts)
- **Issue:** `runSend` (Plan 02) writes per-row progress and banners via `console.log` (stdout). On the real `mail-merge mcp` stdio server, stdout IS the JSON-RPC channel — a live MCP send would interleave progress lines into the protocol stream and corrupt it. The plan's Task 3 note ("do NOT print to stdout") covers only the startup banner, not the send loop.
- **Fix:** Added `runSendQuietStdout(...)` in mcp.ts that swaps `console.log` to `console.error` for the duration of the runSend call (restored in `finally`), so send progress goes to stderr and stdout stays a clean JSON-RPC channel. run.ts (a Plan 02 file) was left untouched — the guard is localized to the MCP adapter.
- **Files modified:** packages/cli/src/mcp.ts
- **Verification:** `mail-merge mcp </dev/null` writes 0 bytes to stdout (captured); the receipts round-trip test performs a real live send through the guard with all assertions green.
- **Committed in:** 202c624 (Task 2 commit)

**2. [Rule 1 - Bug] Malformed-call test aligned to the SDK's isError contract**
- **Found during:** Task 1 (first GREEN run)
- **Issue:** The initial test asserted `assert.rejects(callTool(...))` for a missing required arg, but SDK 1.29.0 returns an `isError:true` result (validation message) instead of rejecting — so the test failed against correct behavior.
- **Fix:** Probed the installed SDK dist to confirm the contract, then changed the test to assert `result.isError === true` and a `/validation/i` message. The behavior itself (malformed args refused before the callback, T-081-05) is intact.
- **Files modified:** packages/cli/test/mcp.test.ts
- **Verification:** Test passes; the same isError-refusal assertion also guards the two-step token wrong/replay cases.
- **Committed in:** 19f8805 (Task 1 GREEN commit)
