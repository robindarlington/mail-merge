# Phase 08.1: Agent access — standalone CLI + MCP server - Research

**Researched:** 2026-07-18
**Domain:** npm-publishable Node CLI + stdio MCP server, reusing an existing pure `lib/core` merge/send engine
**Confidence:** HIGH (stack + API verified against installed package + registry; workspace-risk analysis HIGH; publish mechanics MEDIUM)

## Summary

This phase ships a second consumer of the already-built, dependency-pure `lib/core`
engine: a `mail-merge` CLI and a stdio MCP server, packaged and published as
`@robindarlington/mail-merge`. The engine work is done — `parseCsv`,
`detectEmailColumn`, `fillMessage`, `isValidEmail`, `createSmtpTransport`,
`verifyTransport`, `sendOne`, and `throttle` already exist as pure functions
(nodemailer + papaparse only, zero `@/` imports — verified). The phase is therefore
**wiring + packaging + safety-gating**, not new merge logic.

Two decisions dominate the risk profile. (1) **The MCP TypeScript SDK is at
1.29.0** and the current `main`-branch README describes an *unreleased* v2 with a
different package name (`@modelcontextprotocol/server`) — do not follow it. The
installed 1.29.0 API (`McpServer` + `StdioServerTransport` + `registerTool`) is
verified below and is what the plan must target. (2) **The npm-workspace conversion
(D-01) is the single most destabilizing move** for the just-hardened Docker/Coolify
deploy: the Dockerfile's `npm ci` stages copy only the root `package.json` +
`package-lock.json`, and a workspace root cannot `npm ci` without every member's
`package.json` present — plus workspace hoisting would leak the CLI's MCP-SDK tree
into the production **web** image. A standalone sub-package (own lockfile, esbuild-
bundles `../../lib/core` at build time, **not** a root-`workspaces` member) achieves
D-01's stated goal ("pulls none of the web app's dependency tree") with **zero**
changes to the deployment path.

**Primary recommendation:** Build `packages/cli/` as a **self-contained esbuild
bundle** (like `worker.js`): bundle `lib/core` + `@modelcontextprotocol/sdk` +
nodemailer + papaparse + zod into one ESM file with a `#!/usr/bin/env node` banner,
ship empty runtime `dependencies`, and expose a single `mail-merge` bin that
dispatches `mail-merge mcp` to the stdio server and everything else to the CLI. Use
`node:util.parseArgs` (zero-dep). Keep the package **out of** root `workspaces` to
protect the Phase-8 deploy; flag the D-01 wording divergence for the planner.

The CLI driver and the MCP tools are two front-ends over the **same** `lib/core`
calls — parity is structural, not copied. The only send-orchestration logic new to
this phase (the recipient loop + throttle + receipt append + resume-skip) should live
in **one shared internal module** that both the CLI driver and the MCP `send`/`test-send`
tools call, so the four success criteria can never diverge.

### Pattern 1: stdio MCP server (SDK 1.29.0 — VERIFIED API)
**What:** Register typed tools on `McpServer` and connect over stdio.
**When to use:** the `mail-merge mcp` subcommand.
**Example (import paths + signatures inspected in installed 1.29.0 dist):**
```typescript
// Source: @modelcontextprotocol/sdk@1.29.0 dist/esm/server/{mcp,stdio}.d.ts (inspected)
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({ name: "mail-merge", version: "1.0.0" });

// registerTool(name, config, cb). inputSchema is a RAW ZOD SHAPE object
// ({ key: z.string() }), NOT z.object(...). Verified against the .d.ts overload:
//   registerTool<...>(name, { title?, description?, inputSchema?, outputSchema?,
//                             annotations?, _meta? }, cb)
server.registerTool(
  "validate-csv",
  {
    description: "Parse a CSV and report columns, row count, detected email column, invalid-email count.",
    inputSchema: { csv: z.string().describe("Raw CSV text") },
  },
  async ({ csv }) => {
    const parsed = /* lib/core parseCsv(csv) */ null;
    return { content: [{ type: "text", text: JSON.stringify(parsed) }] };
  },
);

await server.connect(new StdioServerTransport());
```
Notes verified this session: the older `server.tool(name, shape, cb)` overloads are
`@deprecated` in the 1.29.0 types — use `registerTool`. Tool results are
`{ content: [{ type: "text", text }] }`; signal an error with `isError: true` on the
same object. `outputSchema` (also a raw zod shape) additionally emits
`structuredContent` for typed results (success criterion 3 — "typed results").

### Pattern 2: single-bin subcommand dispatch with `parseArgs`
```typescript
// Source: node:util parseArgs (Node 24) — allowPositionals for the subcommand
import { parseArgs } from "node:util";
const argv = process.argv.slice(2);
if (argv[0] === "mcp") { await startMcp(); }         // mail-merge mcp
else {
  const { values } = parseArgs({
    args: argv,
    options: {
      csv:          { type: "string" },
      template:     { type: "string" },
      test:         { type: "string" },                // --test ADDR
      send:         { type: "boolean" },
      "email-column": { type: "string" },
      "delay-ms":   { type: "string" },                // parseArgs has no number type — coerce
      "env-file":   { type: "string" },
      receipts:     { type: "string" },
      "no-receipts":{ type: "boolean" },
      resume:       { type: "boolean" },
      help:         { type: "boolean", short: "h" },
    },
    allowPositionals: true,
  });
  /* dispatch dry-run (default) | --test | --send */
}
```
There is **deliberately no `--smtp-pass` / `--password` option** — omitting it from
the schema means `parseArgs({ strict: true })` (the default) *rejects* an unknown
`--password` flag, which is a testable secret-safety guarantee (success criterion 2).

### Pattern 3: MCP two-step confirm token (D-04)
First `send` call returns a preview summary + a freshly-minted one-time token held in
an in-memory `Map` keyed to the request's parameters' hash; the token is required on
the second call and consumed (deleted) on use. Keep it process-local (the stdio
server is one process per agent session) — no persistence needed. Mirror the web
app's preview→confirm modal semantics.

### Anti-Patterns to Avoid
- **Following the `main`-branch SDK README** (`@modelcontextprotocol/server`, `import * as z from 'zod/v4'`): that is an *unreleased* v2 layout. Target the published 1.29.0 API above.
- **Re-implementing merge/CSV/send in the CLI:** forbidden by CONTEXT canonical-refs — call `lib/core`.
- **Making `packages/cli` a root-`workspaces` member** without the Dockerfile changes in Pitfall 1 — it breaks `npm ci` and bloats the web image.
- **Accepting the SMTP password anywhere in `argv`** — env or hidden prompt only.
- **Building the send loop twice** (once for CLI, once for MCP) — share one `run.ts`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| CSV parsing (quotes, BOM, CRLF) | A comma splitter | `lib/core.parseCsv` | The original `send-credentials.ts` split-at-first-comma is exactly the bug `lib/core` fixed |
| `{{column}}` substitution | A `replaceAll` loop | `lib/core.fillMessage` | Handles spaces-in-keys, `$` literals, unmatched-token passthrough, subject+body |
| Email validity | A regex per call site | `lib/core.isValidEmail` | One predicate keeps counts consistent |
| SMTP transport + TLS + verify | `nodemailer.createTransport` inline | `lib/core.createSmtpTransport` + `verifyTransport` | Explicit `secure` boolean (no port-based inference) is a locked carry-forward fix |
| Per-send error containment | try/catch that aborts | `lib/core.sendOne` (`{ok}` result) | Never-throws contract; receipts serialize it directly |
| MCP JSON-RPC framing / stdio | Manual newline-delimited JSON | SDK `StdioServerTransport` | Protocol version negotiation, batching, cancellation |
| MCP arg parsing/schema | Hand JSON validation | `registerTool` `inputSchema` (zod) | SDK validates + surfaces schema to the agent for discovery |
| Arg parsing | Regex over argv | `node:util.parseArgs` | Strict mode rejects unknown flags for free (secret-safety) |

**Key insight:** Nearly every "hard" part of this phase already exists and is tested
in `lib/core` and the SDK. The phase's real, new surface is *thin*: arg parsing,
secret intake, the receipts/resume file, the confirm-token gate, and the bundle/publish
pipeline. Keep new code confined to those.

## Common Pitfalls

### Pitfall 1: npm-workspace conversion breaks `npm ci` and bloats the web image
**What goes wrong:** Adding `"workspaces": ["packages/*"]` to the root `package.json`
makes `packages/cli/package.json` part of the lockfile graph. The Dockerfile's `deps`
and `prod-deps` stages copy only the **root** `package.json` + `package-lock.json`, so
`npm ci` fails ("cannot install with a lockfile that references a missing workspace
package"). Even if fixed, `npm ci --omit=dev` at a workspace root **hoists and installs
the CLI's dependency tree into the shared `node_modules`** that ships in the production
**web** image — pulling the MCP SDK (and its express/hono/jose transitive tree) into
an internet-facing multi-tenant app that has no use for it.
**Why it happens:** workspace installs are whole-graph by default; the Dockerfile was
hardened around a single root manifest.
**How to avoid (RECOMMENDED — minimal-risk layout):** Make `packages/cli` a
**standalone sub-package with its own `package-lock.json`, NOT listed in root
`workspaces`.** It imports `../../lib/core` relatively and esbuild inlines it at bundle
time, so the published artifact carries no monorepo linkage. Root `package.json`,
`package-lock.json`, and the Dockerfile stay byte-for-byte unchanged. This satisfies
D-01's *stated intent* ("lib/core esbuild-bundled … pulls none of the web app's
dependency tree") — only the "becomes an npm workspace" *mechanism* differs.
**If the workspace mechanism is required anyway** (planner/discuss decision): (a) add
`COPY packages/cli/package.json ./packages/cli/package.json` to **both** the `deps`
and `prod-deps` stages before `npm ci`; (b) scope prod installs so the CLI tree never
enters the web image (`npm ci --omit=dev --workspace=<root>` or prune post-install);
(c) re-run the Phase-8 redeploy acceptance test before merging. This is strictly more
risk than the standalone layout for zero deployment benefit.
**Warning signs:** a red `npm ci` in the Docker build; the web image size jumping; the
redeploy acceptance script failing.
*(This diverges from D-01's literal wording — flagged in Assumptions Log A1 for the
planner to confirm with the user.)*

### Pitfall 2: targeting the unreleased SDK v2 API from the main-branch README
**What goes wrong:** The GitHub `main` README (fetched this session) shows
`import { McpServer } from '@modelcontextprotocol/server'` and `import * as z from
'zod/v4'` — a *different, unpublished* package. Code written to it won't resolve
against the installed 1.29.0.
**How to avoid:** Pin `@modelcontextprotocol/sdk@1.29.0` and use the verified imports
(`@modelcontextprotocol/sdk/server/mcp.js`, `.../server/stdio.js`). Treat any README
example that omits the `/sdk/` path segment as v2 and ignore it.
**Warning signs:** module-not-found on `@modelcontextprotocol/server`.

### Pitfall 3: receipts/resume is at-least-once, not the web app's exactly-once
**What goes wrong:** The web worker gets true exactly-once from an atomic DB row
transition (`pending → sent`). A JSONL file cannot match that: if the process crashes
in the window **after** `sendOne` succeeds but **before** the receipt line is durably
flushed, `--resume` will re-send that one recipient.
**How to avoid:** After each successful send, `appendFileSync` the receipt line **and**
`fsync` it before the next send, shrinking the window to near-zero; document the
residual honestly. Resume builds a `Set` of `to` addresses whose recorded status is
`"sent"` and skips them (file-based analogue of `UNIQUE(campaign,to_addr)`). Do not
claim exactly-once in the README — say "resume skips already-recorded sends."
**Warning signs:** a duplicate `to` line in the receipts file after an interrupted run.

### Pitfall 4: esbuild bundling of the MCP SDK / nodemailer
**What goes wrong:** Naively marking `@modelcontextprotocol/sdk` as an external dep
forces `npx` to install its ~24 MB tree (express/hono/jose) every run; conversely,
bundling can trip on nodemailer's dynamic requires.
**How to avoid:** **Bundle** the stdio-only path — measured this session: `esbuild
--bundle --platform=node --format=esm` of a `McpServer`+`StdioServerTransport`+
`registerTool` entry produced a **454 KB** file with express/hono tree-shaken out (grep
count 2 = residual strings, not code). Follow the `build:worker` recipe; if nodemailer
needs it, reuse the existing `--banner:js` `createRequire` shim. Verify the bundle
actually performs a real send against `stub-smtp` in CI (don't trust the build alone).
**Warning signs:** an unexpectedly large `dist`; a runtime `Dynamic require of "x"`
error.

### Pitfall 5: scoped package publishes as private by default
**What goes wrong:** `npm publish` on a scoped name (`@robindarlington/…`) defaults to
**restricted** (paid) access and errors on a free account.
**How to avoid:** Set `"publishConfig": { "access": "public" }` in
`packages/cli/package.json` and/or publish with `--access public`. First publish is a
human checkpoint (D-03) — `npm whoami` returned ENEEDAUTH this session (verified), so
the plan must queue the login, not attempt it.
**Warning signs:** `402 Payment Required` on publish.

function readSmtpPass(): Promise<string> {
  const fromEnv = process.env.SMTP_PASS;
  if (fromEnv) return Promise.resolve(fromEnv);          // env only
  return promptHidden(`SMTP password (input hidden): `); // never argv, never echoed

## Security Domain

### Applicable ASVS Categories
| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | no | Standalone-local; no API-token layer (LOCKED). The *user's* SMTP auth is the only credential, handled by `lib/core`. |
| V3 Session Management | no | stdio server is one process per session; confirm-token is ephemeral in-memory |
| V4 Access Control | partial | File reads (`--csv`/`--template`/`--receipts`) run with the invoking user's own privileges; validate paths are files, resolve before read |
| V5 Input Validation | **yes** | zod `inputSchema` on every MCP tool; `parseArgs` strict mode on the CLI; `lib/core.parseCsv` handles malformed CSV |
| V6 Cryptography | no | CLI/MCP never encrypts at rest (no DB); SMTP password stays in memory only. TLS to SMTP is nodemailer's explicit `secure` boolean |
| V7 Error handling / logging | **yes** | Never log the SMTP password/auth object (grep-enforced discipline carried from `lib/core`); receipts store `to`/status/messageId/error — never the password |

### Known Threat Patterns for a published Node CLI + stdio MCP
| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| SMTP password leaking via `argv` (visible in `ps`/shell history) | Information disclosure | No password flag in `parseArgs` schema; env or hidden prompt only (SC-2) — testable |
| Password echoed to stdout/logs or into JSONL receipts | Information disclosure | Port `promptHidden` (no-echo); receipts schema excludes secrets; capture-and-grep test |
| Path traversal via `--csv`/`--template`/`--receipts` | Tampering | These are the *operator's own* paths (not attacker-supplied), but resolve + `statSync` isFile before read; MCP `validate-csv`/`preview-merge` take CSV **text**, not a path, avoiding server-side FS reach |
| Agent-driven unattended mass send | Elevation / abuse | D-04 two-step confirm token forces the agent to "see" the preview before firing; per-send throttle (D-06) |
| Slopsquatted / wrong MCP SDK package | Supply chain | Pin `@modelcontextprotocol/sdk@1.29.0`; identity corroborated by inspecting installed dist + github org (A2) |
| Malicious bundle content post-publish | Tampering | `files` whitelist limits shipped surface; publish is a human-reviewed checkpoint (D-03) |
