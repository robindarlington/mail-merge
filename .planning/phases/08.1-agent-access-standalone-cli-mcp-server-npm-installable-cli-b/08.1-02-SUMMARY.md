# Phase 08.1 Plan 02: CLI Live/Test Send Driver Summary

**A `mail-merge --test ADDR` / `--send` CLI that delivers one personalized plain-text email per CSV row over the operator's own SMTP — verified before the loop, throttled between sends, JSONL-receipted, and `--resume`-safe — with the send loop living in ONE `runSend` module the MCP tools will reuse, and the SMTP password structurally absent from argv, logs, and receipts.**

## Accomplishments
- `secrets.ts` — `readSmtpConfig` intakes `SMTP_HOST/PORT/USER/FROM_ADDR` (+ optional `FROM_NAME`) from env, reads TLS EXPLICITLY from `SMTP_SECURE`, and sources the password from `SMTP_PASS` or the ported no-echo `promptHidden`. Missing-var errors name the VAR only; a present password never appears in any thrown message.
- `receipts.ts` — `deriveReceiptsPath` (`data.csv` → `data.receipts.jsonl`), `appendReceipt` (append + `fsync` per row), `readSentSet` (resume set of `sent` addresses; missing file → empty; blank/trailing lines tolerated). Receipt entries are secret-free by construction.
- `run.ts` — `runSend`, the single shared driver: mode banner → `createSmtpTransport` → pre-loop `verifyTransport` gate ("SMTP connection OK") → per-row `fillMessage` → `sendOne` (never throws) → `[i+1/N] sent -> addr` / `FAILED -> addr` line → receipt → `throttle` between sends → `transport.close()`. `--resume` skips recorded `sent` addresses.
- `bin.ts` — dry path unchanged; `--test`/`--send` now intake SMTP via `readSmtpConfig`, resolve the receipts path, and hand off to `runSend`. HELP documents `--test/--send/--receipts/--no-receipts/--resume` and the env-based SMTP intake.
- Integration proof (`send.test.ts`): live send delivers exactly one RCPT per row over a real stub-smtp; `--resume` re-run adds ZERO deliveries (scan finds no duplicate); test mode addresses one address while keeping each row's real subject/body fill; a known `SMTP_PASS` is absent from captured stdout/stderr AND the receipts file.

## Decisions Made
- **Transport-factory seam on `runSend`:** the RCPT-only stub log cannot observe message subject/body, so `createTransport` (default `createSmtpTransport`) lets tests inject a recording transport to assert per-row fill parity and no-secret-leak. Additive optional; the CLI path uses the real default. Also readies Plan 03 MCP-tool testing.
- **Explicit TLS everywhere:** `SMTP_SECURE` drives `secure` (default `false`); the port is never consulted (T-081-TLS carry-forward fix).
- **Resume is at-least-once:** each receipt line is fsynced before the next send, shrinking but not closing the crash-resend window; `readSentSet` skips recorded `sent` addresses. Documented in `receipts.ts`.
