# Phase 08.1: Code Review Report

**Reviewed:** 2026-07-18T21:37:05Z
**Depth:** standard
**Files Reviewed:** 16
**Status:** issues_found

## Summary

Reviewed the standalone CLI + stdio MCP server sub-package (`packages/cli`), cross-referenced
against the `lib/core` contracts it consumes (`lib/core/csv.ts`, `lib/core/send.ts`,
`lib/core/index.ts`). The headline security disciplines mostly hold up: the password is
structurally absent from the argv schema (strict `parseArgs` with no password option), receipts
carry no secret field by construction, per-row failure logging emits only `res.error.message`,
and the confirm-token hash binds the CSV/subject/body/SMTP/delay/receipts params.

Two Critical findings survive that stance-check: (1) the SMTP intake never sets
`requireTLS` when `secure:false` — the exact STARTTLS-stripping defense (`T-2-TLS`) that
`lib/core/send.ts` was extended to provide for onboarding is omitted here, so a
`secure:false` CLI/MCP send can authenticate over cleartext against an active MITM or a
non-TLS server; and (2) `fromName` is excluded from the confirm-token parameter hash, so an
agent can preview one From display name and deliver a different one — a hole in the two-step
gate's "identical parameters" guarantee, aggravated by the preview never showing `fromName`.

Warnings cluster around resume robustness (a torn JSONL line crashes `--resume`; test-mode
resume skips the whole batch), the MCP server's global `console.log` monkey-patch (not
concurrency-safe on the JSON-RPC channel), silently ignored inputs (`--env-file`,
`parseErrors`), and release hygiene (no `prepack` build; failures don't affect exit code).

## Critical Issues

### CR-01: STARTTLS-stripping defense (`requireTLS`) omitted — credentials can travel in cleartext

**Status:** fixed (commit 4b1517c)

**File:** `packages/cli/src/secrets.ts:112-117` (also `packages/cli/src/mcp.ts:64-76`)
**Issue:** `readSmtpConfig` builds the `SmtpConfig` with only `host/port/secure/auth`, and the
MCP `smtpSchema`/`toSmtpConfig` does the same. `lib/core/send.ts:42-44` provides
`requireTLS` precisely so that "a STARTTLS-capable server cannot silently keep the connection
in cleartext (T-2-TLS)" — the project's own onboarding flow treats this as security-critical.
Neither front-end sets it and neither exposes any way to set it. With `secure:false` (the
default when `SMTP_SECURE` is unset, and the normal setting for port 587), nodemailer will
proceed — including AUTH — on a plaintext connection when STARTTLS is stripped by an active
MITM or simply not offered by the server. The README's "Security notes" claim "Explicit TLS"
while the connection can silently be no TLS at all. The user's SMTP password is the asset the
whole package is built to protect.
**Fix:** Default `requireTLS: true` whenever `secure` is `false`, mirroring the onboarding
contract, with an explicit documented opt-out for genuinely plaintext local servers (needed by
the stub-smtp tests):
```ts
// secrets.ts
const smtp: SmtpConfig = {
  host, port, secure,
  auth: { user, pass },
  ...(secure ? {} : { requireTLS: (env.SMTP_REQUIRE_TLS ?? "true").trim().toLowerCase() === "true" }),
};
```
and add an optional `requireTls: z.boolean().optional()` (default `true` when `secure:false`)
to `smtpSchema` in `mcp.ts`, threading it through `toSmtpConfig`. Update README/HELP to match.

### CR-02: `fromName` escapes the confirm-token parameter hash — the two-step gate can be bypassed for a delivered header

**Status:** fixed (commit 54df4b7)

**File:** `packages/cli/src/mcp.ts:83-107` (hash), `packages/cli/src/mcp.ts:303-311` (call site), `packages/cli/src/mcp.ts:348-357` (preview)
**Issue:** `hashSendParams` canonicalizes `csv, subject, body, smtp, from, delayMs,
receiptsPath` — but not `fromName`, even though `fromName` is a `send` input
(`sendInputBase`, line 221) and is delivered to every recipient as the From display name
(`run.ts:83-85` → `formatFrom`). The tool description and D-04 promise that a
"parameter-mismatched token is refused"; in reality an agent can obtain a token by previewing
with `fromName: "Newsletter"` and confirm with `fromName: "IT Security — Urgent"` — the token
matches and the batch delivers with the swapped display name. The preview object also never
includes `fromName`, so the confirming human/agent never sees the value that will be sent.
This is a defect in the security control itself, not just a cosmetic gap: From display-name
spoofing is a standard phishing lever.
**Fix:** Bind it into the hash and surface it in the preview:
```ts
const canonical = JSON.stringify([
  input.csv, input.subject, input.body,
  input.smtp.host, input.smtp.port, input.smtp.secure, input.smtp.user, input.smtp.pass,
  input.from, input.fromName ?? null,          // <-- add
  input.delayMs, input.receiptsPath,
]);
// preview:
preview: { recipientCount, subject, from, fromName: fromName ?? null, delayMs: resolvedDelay, ... }
```
(Update `hashSendParams`'s input type and both call sites to pass `fromName ?? null`.)

## Warnings

### WR-01: `runSendQuietStdout` global `console.log` swap is not concurrency-safe — overlapping tool calls can corrupt the JSON-RPC stdout channel

**Status:** fixed (commit 87e2dc7)

**File:** `packages/cli/src/mcp.ts:118-128`
**Issue:** The stdout guard works by mutating the process-global `console.log` and restoring
it in `finally`. Tool callbacks are async and the MCP SDK does not serialize incoming
requests: if two `send`/`test-send` calls overlap (or any future code logs while a send is in
flight), the first call to finish restores the original `console.log` while the second send is
still looping — its `[i/N] sent -> addr` lines then land on real stdout, corrupting the
JSON-RPC channel and killing the whole agent session. It is also non-reentrant (a nested call
would capture the already-swapped logger as `origLog`). The comment acknowledges stdout is the
protocol channel; the mechanism just isn't robust.
**Fix:** Make the sink explicit instead of patching globals — add an optional
`log?: (line: string) => void` to `RunSendOpts` (default `console.log`), replace every
`console.log` in `run.ts` with `log(...)`, and have the MCP tools pass
`log: (l) => console.error(l)`. Delete `runSendQuietStdout`.

### WR-02: `readSentSet` crashes on a torn/corrupt receipt line — `--resume` fails in exactly the crash scenario it exists for

**Status:** fixed (commit 4381400)

**File:** `packages/cli/src/receipts.ts:71-82`
**Issue:** `JSON.parse(line)` has no error handling. The documented crash window (process
dies mid-`appendFileSync`, receipts.ts:10-14) can leave a truncated final line — e.g.
`{"to":"a@x.com","sta`. On the very next `--resume` run, `readSentSet` throws
`SyntaxError: Unexpected end of JSON input`, the whole run aborts before any send, and the
operator gets a JSON parse error with no pointer to the receipts file. The module tolerates
blank lines but not the far more likely partial-write artifact of the interrupt it is designed
to recover from.
**Fix:** Parse defensively, skipping (or warning about) unparseable lines:
```ts
.reduce((set, line) => {
  try {
    const entry = JSON.parse(line) as ReceiptEntry;
    if (entry.status === "sent" && typeof entry.to === "string") set.add(entry.to);
  } catch {
    // torn line from an interrupted append — ignore; at-least-once semantics re-send it
  }
  return set;
}, new Set<string>());
```

### WR-03: Resume is keyed on the destination address, so `--test --resume` skips the ENTIRE batch after one receipt

**Status:** fixed (commit 9089f26)

**File:** `packages/cli/src/run.ts:139-145` (also `packages/cli/src/receipts.ts:33-41`)
**Issue:** In test mode every row's `to` is `testAddr` (`run.ts:139`), and receipts record
that `to`. Nothing stops `mail-merge --test addr --resume` (args.ts imposes no mode
restriction; bin.ts passes `resume` through for both modes). On a resumed test run,
`sentSet = { testAddr }`, so the loop's `sentSet.has(to)` matches for every row — all rows
are "skip (already sent)" including the ones never proofed. Result: `Done. 0/3 sent, 3
skipped.` with exit code 0 — a silently wrong outcome. Receipts also cannot distinguish which
row a test-mode receipt belongs to, so resume is structurally meaningless in test mode.
**Fix:** Either reject the combination in `args.ts` / `runSend`
(`if (mode === "test" && resume) throw new Error("--resume cannot be combined with --test")`)
or record row identity (e.g. the row's own email cell) in test-mode receipts and key the skip
set on that.

### WR-04: CLI exits 0 even when every send failed; summary line hides the failure count

**Status:** fixed (commit 67dd87f)

**File:** `packages/cli/src/bin.ts:148-161`, `packages/cli/src/run.ts:186-188`
**Issue:** `main()` discards `runSend`'s `RunSendResult`; the process exit code is 0 unless an
exception was thrown. A `--send` run where all N recipients FAILED (auth ok, every
`sendMail` rejected) prints per-row `FAILED ->` lines, then `Done. 0/N sent.`, then exits 0 —
cron jobs, CI, and shell scripts (`&&` chains) see success. The `Done.` summary also omits
`failed` entirely (`${sent}/${total} sent${skipped…}`), so the one line a script or human
scans last never mentions failures.
**Fix:** In `bin.ts`, capture the result and set the exit code; include failures in the
summary:
```ts
const result = await runSend({ ... });
if (result.failed > 0) process.exitCode = 1;
```
and in `run.ts`:
```ts
console.log(`\nDone. ${sent}/${total} sent${failed ? `, ${failed} FAILED` : ""}${skipped ? `, ${skipped} skipped` : ""}.`);
```

### WR-05: CSV structural `parseErrors` are ignored by both front-ends — including the dedicated `validate-csv` tool

**Status:** fixed (commit 8b611f5)

**File:** `packages/cli/src/mcp.ts:157-174` (validate-csv), `packages/cli/src/mcp.ts:192-211`, `packages/cli/src/bin.ts:116`
**Issue:** `lib/core.parseCsv` returns `parseErrors: Papa.ParseError[]` (quote errors,
field-count mismatches — `lib/core/csv.ts:27`). No caller in this package reads it. A ragged
or mis-quoted CSV silently yields rows with missing/shifted cells: merge tokens fill with
empty strings, `to` can be `""`, and `send` will happily attempt delivery. Worst is
`validate-csv` — the tool whose entire job is to tell an agent whether the CSV is sound
reports `columns/rowCount/invalidEmailCount` but not that the parse itself produced structural
errors, so an agent gets a green light on a malformed file.
**Fix:** Add `parseErrorCount: z.number()` (and optionally a capped `parseErrors` message
list) to the `validate-csv` output and return `parsed.parseErrors.length`; in `bin.ts` and
the `send`/`test-send` paths, warn (or refuse in live mode) when `parseErrors.length > 0`.

### WR-06: `--env-file` is accepted and silently ignored

**Status:** fixed (commit 2d409cf)

**File:** `packages/cli/src/args.ts:67` (schema), `packages/cli/src/args.ts:108`, `packages/cli/src/bin.ts` (never read)
**Issue:** The strict `parseArgs` schema deliberately rejects unknown flags — yet it includes
`"env-file"`, which no code consumes (`opts.envFile` is never referenced in `bin.ts`), and
which appears in neither `HELP` nor the README. A user who runs
`mail-merge --send --env-file .env …` reasonably believes their `.env` was loaded; it wasn't.
Best case they get `Missing env var SMTP_HOST`; worse case `SMTP_*` vars are inherited from
the shell but `SMTP_SECURE`/`FROM_NAME` overrides in the `.env` are silently dropped and the
send proceeds with different settings than intended. A silently ignored security-adjacent
flag is a trap.
**Fix:** Remove the option from the schema (strict mode will then reject it with a clear
error), or implement it (parse the file and merge into the env passed to `readSmtpConfig`),
or throw with guidance: `"--env-file is not supported; run via node --env-file=.env"`.

### WR-07: Confirm tokens never expire and the token map grows without bound; `createdAt` is dead

**Status:** fixed (commit 73a90b4)

**File:** `packages/cli/src/mcp.ts:141`, `packages/cli/src/mcp.ts:346-347`
**Issue:** Every preview call mints a token into `confirmTokens`; entries are deleted only
when successfully confirmed. Unconfirmed tokens live for the whole stdio session — an agent
that previews in a loop grows the Map indefinitely, and a token minted hours ago (params long
since stale in the conversation) still fires a live batch. `createdAt: Date.now()` is stored
but never read anywhere — dead code that signals an intended-but-missing TTL.
**Fix:** Enforce a TTL and purge on access:
```ts
const CONFIRM_TTL_MS = 10 * 60 * 1000;
// on every send call:
for (const [t, rec] of confirmTokens) {
  if (Date.now() - rec.createdAt > CONFIRM_TTL_MS) confirmTokens.delete(t);
}
```
and treat an expired token the same as an unknown one (the existing error message already says
"expired").

### WR-08: `test-send` never surfaces the no-receipts warning its own spec (D-12) claims it does

**Status:** fixed (commit a80f9bd)

**File:** `packages/cli/src/mcp.ts:42-43` (constant doc), `packages/cli/src/mcp.ts:242-276` (test-send)
**Issue:** The constant is documented as "the exact no-receipts warning surfaced when
`send`/`test-send` gets no path (D-12)", and README says both tools take the same
`receiptsPath` semantics — but the `test-send` result never includes `receiptsWarning`
(its declared `outputSchema` at lines 242-247 has no such field, so it could not legally
return it). An agent calling `test-send` without `receiptsPath` gets no signal that nothing
was recorded, while `send` does. Spec and implementation diverge inside the same file.
**Fix:** Add `receiptsWarning: z.string().optional()` to the `test-send` `outputSchema` and
mirror the `send` result: `...(receiptsPath ? {} : { receiptsWarning: NO_RECEIPTS_WARNING })`.
(Or, if the divergence is intentional, correct the constant's doc comment and README.)

### WR-09: No `prepack`/`prepublishOnly` build — `npm publish` can ship a stale or missing `dist/`, i.e. a broken `mail-merge` bin

**Status:** fixed (commit dc3ab5c)

**File:** `packages/cli/package.json:8-21`
**Issue:** `bin` points at `dist/bin.js` and `files` publishes only `dist` + README, but
nothing ties `npm publish` to `npm run build` (`dist/` is even gitignored, so a fresh clone
has none). Publishing without a manual build ships either no `dist` at all (npm silently
omits the missing dir; `npx @robindarlington/mail-merge` then fails with a dead bin path) or
— more insidiously — a stale bundle that predates fixes like CR-01/CR-02.
**Fix:**
```json
"scripts": {
  "prepack": "npm run build",
  ...
}
```

## Info

### IN-01: Test harness duplication and pid-derived fixed ports

**File:** `packages/cli/test/mcp.test.ts:64-103`, `packages/cli/test/send.test.ts:63-102`
**Issue:** `withStubSmtp`/`readRcptAddrs` are copy-pasted between the two files (only the
port base differs), and `20000/30000 + pid % 10000` can collide with an unrelated process
already bound to that port, failing the suite spuriously.
**Fix:** Extract a shared `test/helpers.ts`; prefer `--port 0` + parse the actual port from
the stub's "listening on" line if the stub supports it.

### IN-02: Server version string duplicates `package.json`

**File:** `packages/cli/src/mcp.ts:136`
**Issue:** `version: "1.0.0"` is hardcoded; it will silently drift from the published package
version on the first bump.
**Fix:** Import/inject the version (esbuild `define`, or read from a generated constant).

### IN-03: `promptHidden` trims the password and can hang on closed stdin

**File:** `packages/cli/src/secrets.ts:59-63`
**Issue:** `answer.trim()` silently alters passwords with genuine leading/trailing
whitespace, and if stdin closes (Ctrl+D) before an answer, the promise never settles — the
process hangs. Both are inherited from `send-credentials.ts`, but they are now in a published
tool.
**Fix:** Drop the `trim()` (or trim only trailing `\r\n`); add `rl.on("close", ...)` to
reject/resolve when the interface closes without an answer.

### IN-04: A `Subject:`-only template with no trailing newline is rejected with a misleading error

**File:** `packages/cli/src/template.ts:36-38`
**Issue:** The regex `/^Subject:\s*(.*)\r?\n/i` requires a newline after the subject line, so
a file containing exactly `Subject: Hi` fails with `--template must start with a "Subject:
..." line` even though it does.
**Fix:** Allow EOF: `/^Subject:[ \t]*(.*)(?:\r?\n|$)/i` (or improve the error text).

### IN-05: `send` is the only tool without an `outputSchema`

**File:** `packages/cli/src/mcp.ts:280-292`
**Issue:** The other three tools declare typed output schemas for discovery; `send` returns
two distinct shapes (preview vs. result) and declares none, so clients get no structured
contract for the most consequential tool.
**Fix:** Declare a union-ish schema (all fields optional: `preview?`, `confirmToken?`,
`attempted?`, `sent?`, `failed?`, `receiptsPath?`, `receiptsWarning?`) or split preview into
its own tool.

### IN-06: `send` preview's `recipientCount` ignores the resume skip-set and invalid emails

**File:** `packages/cli/src/mcp.ts:348-357`
**Issue:** With `receiptsPath` set, the confirming call runs with `resume: true`
(`mcp.ts:334`), so a preview can say `recipientCount: 500` while the confirmed send delivers
0 (all previously sent) — the number the agent confirms is not the number that will be
attempted. Rows with invalid/empty emails are also counted.
**Fix:** Compute the preview count after applying `readSentSet(receiptsPath)` and report
`alreadySent`/`invalidEmailCount` alongside it.
