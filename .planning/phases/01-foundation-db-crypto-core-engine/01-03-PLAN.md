<objective>
Build the AES-256-GCM credential-encryption helper (SMTP-04) that Phase 2 uses to encrypt SMTP passwords at onboarding and Phase 6's worker uses to decrypt them at send time. This plan delivers Phase 1 success criterion #2: a value encrypted with a runtime-injected key round-trips to plaintext, the key lives only in the environment (never the repo or DB volume), and the helper fails closed if the key is missing or malformed.

Purpose: This is the security linchpin of the whole product (PITFALLS #1/#2 — bulk credential leak is the worst-case failure). Done as a dedicated TDD plan because correctness (unique IV, auth-tag verification, fail-closed key handling, no secret leakage) is precisely expressible as tests before implementation.
Output: `lib/crypto/key.ts` (fail-closed key loader), `lib/crypto/index.ts` (encrypt/decrypt), and a thorough `crypto.test.ts`.
</objective>
