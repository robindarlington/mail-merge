# Phase 7 — UI Design Contract

> Visual and interaction contract for per-row attachments: the **Attachments card** on `/compose` (multi-file upload + attachment-column picker + server-computed match summary), the **blocking presence/size validation** at the confirm-send gate, and the **attachment trace** in the campaign detail results table and results CSV. Generated by gsd-ui-researcher (autonomous mode — user offline; judgment calls recorded in Assumptions), verified by gsd-ui-checker.

This phase adds one new compose-page section and extends two existing surfaces. Locked decisions from 07-CONTEXT.md shape everything here:

1. **`/compose` — a new "Attachments (optional)" `Card`** between the compose card and the preview stepper. The user uploads files with a multi-file picker (no zip in v1), then confirms/overrides an **auto-detected attachment column** — the exact confirm-column UX of the CSV uploader's email-column step (CONTEXT.md: "Follow the CSV upload flow's confirm-column UX as the closest analog"). A server-computed **match summary** shows, per row, whether the cell's filename matched an uploaded file. Rows with an empty cell send without an attachment (not an error); rows referencing a file that was never uploaded are a **blocking error**.
2. **The confirm-send dialog** (`components/campaign/confirm-send-dialog.tsx`) — the server-authoritative summary gains attachment lines: a count line, the row-1 sample's attachment, an all-matched success check, and a **blocking destructive block** (confirm button disabled, exactly like the existing `noneSendable` gate) when files are missing or a row exceeds the per-message size limit.
3. **The campaign detail results table** (`components/campaign/recipient-results-table.tsx`) — a new **"Attachment"** column showing each row's original filename, and a mapped friendly reason for the worker's send-time `rejected: attachment missing` failure. The results CSV export gains the same column.

**Limits (LOCKED, centralized constants):** 10 MB per file (enforced at upload — an oversized file is rejected and never stored) and 15 MB of attachments per message (summed per row at validation time — a violation blocks the send). Both appear verbatim in copy below.

**Trust boundaries (LOCKED):** files are stored under opaque server-generated IDs — the UI never sees or sends a storage path; the **original filename is display-and-match only** and renders as escaped JSX text everywhere (it is user-supplied, exactly as untrusted as a CSV cell — the T-06-15 stored-XSS rule applies). Match counts, presence checks, and size sums are **server-computed** (extending `previewCampaign`/`buildConfirmSummary`); the client displays them and never derives them from sample rows — the same server-authoritative discipline as `invalidCounts` (Phase 3) and `ConfirmSummary` (Phase 5).

**This phase has ONE low-stakes destructive-adjacent action** — removing an uploaded file from the pre-send list. It is fully recoverable (re-upload) and pre-commitment (nothing has been sent), so it gets **no confirmation dialog** (Assumption W5). The phase's real gate is the existing undismissable confirm-send dialog, which this phase extends but does not restructure.

It inherits the design system, spacing, typography, and color contract established in **02-UI-SPEC.md**, carried verbatim through **03-/04-/05-/06-/06.1-UI-SPEC.md**. The shell is unchanged (`app/(app)/layout.tsx` — 640px `max-w-2xl` content column, `p-8`, sidebar + top bar + footer). No new palette tokens, sizes, weights, spacing values, shadcn components, or npm packages are introduced.

Blocking = destructive; skippable = muted. Empty attachment cells are the skippable kind → muted.

## Interaction & State Contract

### Attachments card (`/compose`)

- **Placement:** a new `Card` directly after the compose ("Compose your email") card and before the `PreviewStepper`, inside `ComposeEditor`'s existing `gap-6` column (Assumption W1). It renders only when a recipient list is selected (`selectedId` truthy) — the column picker needs the list's columns (Assumption W2).
- **Upload flow mirrors csv-uploader:** `Input type="file" multiple` (no `accept` restriction — any file type is attachable; Assumption W13) → explicit accent "Upload files" submit → server action stores each file under an opaque ID and returns the updated file list (id, original filename, size). Repeated uploads accumulate. No drag-and-drop in v1 (deferred with zip ingestion).
- **Per-file 10 MB is checked client-side with the shared schema AND re-checked server-side** (the T-4-CLIENTVAL no-divergence rule); an oversized or duplicate-named file is rejected with a field-anchored error while the rest of the batch proceeds (Assumption W11).
- **Filename matching is server-computed**: trim whitespace, case-insensitive compare of the row's cell against uploaded original filenames (Assumption W12); consequently a filename differing only by case from an existing upload is rejected as a duplicate. The client renders the returned summary; it never re-derives matches from the 5 cosmetic sample rows.
- **Column auto-detect mirrors the email-column pattern** (Phase 3): a likely column (`attachment`, `file`, `filename`, or similar) is pre-selected; the user confirms or overrides via the `Select`. There is **no "none" option** — to send without attachments, remove the uploaded files (empty file list = attachments off; Assumption W4).
- **Switching the recipient list** keeps uploaded files but resets the column selection and re-runs auto-detect + matching against the new list's columns (Assumption W3).
- **Empty cells are fine** (LOCKED): a row with an empty attachment cell sends without one — surfaced as the muted "the rest will send without one" clause, never an error.
- **Blocking states surface twice, block once:** the compose card shows missing-file / over-size errors immediately (early feedback), and the confirm dialog re-checks server-side and **disables the confirm button** (the actual gate — server-authoritative, exactly like `noneSendable`). "Review and send" itself stays enabled so the user reaches the authoritative summary; the dialog is where sending is impossible.
- **Remove** deletes the uploaded file record (and its stored bytes) for this draft; matching recomputes; a previously-matched row may become missing → the blocking error appears. No confirmation (pre-send, recoverable).

### Confirm gate

- The dialog's structure, undismissability (`showCloseButton={false}` + prevented outside/escape), stale-summary guard, and footer are **unchanged**. Attachment data arrives inside the extended `ConfirmSummary` — the client computes nothing.
- Confirm disabled conditions extend from `loading || !summary || noneSendable || submitting` to also include `summary.missingAttachmentCount > 0 || summary.oversizeRowCount > 0`.
- Missing-filename lists in the dialog and the compose card cap at 5 names + "and {k} more." (Assumption W6).

### Send path & history

- The worker resolves attachments **by DB id → opaque storage path only** (LOCKED — never a CSV-provided path); a file missing on disk at send time fails that one row with the stored reason `rejected: attachment missing` and the campaign continues (poison-pill rule, Phase 6).
- `recipient-results-table.tsx` maps the `rejected: attachment missing` prefix to the friendly reason copy, exactly as it maps `interrupted:` today — no new status enters the Phase 6 Status Vocabulary; this is a `failed` reason variant with **destructive** treatment (the send definitively did not happen — unlike interrupted, there is no uncertainty; Assumption W8).
- The attachment filename shown per result row comes from the `attachments` table's `filename` column (display-only), rendered as escaped JSX text; the export CSV includes it with the existing formula-injection guard.
- All reads stay userId-scoped (AUTH-02): attachments inherit tenancy via the campaign FK; an upload, list, or remove call for another tenant's draft resolves to not-found.
