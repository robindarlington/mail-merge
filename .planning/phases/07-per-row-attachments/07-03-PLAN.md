<objective>
Make attachments a first-class part of the server-authoritative confirm gate: stamp the campaign onto the user's pending uploads when the draft is created, recompute attachment match/presence/size numbers inside `buildConfirmSummaryCore` (never trusting the client), and BLOCK `enqueueCampaignCore` server-side when a referenced file is missing or a row exceeds the 15 MB per-message limit — closing success criterion #2 (blocking validation before send).

Purpose: The confirm gate is the single server-authoritative choke point already used for invalid-email/gaps validation (T-5-TAMPER). Attachment validation slots in exactly there so the "you cannot send until every named file is present" guarantee is enforced by the server, not merely dimmed in the UI.
Output: extended `lib/campaign/actions-core.ts` (prepare stamp + ConfirmSummary fields + enqueue block) with tests.
</objective>

## Phase Goal (user story)

**As a** signed-in sender, **I want to** attach a different file to each row of my CSV — with every referenced file validated as present and resolved safely — **so that** I can send personalized emails that each carry the right per-recipient file with confidence nothing is missing or unsafe.

In prepareCampaignCore, after the draft campaign is created and its id known, `await stampCampaignOnPendingAttachments(userId, campaignId)` (so the pre-campaign uploads are now owned by this campaign — PATTERNS "Prepare-time stamping").
