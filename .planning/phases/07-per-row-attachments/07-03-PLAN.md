<objective>
Make attachments a first-class part of the server-authoritative confirm gate: IDEMPOTENTLY stamp the campaign onto the user's uploads when the draft is created (so re-opening the send dialog never strands files on an abandoned draft), recompute attachment match/presence/size numbers inside `buildConfirmSummaryCore` via the SHARED `computeAttachmentMatch` (never trusting the client), and BLOCK `enqueueCampaignCore` server-side when a referenced file is missing or a row exceeds the 15 MB per-message limit — closing success criterion #2 (blocking validation before send).

Purpose: The confirm gate is the single server-authoritative choke point already used for invalid-email/gaps validation (T-5-TAMPER). Attachment validation slots in exactly there so the "you cannot send until every named file is present" guarantee is enforced by the server, not merely dimmed in the UI. CRITICAL re-prepare fix (BLOCKER 1): `components/campaign/confirm-send-dialog.tsx` calls `prepareCampaign` on EVERY dialog open, minting a NEW draft campaign each time. If the stamp only claimed campaign_id-IS-NULL rows, the first open would move the uploads onto draft C1, and re-opening (C2) would find nothing to stamp — the confirm summary for C2 would show every file missing and the block could never clear, and cancel+reopen would appear to lose all attachments. The idempotent stamp (Plan 02) claims unstamped OR still-draft-owned rows, so each re-prepare re-points the same uploads onto the current draft.
Output: extended `lib/campaign/actions-core.ts` (idempotent prepare stamp + ConfirmSummary fields via the shared matcher + enqueue block) with tests.
</objective>

## Phase Goal (user story)

**As a** signed-in sender, **I want to** attach a different file to each row of my CSV — with every referenced file validated as present and resolved safely — **so that** I can send personalized emails that each carry the right per-recipient file with confidence nothing is missing or unsafe.
