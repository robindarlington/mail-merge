<objective>
Build the tenancy backbone, the upload endpoint, and the compose-time match seam for attachments: a userId-scoped DAL with the AUTH-02 owner-filter discipline (two-tenant IDOR test) whose stamp is IDEMPOTENT across re-prepares and whose send-time resolver follows the inverted attachment_id link; the shared `computeAttachmentMatch` matcher plus the `actions-core` seams + thin `"use server"` wrappers (upload one file per call, list, delete, confirm-column, AND `matchAttachments` for the compose card); the persistence of the user's chosen attachment column onto `recipient_sets`; and the `next.config.ts` bodySizeLimit bump that lets a 10 MB file actually reach the action.

Purpose: The pre-campaign upload window has no campaign FK to inherit tenancy from, so the DAL must carry a direct `userId` owner scope on every read/write — this is the structural IDOR defense for attachments (T-07-IDOR). Because the confirm dialog re-prepares a NEW draft campaign on every open (`components/campaign/confirm-send-dialog.tsx`), the stamp MUST be idempotent — claiming a user's unstamped uploads AND any still owned by their draft campaigns — or a re-open strands the files on an abandoned draft and the missing-file block can never clear. The compose card needs a server-computed match summary BEFORE any campaign exists, so `matchAttachmentsCore` re-reads the stored CSV and matches against the user's pending uploads using the SAME `computeAttachmentMatch` the confirm gate uses (Plan 03) — one matcher, zero divergence. Uploads go one-file-per-action because a 10 MB attachment cannot pass the current 4 MB Server-Action body limit.
Output: `lib/data/attachments.ts` (+ two-tenant test), a recipient_sets attachment-column setter, `lib/attachments/{match,actions-core,actions,index}.ts` (+ match & core tests), `next.config.ts` bumped — all green.
</objective>

## Phase Goal (user story)

**As a** signed-in sender, **I want to** attach a different file to each row of my CSV — with every referenced file validated as present and resolved safely — **so that** I can send personalized emails that each carry the right per-recipient file with confidence nothing is missing or unsafe.
