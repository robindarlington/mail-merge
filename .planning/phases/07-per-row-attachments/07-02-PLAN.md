<objective>
Build the tenancy backbone and the upload endpoint for attachments: a userId-scoped DAL with the AUTH-02 owner-filter discipline (two-tenant IDOR test), the `actions-core` seams + thin `"use server"` wrappers that store one file per call under an opaque id, the persistence of the user's chosen attachment column onto `recipient_sets`, and the `next.config.ts` bodySizeLimit bump that lets a 10 MB file actually reach the action.

Purpose: The pre-campaign upload window has no campaign FK to inherit tenancy from, so the DAL must carry a direct `userId` owner scope on every read/write — this is the structural IDOR defense for attachments (T-07-IDOR). Uploads go one-file-per-action because a 10 MB attachment cannot pass the current 4 MB Server-Action body limit (07-PATTERNS second finding). The attachment column is persisted like `email_column` so the confirm gate and worker never silently re-detect and drop the user's override.
Output: `lib/data/attachments.ts` (+ two-tenant test), a recipient_sets attachment-column setter, `lib/attachments/{actions-core,actions,index}.ts` (+ core test), `next.config.ts` bumped — all green.
</objective>

## Phase Goal (user story)

**As a** signed-in sender, **I want to** attach a different file to each row of my CSV — with every referenced file validated as present and resolved safely — **so that** I can send personalized emails that each carry the right per-recipient file with confidence nothing is missing or unsafe.
