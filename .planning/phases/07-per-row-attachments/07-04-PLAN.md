<objective>
Wire the send path so the worker attaches the right file per recipient — resolving strictly by the send_record's own `attachment_id` → opaque storage path, verifying on-disk presence, and failing a single row `rejected: attachment missing` through the EXISTING fenced transaction if the file vanished (never a throw that aborts the batch). Extend `lib/core/send.ts` additively so a no-attachment send is byte-for-byte unchanged, and stamp the send_record→attachment link at materialize time on the INVERTED FK (`send_records.attachment_id`), so a file referenced by many CSV rows links every one of them.

Purpose: This closes success criterion #1 (worker attaches the correct file per recipient) and the send-time half of #2/#3 (safe resolution + graceful per-row failure). The inverted link (Plan 01) is what makes a shared filename correct — stamping `send_records.attachment_id` per matched row means 300 rows referencing `welcome.pdf` each carry it, where the old per-attachment `send_record_id` would have carried it on only the last row. It must preserve every Phase 6 invariant: per-row synchronous-transaction commits fenced on `status='sending'`, and the poison-pill rule that one bad row never crashes the campaign.
Output: extended `lib/core/send.ts`, `lib/worker/materialize.ts`, `lib/worker/process.ts` + updated `process.test.ts`.
</objective>

## Phase Goal (user story)

**As a** signed-in sender, **I want to** attach a different file to each row of my CSV — with every referenced file validated as present and resolved safely — **so that** I can send personalized emails that each carry the right per-recipient file with confidence nothing is missing or unsafe.
