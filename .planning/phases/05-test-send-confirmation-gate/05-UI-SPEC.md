# Phase 5 — UI Design Contract

> Visual and interaction contract for the whole-batch test-send control and the hard live-send confirmation gate. Generated by gsd-ui-researcher (autonomous mode), verified by gsd-ui-checker.

This phase adds NO new route and NO new top-level surface. It extends the existing **Compose** page (`app/(app)/compose/`) with a **Send** section below the editor: a **test-send panel** (address input + "Send test batch" action that routes the whole personalized batch to one address, with progress/duration expectations) and a **"Review and send"** action that opens the **live-send confirmation modal** (shadcn `Dialog`). The modal shows a server-authoritative review — recipient count, sender identity, one merged sample recipient, and validation warnings — behind an explicit confirm action, and ends at the campaign being **queued** (the live send itself is Phase 6). It inherits the design system, spacing, typography, and color contract established in **02-UI-SPEC.md** and carried verbatim through **03-UI-SPEC.md** and **04-UI-SPEC.md**. Consistency with the SMTP wizard's test-send step (`components/smtp/step-test-send.tsx` — address input + verify banner + typed-failure Alert + spinner CTA) and the compose editor (`components/compose/compose-editor.tsx` — RHF+zod, field-anchored errors, one-accent discipline, sonner toasts) is the primary design constraint.

**This modal IS a destructive-action-class confirmation.** Enqueuing a live send is an irreversible commitment of real email over the user's SMTP. The confirmation-pattern rules that Phases 2–4 explicitly deferred (04-UI-SPEC.md Assumption U9: "no destructive actions this phase") are applied here for the first time: the gate cannot be dismissed accidentally, states its irreversible consequence in plain copy, and requires an explicit confirm action.

## Typography

Inherited verbatim from 02-UI-SPEC.md. Exactly 4 sizes, 2 weights (400 regular, 600 semibold). No monospace — the merged sample renders in Geist sans, plain text.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px (`text-base`) | 400 | 1.5 |
| Label | 14px (`text-sm`) | 400 (form labels, help text, warning rows, review-summary rows, per-message failure lines, duration note) | 1.5 |
| Heading | 20px (`text-xl`) | 600 | 1.2 |
| Display | 28px (`text-[28px]`) | 600 | 1.2 |

- The Send card title ("Send") uses Heading (20px, `text-xl`, `CardTitle`), matching the editor card's "Compose your email".
- The confirm-panel sub-title inside the test-send area ("Send a test batch") uses Heading (20px) OR the wizard's `h2 text-xl font-semibold leading-[1.2]` idiom (see `step-test-send.tsx`) — same size/weight, either markup.
- Help text, the duration warning, the modal's review rows, warning lines, and the send-result summary use Label (14px, `text-sm`); muted variants use `text-muted-foreground` at Label size — never a new size.
- The merged sample-recipient body in the modal renders at Body (16px, `text-base`), preserving line breaks (`whitespace-pre-wrap`) — a faithful plain-text render, never `dangerouslySetInnerHTML`.
- **Accepted library-default exception:** `DialogTitle` ships `font-heading text-base leading-none font-medium` (16px / weight 500) and `Dialog` content is `text-sm`. These are shadcn primitive defaults — accepted as-is exactly like the Button/Label `font-medium` (500) exception recorded in 04-UI-SPEC.md. Do NOT introduce 500 or 16px-titles in hand-written classes; only the primitive keeps its default.

## Color

All values are existing `app/globals.css` tokens (oklch). Reference by semantic name, never raw values. No new palette tokens this phase.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--background` oklch(1 0 0) | Page background, main surfaces, test-address input |
| Secondary (30%) | `--card` / `--muted` / `--border` / `--popover` / `--sidebar` | Send card, dialog surface (`bg-popover`), `DialogFooter` (`bg-muted/50`, library default), merged-sample surface (`bg-muted`), review-row separators, technical-detail `pre` (`bg-muted`) |
| Accent (10%) | `--primary` oklch(0.205 0 0) | See reserved list below |
| Destructive | `--destructive` oklch(0.577 0.245 27.325) | FAILURE alerts only: test-send failure (`Alert variant="destructive"`, mirrors `step-test-send.tsx`), enqueue failure, and field-anchored "enter a test address" on a blank/invalid test send. NOT used for the confirm button. See note below |
| Success (semantic, minimal) | `--success` oklch(0.627 0.17 149) — existing token, do NOT redeclare | ONLY: the "connection verified" style positive indicators (`CheckCircle2` + text), the all-clear validation lines in the modal ("all rows have a valid email address"), the test-send success toast icon, and the "queued" success state |

Accent (`--primary`) reserved for:
1. **The single forward CTA visible per view.** On the compose surface the forward accent is **"Review and send"** in the Send card. Phase 4's **"Save template"** is accent ONLY while the compose form is dirty-and-valid; once the composed content is saved (form clean / not dirty) that button is disabled and visually recedes, so **at most one accent button is bright at a time** (Save while editing → Review and send once saved). See Assumption U1.
2. **The confirm modal's single confirm action: "Send to {N} recipients".** One accent button in the footer; the "Cancel" action is `outline`/`ghost`.
3. Active sidebar nav indicator (inherited — "Compose" route).
4. Focus rings (via `--ring`) on the test-address input, all buttons, and the modal's focusable controls.

Everything else uses `secondary` / `ghost` / `outline`: **"Send test batch"** is `outline`/`secondary` (a supporting safety action, never accent), the modal "Cancel", and "Show technical details" collapsible triggers. **Never style all interactive elements with accent** — the test-send action and the confirm-modal actions must not both read as bright primary.

**The confirm button is `--primary`, NOT `--destructive`.** Enqueuing a live send is the user's intended forward action, not a data-destroying one; the red destructive token stays reserved for FAILURE states (a send/enqueue that could not proceed), consistent with the Phase 2–4 discipline. The irreversibility of the send is communicated through **friction and copy** (undismissable gate + explicit consequence line + count-in-label confirm), not through a red button. See Assumption U3.

**Warning classification (inherited verbatim from 04-UI-SPEC.md — the modal reuses the exact treatment):**
- **Invalid emails** and **empty merge values** are informational / non-blocking: neutral `text-muted-foreground` + lucide `AlertCircle`. Those rows are skipped / that spot is blank — an expected outcome, never destructive.
- **Unknown tokens** (`{{typo}}` not in the recipient list's columns) are a prominent authoring warning: `text-foreground` (not muted) + lucide `AlertTriangle`, surfaced near the top of the modal's warning block. Prominent but still NOT destructive, and — per Assumption U4 — they do NOT hard-disable the confirm button.

Interaction rules (confirmation-pattern rules — applied here for the first time in the project):
- **The confirm modal is undismissable by accident.** It renders with `showCloseButton={false}` and blocks click-outside + Escape dismissal (Radix `onInteractOutside` / `onEscapeKeyDown` `preventDefault`). It closes ONLY via an explicit "Cancel" or a successful "Send to {N} recipients". This is the destructive-action friction the earlier specs deferred (04-UI-SPEC.md U9).
- **The confirm action is explicit and count-bearing.** The confirm button reads "Send to {N} recipients" (the count is the explicit consequence), sits alone as the single accent action in the footer, and is disabled while the summary is loading, while there are no sendable recipients, and while queuing is in flight.
- **Both submit paths are double-submit-safe.** The button disables in-flight (cosmetic first line), but the true guard is the DB-side atomic `UPDATE ... WHERE status='draft'`; a second submission surfaces the benign "already queued" toast, never a duplicate enqueue or an error (05-RESEARCH.md Pattern 2 / Pitfall 3).
- **The gate is server-authoritative.** Recipient count, sender identity, the merged sample, and every warning aggregate are computed server-side at confirm time (`countInvalidEmails`, `analyzeMerge`, `fillMessage`, `toSmtpConfigDto`) — the modal DISPLAYS them; a tampered client payload cannot suppress a warning or bypass the gate (05-RESEARCH.md Pitfall 4).
- **The merged sample renders as escaped text** (JSX / `whitespace-pre-wrap`); a CSV cell value is never injected as HTML and never logged (stored-XSS + T-*-LOG defense).
- **Sender identity is redacted** — sourced from `toSmtpConfigDto` (host/port/secure/username/from_addr/from_name/verified_at); the SMTP password structurally cannot reach the client (SMTP-04).
- **Errors never clear silently** — field-anchored test-address errors clear when the input becomes valid; destructive failure Alerts clear on the next attempt.
- **No live progress this phase** — the test-send returns a synchronous summary and the modal ends at "queued". Per-recipient live progress and the send-record state machine are Phase 6 (STATE.md decision).
