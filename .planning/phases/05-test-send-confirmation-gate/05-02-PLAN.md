<objective>
Build the whole-batch test-send as a chunked, client-drivable Server Action seam (TEST-01). Each call sends a bounded slice of recipient rows — every message personalized (subject AND body) and redirected to a single test address — and returns a cursor (`nextOffset`, `done`) so the client loops the chunks over the full row set. This is the deliberate resolution of the RESEARCH Pitfall-1 / decision-A2 tension: "whole batch, 500ms throttle, no hard cap" AND a self-hosted Coolify/Traefik reverse-proxy read timeout (~60s) cannot both hold for a single synchronous request over a large CSV, so the batch is split into short requests the client drives — no worker (that is Phase 6), no row cap, real progress.

Purpose: prove the full decrypt → verify → fill → sendMail path works against injected fakes, with per-row fill preserved and the decrypted password contained server-side. This seam is the payoff for Phases 1–4's engine discipline — it reuses `fillMessage`, `sendOne`, `createSmtpTransport`, `verifyTransport`, `throttle`, and `decrypt` verbatim.
Output: `lib/campaign/schema.ts`, `lib/campaign/index.ts`, `lib/campaign/actions.ts`, `lib/campaign/actions-core.ts`, `lib/campaign/actions-core.test.ts`.
</objective>
