<objective>
Build the read-and-monitor UI slice: the "Campaigns" nav item, the history list page, the campaign detail page (per-recipient results table), and the live progress panel that polls every ~2s. This makes the background send visible (SEND-05) and gives the user history + drill-down (HIST-01 / HIST-02), following the approved 06-UI-SPEC contract.

Purpose: The worker (Plans 01-04) sends in the background; without these surfaces the user cannot watch progress or review what was sent. Every surface is a read-only view over the persisted send_records state machine — no email is sent from here.
Output: sidebar slot + `/campaigns` list + `/campaigns/[id]` detail + progress panel + results table + the shadcn `Progress` primitive.
</objective>

## Phase Goal (user story)

**As a** signed-in sender, **I want to** have my queued campaign sent in a crash-safe background job with live progress and a durable per-recipient record, **so that** I can reliably deliver my whole batch, watch it happen, resume it after any crash without double-sending, and later review or download exactly what was sent.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → RSC / progress action | Client supplies a campaignId in the URL and poll calls; server scopes to Clerk userId |
| stored CSV value → rendered cell | Recipient emails/reasons originate from an uploaded CSV (untrusted) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
