<objective>
Build the userId-scoped read/service layer that the history and live-progress UI (Plan 05) and the CSV export route (Plan 06) consume: DAL reads for the campaign list, per-recipient drill-down, and progress counts, plus the `getCampaignProgress` server action (auth → core seam split). Realizes SEND-05 / HIST-01 / HIST-02 on the read side.

Purpose: Every web surface onto the send is a view over the persisted `send_records` state machine, and every read must be IDOR-safe (AUTH-02). Centralizing these reads behind the userId-scoped DAL is the structural defense so a guessed campaign id never leaks another tenant's data.
Output: extended `lib/data/campaigns.ts` + barrel, extended `lib/campaign/actions-core.ts` + `actions.ts`, extended tests, green under `npm test`.
</objective>

## Phase Goal (user story)

**As a** signed-in sender, **I want to** have my queued campaign sent in a crash-safe background job with live progress and a durable per-recipient record, **so that** I can reliably deliver my whole batch, watch it happen, resume it after any crash without double-sending, and later review or download exactly what was sent.
