# Phase 6 — UI Design Contract

> Visual and interaction contract for the live-send progress view, campaign history list, per-recipient drill-down, and results CSV download. Generated by gsd-ui-researcher (autonomous mode), verified by gsd-ui-checker.

This phase adds the first **read-and-monitor** surfaces in the app: a new top-level **Campaigns** area (`app/(app)/campaigns/`) with a **history list** (`/campaigns`), a **campaign detail / drill-down** page (`/campaigns/[id]`), and — hosted on that detail page while a send is `running`/`queued` — a **live progress panel** that polls a userId-scoped read action every ~2s (06-RESEARCH.md A1: polling, not SSE) and renders `sent / failed / remaining + current recipient`. The detail page also exposes a **results CSV download** (`/campaigns/[id]/export`, a `GET` route handler). No email is composed or sent from these surfaces — the enqueue happened in Phase 5; the worker (background Node process) does the sending; these pages are **windows onto the persisted `send_records` state machine**. Progress, history, drill-down, and export are all *views over that one table* (06-RESEARCH.md: "progress/history/idempotency are all views over the persisted state").

It inherits the design system, spacing, typography, and color contract established in **02-UI-SPEC.md** and carried verbatim through **03-/04-/05-UI-SPEC.md**. The primary design constraints are: the app shell (`app/(app)/layout.tsx` — 640px `max-w-2xl` content column, `p-8`, sidebar + top bar + footer), the established status/verify visual idioms (`CheckCircle2`/`AlertCircle` inline, destructive `Alert` for failures, `text-success` for positive indicators, `Loader2` for in-flight), and the `Table`/`Badge`/`Card` primitives already installed. This phase introduces the project's first **status-vocabulary** — a small, fixed set of lifecycle badges shared by campaigns and recipients — so a dedicated **Status Vocabulary** section below is the executor's source of truth for it.

**This phase has NO destructive actions.** Every surface is read-only or a safe download. The irreversible commitment (enqueuing a live send) already happened behind Phase 5's confirmation gate; nothing here deletes, cancels, or re-sends. (A future "cancel a running send" control is out of scope for Phase 6 — see Assumptions U7.)

## Typography

Inherited verbatim from 02-UI-SPEC.md. Exactly 4 sizes, 2 weights (400 regular, 600 semibold). No monospace introduced — recipient emails, error reasons, and counts render in Geist sans.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px (`text-base`) | 400 | 1.5 |
| Label | 14px (`text-sm`) | 400 (table cells, help text, status badge text, error-reason lines, "current recipient" line, count captions) | 1.5 |
| Heading | 20px (`text-xl`) | 600 | 1.2 |
| Display | 28px (`text-[28px]`) | 600 | 1.2 |

- The Campaigns list page title ("Campaigns") and the detail page campaign title use Heading (20px, `text-xl`, weight 600), matching every prior page heading.
- The **big live counts** (the sent/failed/remaining numbers, and the running progress percentage) render at **Display (28px, weight 600)** — this is the one place Display is used off the auth page, justified because the count is the page's focal data while a send runs. Their captions ("sent", "failed", "remaining") sit below at Label (14px, `text-muted-foreground`). See Assumption U2.
- Table headers, table cells, status-badge text, the "Currently sending to {email}" line, error reasons, timestamps, and recipient counts all use Label (14px, `text-sm`); muted metadata uses `text-muted-foreground` at Label size — never a new size.
- **Accepted library-default exceptions:** `Badge` ships `text-xs font-medium` (12px / weight 500) and `TableHead` ships `font-medium` (500). These are shadcn primitive defaults — accepted as-is exactly like the Button/Label `font-medium` exception recorded in 04-/05-UI-SPEC.md. Do NOT introduce 12px or weight-500 in hand-written classes; only the primitives keep their defaults. (Badge text at 12px is the sole 5th size, confined to the primitive.)

## Color

All values are existing `app/globals.css` tokens (oklch). Reference by semantic name, never raw values. No new palette tokens this phase.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--background` oklch(1 0 0) | Page background, main surfaces, table body rows |
| Secondary (30%) | `--card` / `--muted` / `--border` / `--sidebar` | Progress `Card` surface, stat-tile backgrounds (`bg-muted`), the progress-bar **track** (`bg-muted`/`bg-secondary`), table header + row separators (`border`), sidebar, "Queued"/"Pending" neutral badges (`secondary`) |
| Accent (10%) | `--primary` oklch(0.205 0 0) | See reserved list below |
| Destructive | `--destructive` oklch(0.577 0.245 27.325) | The **failed** status (recipient + campaign): `XCircle` icon, `destructive` Badge, failed-count number, and the destructive `Alert` for a whole-campaign abort or a progress-load error. See note below on failed vs. interrupted |
| Success (semantic, minimal) | `--success` oklch(0.627 0.17 149) — existing token, do NOT redeclare | ONLY: the **sent** status (`CheckCircle2` + `text-success`), the sent-count number, the **completed** campaign badge/check, and the "all delivered" all-clear line |

Accent (`--primary`) reserved for:
1. **The progress-bar fill** on a `running` campaign — the single dominant accent on the live-progress view; the track is neutral (`bg-muted`), the fill is `bg-primary`. At most one campaign is `running` at a time, so at most one accent bar shows.
2. **The single forward action per page.** On the campaign **detail** page (completed campaign) that is **"Download results"** (the `Download` primary button). On the history **list** the forward accent is the **empty-state CTA** ("Go to compose") when there are no campaigns; when the list is populated the list is a read surface with **no accent button** (rows link in neutral style) — the active-nav indicator is the only accent.
3. Active sidebar nav indicator (inherited — now including the "Campaigns" slot when active).
4. Focus rings (via `--ring`) on table row links, the download button, and any interactive control.

Everything else uses `secondary` / `ghost` / `outline` / neutral text. Row-to-detail navigation is a neutral clickable row (or a `ghost`/`link` "View" affordance), never an accent button per row. **Never style all interactive elements with accent** — a table of 100 recipients must not render 100 accent controls.

**Failed vs. interrupted (a phase-specific classification — read carefully):**
- A **`failed`** recipient (the SMTP server rejected the message — a real delivery failure the user should notice) is **destructive**: `XCircle` + `destructive` Badge + `text-destructive` on its count. This is a deliberate departure from the Phase 4/5 rule that "invalid/skipped emails are neutral muted" — those were *expected pre-send skips*; a send that was *attempted and rejected* is a genuine failure and earns the destructive treatment. See Assumption U3.
- An **interrupted** row (orphaned `sending` on crash → recorded `failed` with reason `"interrupted: delivery status unknown"`, 06-RESEARCH.md Pattern 5) is rendered **not** in destructive red but as a **neutral-uncertain** state: `AlertTriangle` + `text-muted-foreground`, badge label **"Interrupted"** (`outline`/`secondary` variant), because its delivery genuinely may have succeeded — red would over-claim failure. It still counts toward the campaign's `failed_count` in the data, but the UI distinguishes it from a hard reject. See Assumption U4.
- A **completed campaign that had some failures** is still **success-toned** overall (`CheckCircle2`, "Completed") with a muted secondary note "{failed} failed" — a completed *run* with partial failures is not a failed campaign (06-RESEARCH.md Pitfall 5 / A5). Campaign `failed` (destructive) is reserved for a **whole-campaign abort** (SMTP `verify()` failed / missing config / decrypt error before any send).

## Status Vocabulary

The fixed lifecycle vocabulary for this phase. Every badge below uses the shadcn `Badge` primitive; icons are lucide; colors are the tokens above. The executor MUST implement exactly these — no ad-hoc statuses, no extra colors.

**Campaign status** (drives the list-row badge and the detail-page header badge):

| Status | Badge label | `Badge` variant | Icon | Tone |
|--------|-------------|-----------------|------|------|
| `queued` | "Queued" | `secondary` | `Clock` | neutral / muted (waiting for the worker to claim) |
| `running` | "Sending" | `secondary` + inline `Loader2` (spin) | `Loader2` | neutral foreground + live progress panel below |
| `completed` (all sent) | "Completed" | `secondary` | `CheckCircle2` (`text-success`) | success |
| `completed` (with failures) | "Completed" + muted note "{failed} failed" | `secondary` | `CheckCircle2` (`text-success`) + `AlertCircle` (muted) on the note | success overall, muted failure note |
| `failed` (whole-campaign abort) | "Failed" | `destructive` | `XCircle` | destructive |

**Per-recipient status** (drives the drill-down results-table row + the exported CSV `status` column):

| Status | Cell label | Indicator | Icon | Tone |
|--------|------------|-----------|------|------|
| `pending` | "Pending" | plain text / `secondary` badge | `Clock` | muted |
| `sending` | "Sending" | `Loader2` (spin) | `Loader2` | foreground (at most one row, sequential) |
| `sent` | "Sent" | `text-success` | `CheckCircle2` | success |
| `failed` (rejected) | "Failed" | `text-destructive` | `XCircle` | destructive; error reason shown in the adjacent column |
| `failed` (interrupted) | "Interrupted" | `text-muted-foreground` | `AlertTriangle` | neutral-uncertain; reason "interrupted: delivery status unknown" |

**Resumed run:** a campaign re-claimed after a crash renders **identically to `running`** — there is no distinct "resumed" badge; the only visible trace is interrupted rows appearing in the results (Assumption U5). This keeps the vocabulary to the five states above.

## Screen States

Every state must be implemented by the executor and checked by the auditor.

**Focal point (detail page, running):** the live progress bar + the big Display counts are the anchor; the results table sits below as supporting detail; the "Download results" button is disabled (secondary presence) until the run finishes. **Focal point (detail page, completed):** the "Download results" accent button + the completed summary line; the results table is the substance. **Focal point (list page):** the newest campaign row (top of the table); on an empty list, the single "Go to compose" CTA.

| Screen / Area | States |
|--------|--------|
| List — loading | `Skeleton` table rows inside the shell column |
| List — empty | Centered empty state: glyph + "No campaigns yet" + body + accent "Go to compose" |
| List — populated | `Table`, newest first, each row: campaign name, "{N} recipients", status `Badge`, "{sent}/{total} sent" (+ muted "{failed} failed"), started timestamp; whole row links to detail |
| List — load failed | Destructive `Alert` with the list-load-failed copy; no partial/garbled table |
| Detail — queued | Status "Queued" badge; progress panel shows the "waiting for the worker" line (`Clock`, muted); results table shows "Preparing recipients…" until `send_records` materialize; download disabled |
| Detail — running | "Sending" badge (`Loader2`); progress bar (accent fill) at `sent+failed / total`; Display counts for sent/failed/remaining; "Currently sending to {email}" line (when a row is `sending`); results table live-updates row statuses; download disabled with tooltip; polling every ~2s |
| Detail — running, poll hiccup | Last-known counts remain; a muted "Couldn't refresh progress — retrying." line appears; the poller keeps retrying (Assumption U9); no destructive Alert, no data loss |
| Detail — completed (all sent) | "Completed" badge (`CheckCircle2`, success); "Done — all {total} messages sent."; polling stopped; download **enabled** (accent); full results table (all "Sent") |
| Detail — completed (partial failure) | "Completed" badge + muted "{failed} failed" note; success-toned summary with the partial-failure copy; results table mixes "Sent" (success) and "Failed" (destructive) / "Interrupted" (muted) rows with reasons; download enabled |
| Detail — completed (all failed) | "Completed" badge; muted "none delivered" copy (NOT a destructive whole-campaign Alert — the run completed); every row "Failed"/"Interrupted" with reasons; download enabled |
| Detail — whole-campaign aborted (`failed`) | "Failed" badge (destructive); destructive `Alert` with the abort copy (verify/config/decrypt failure); no per-recipient rows sent; download still available (empty/partial results) |
| Detail — interrupted rows present (after resume) | Renders as normal running/completed; interrupted rows show the "Interrupted" (`AlertTriangle`, muted) status + the "delivery status unknown; not retried" reason — surfaced so the user can manually re-target (06-RESEARCH.md Pattern 5) |
| Download — running | Button disabled + `Tooltip` "Available once the send finishes." |
| Download — ready | Accent "Download results"; `GET /campaigns/[id]/export` streams the userId-scoped, formula-injection-safe CSV (06-RESEARCH.md Pitfall/PITFALLS #12) |
| Any detail — cross-tenant / not found | The userId-scoped DAL / route returns 404; the page renders the framework not-found (no leak of another tenant's campaign) |

Interaction rules:
- **Progress is polled, not pushed.** The client polls a userId-scoped read every ~2s while `queued`/`running` and **stops** when `status` is `completed`/`failed` (06-RESEARCH.md A1 / progress example). No SSE, no websocket.
- **Every count is server-authoritative.** sent/failed/remaining/current are computed server-side from `send_records` (`remaining = total − sent − failed`; current = the lone `sending` row). The client DISPLAYS them; it never derives progress from a cached client list.
- **The page is safe to leave.** Because the send runs in the worker (not the request), navigating away or closing the tab does not stop it — the copy says so explicitly. Returning to the detail page resumes polling from the live DB state.
- **Reads are IDOR-safe.** List, detail, progress, and export all filter by the Clerk `userId`; the export route re-derives `userId` via `auth()` and 404s on a mismatch (06-RESEARCH.md Pitfall 6 / AUTH-02). A guessed campaign id never leaks another tenant's data.
- **Error reasons are message-only.** Per-recipient failure reasons render the stored message string, never a raw `Error` and never the SMTP password/auth config (D-06 / PITFALLS #2).
- **Recipient emails and reasons render as escaped text** (JSX) — a CSV-derived value is never injected as HTML (stored-XSS defense), consistent with Phase 5's merged-sample rule.
- **No destructive control.** There is no cancel/delete/re-send button this phase (Assumption U7); the only actions are navigation and a safe download.
- **The failed vs. interrupted distinction is visible**, per the Color section — a hard reject reads destructive, an interrupted row reads uncertain-muted.

| U6 | History and per-recipient tables stay inside the shell's 640px (`max-w-2xl`) column and **scroll horizontally** on overflow (the `Table` primitive's built-in wrapper), rather than widening the shell | Preserves shell consistency across all pages; avoids a per-page layout escape hatch. Column set is kept lean (≤4–5 columns) to minimize overflow | Medium — widening to `max-w-4xl` for tabular pages would require a shell/layout change affecting every page |
| U7 | **No cancel/pause/delete/re-send** control this phase — surfaces are read-only + a safe download | Roadmap scope is worker + progress + history + export; cancel-a-running-send is unspecified and adds worker-signalling complexity. No destructive action to gate | Medium — a "Cancel send" control would add a new worker signal path + a confirmation gate (a later phase) |
| U8 | The detail-page title uses the campaign's **template subject** (falling back to "Campaign #{id}"); campaigns have no user-given name field | The subject is the most human-recognizable identifier already stored; avoids a schema/name-input addition | Low — add a name field + migration later if desired |
| U9 | A **poll error** shows a muted "retrying" line and keeps last-known counts, rather than a destructive Alert or a cleared UI | A transient read hiccup shouldn't alarm the user or blank the data; the poller self-heals (06-RESEARCH.md A1, polling is a safe default) | Low — escalate to an Alert after N consecutive failures if desired |
| U10 | Progress uses the official shadcn **`Progress`** primitive (one `shadcn add`, no npm dep); a hand-rolled `bg-primary`-fill bar is an accepted equivalent | 06-RESEARCH.md "no new packages"; `Progress`'s Radix dep is already in `radix-ui`. Determinate bar (`sent+failed / total`) is standard for a bounded batch | Low — swap primitive for a `div` bar (or vice-versa) with no contract change |
| U11 | Progress bar is **determinate** (`sent+failed / total`), advancing as rows reach a terminal state; the `queued`/pre-materialize phase shows an indeterminate/`Clock` state instead | `total` is known once `send_records` materialize, so a determinate bar is honest and reassuring; before materialize there's no denominator | Low — an indeterminate bar throughout is a minor visual change |
