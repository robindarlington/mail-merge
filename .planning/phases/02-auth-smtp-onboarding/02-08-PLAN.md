**As a** signed-in user, **I want to** onboard and persist my own SMTP server proven functional by a live connection check, **so that** my credentials are stored encrypted at rest, reused across sessions, and never exposed to the client or logs.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → verifyAndSave Server Action | Untrusted edit payload (incl. blank password) crosses here; userId is NOT taken from the payload |
| applyVerifiedConfig → SQLite (userId-scoped DAL) | Stored encrypted credential is read and decrypted transiently here |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-2-08-CRED | Information Disclosure | applyVerifiedConfig blank-password merge | mitigate | Decrypted plaintext is assigned only to `parsed.data.password` (local scope), flows to verifyFn/encrypt, and is never placed on any ActionResult, thrown, or logged; ActionError remains the closed message-only union (existing redaction tests still assert no MARKER_PASSWORD leak) |
| T-2-08-IDOR | Spoofing / Elevation of Privilege | stored-config lookup | mitigate | The merge fetches via `getSmtpConfigForUser(userId)` where `userId` is re-derived by verifyAndSave from Clerk `auth()`; no client-supplied config id is ever accepted; the DAL has no owner-less lookup path |
| T-2-08-BLANK | Tampering | persist path | mitigate | A blank password with no stored row returns a validation error and saves nothing; encrypt runs only AFTER a real (typed or merged) password exists, so an empty credential can never be persisted or overwrite a good one |
| T-2-08-SC | Tampering | dependencies | accept | No new packages installed — reuses existing `zod`, `../crypto`, `../data/smtp`; no supply-chain surface added |
</threat_model>
