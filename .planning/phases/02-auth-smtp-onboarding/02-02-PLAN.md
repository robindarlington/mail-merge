<objective>
Build the SMTP verification engine that proves a user's server is functional before anything is saved. This is the backend half of wizard step 2: parse the form, dial the server with short timeouts, classify the failure into an actionable field-anchored message, and (on a TLS-shaped failure) silently probe the alternate TLS mode to offer a one-click switch. It composes and EXTENDS the Phase-1 `lib/core/send.ts` transport factory — it does not re-implement transport code.

Test-first: the three `*.test.ts` files (including the `smtp-server` local fixture) are the Wave 0 scaffolds from RESEARCH Validation Architecture. They pin assumption A1 (the exact TLS-mismatch error signature) empirically.

Purpose: SMTP-01 (field validation), SMTP-02 (explicit TLS, never port-inferred), SMTP-03 (live check distinguishing auth vs host/port vs TLS, fails fast).
Output: `lib/smtp/{schema,errors,verify,index}.ts` + tests; additive timeout fields on `lib/core/send.ts`.
</objective>

## Phase Goal

**As a** signed-in user, **I want to** onboard and persist my own SMTP server proven functional by a live connection check, **so that** my credentials are stored encrypted at rest, reused across sessions, and never exposed to the client or logs.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser form input → verify engine | Host/port/username/password are attacker-controlled untrusted input |
| verify engine → user-supplied SMTP host | Server dials an arbitrary host:port (SSRF surface) |
| verify engine → outward return value | Classified result must never carry raw credentials/config |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-2-TLS | Information disclosure | lib/smtp/verify.ts transport build | mitigate | `requireTLS: true` whenever `secure: false` so a STARTTLS-capable server can't silently send cleartext |
| T-2-MITM | Spoofing / Info disclosure | verify transport TLS options | mitigate | Never set `tls.rejectUnauthorized: false`; rely on nodemailer secure defaults (grep-asserted absent) |
| T-2-SSRF | Information disclosure | lib/smtp/schema.ts host field | mitigate | zod refinement rejects loopback/link-local/RFC1918 host literals before any dial |
| T-2-CRED | Information disclosure | classifier + verify return | mitigate | Return typed `{kind, field, raw:message-only}` values; raw text is message-only (no config/credentials); no logging of pass/auth/transport (carry-forward lib/core grep rule) |
| T-2-SPAM | Elevation of privilege / abuse | verify() resource use | mitigate (partial) | Short ONBOARDING_TIMEOUTS bound each attempt; per-user rate limiting is applied at the Server Action layer in plan 02-05 |
</threat_model>
