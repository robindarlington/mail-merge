## Phase Goal

**As a** signed-in user, **I want to** onboard and persist my own SMTP server proven functional by a live connection check, **so that** my credentials are stored encrypted at rest, reused across sessions, and never exposed to the client or logs.

> Note: the ROADMAP `**Goal:**` line for Phase 2 is a prose sentence; the story above is a faithful restatement of it (actor = signed-in user, capability = onboard + persist SMTP proven by live check, outcome = encrypted + reused + never exposed). No scope was invented.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| unauthenticated browser → app routes | Every request must prove a Clerk session before reaching app content |
| build stage → client bundle | `NEXT_PUBLIC_*` values are inlined at build time; secrets must not be among them |
| npm registry → repo | Newly installed packages cross into the trusted build |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-2-AUTH | Elevation of privilege | app routes (proxy.ts) | mitigate | `clerkMiddleware` + `auth.protect()` on all non-public paths; verified by signed-out `/dashboard` redirect check |
| T-2-SPOOF | Spoofing | session handling | mitigate | Sessions delegated entirely to Clerk; server never trusts a client-supplied user id (defense-in-depth userId checks land in 02-03/02-05) |
| T-2-SC | Tampering | npm installs (react-hook-form, @hookform/resolvers, @clerk/ui, smtp-server) | accept | All packages verified [OK] in RESEARCH Package Legitimacy Audit (slopcheck run, no postinstall scripts); no [ASSUMED]/[SUS] packages, so no blocking legitimacy checkpoint required |
</threat_model>
