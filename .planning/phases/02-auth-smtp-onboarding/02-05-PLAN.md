<objective>
Build the three Server Actions that are the backend of the onboarding wizard and edit flow — the seam that ties Clerk identity, the verify engine (02-02), the DAL (02-03), and crypto together. This plan defines the typed `ActionResult` contract the wizard UI (02-06) consumes, so it ships first: interface before implementation.

`verifyAndSave` (D-04/D-05) runs verify-then-save as one atomic action; `updateFromFields` (D-08) saves from-fields without a verify round-trip and never clears `verified_at` for the wrong reason; `sendTestEmail` (D-03/SMTP-05) reuses `lib/core/send.ts` to prove the saved transport actually delivers. Every action re-derives `userId` server-side and returns typed results only — never a raw nodemailer error, never the password.

Purpose: SMTP-05 (save only after verify; optional test-send), AUTH-02 (actions re-derive userId — defense in depth), SMTP-04 (no secret ever crosses outward).
Output: `lib/smtp/actions.ts` + `lib/smtp/actions.test.ts`.
</objective>

**As a** signed-in user, **I want to** onboard and persist my own SMTP server proven functional by a live connection check, **so that** my credentials are stored encrypted at rest, reused across sessions, and never exposed to the client or logs.

Consumed:
- `import { smtpFormSchema, verifySmtp, classifyVerifyError } from "@/lib/smtp"` (02-02): verifySmtp returns `{ ok:true } | { ok:false; kind; field; raw; suggestion? }`; `classifyVerifyError(err)` maps a nodemailer error to `{ kind, field }` (reused to classify a failed pre-send verify).
- `import { getSmtpConfigForUser, upsertSmtpConfig, updateFromFields as dalUpdateFromFields } from "@/lib/data/smtp"` (02-03).
- `import { encrypt, decrypt } from "@/lib/crypto"`: encrypt() → {enc,iv,tag}; decrypt(payload) → plaintext.
- `import { createSmtpTransport, verifyTransport, sendOne } from "@/lib/core"`: build transport, run the thin `verify()` wrapper, send one structured result.
- `import { auth, currentUser } from "@clerk/nextjs/server"`: `const { userId } = await auth()`; `currentUser()` exposes the primary email for the test-send prefill (Open Question 1).

Defined (the contract 02-06 consumes) — RESEARCH Code Example 7:
- `type ActionResult = { ok: true } | { ok: false; error: ActionError }`.
- `ActionError = { kind: "unauthenticated" } | { kind: "validation"; issues } | { kind: "auth"|"connection"|"tls"|"unknown"; field; raw; suggestion? } | { kind: "rate_limited" } | { kind: "send_failed"; raw }`.
- `verifyAndSave(raw: unknown): Promise<ActionResult>` — auth → zod parse → verifySmtp → on ok: encrypt + upsertSmtpConfig(userId, …) (sets verified_at); on suggestion: return without saving.
- `updateFromFields(raw: unknown): Promise<ActionResult>` — auth → parse from_* → dalUpdateFromFields(userId, …); verified_at untouched.
- `sendTestEmail(toAddress: string): Promise<ActionResult>` — auth → load config → decrypt password → createSmtpTransport → **verifyTransport (carry-forward CLAUDE.md `transport.verify()` before any send)** → sendOne to the given address (defaults to currentUser primary email). A failed verify is classified with `classifyVerifyError` and returned via the same `{kind, field, raw:message}` path as verifyAndSave — sendOne is never reached when verify fails.

Testability seam: factor the persistence orchestration into a plain (non-"use server") helper, e.g. `applyVerifiedConfig(userId, input, verifyFn = verifySmtp)`, so actions.test.ts can inject a fake verifyFn and drive verified_at semantics against a temp DB without live SMTP or Clerk. The "use server" wrappers call the helper with the real deps.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client form submit → Server Action | Untrusted args; server re-parses with zod and re-derives userId |
| Server Action → client return value | Must never carry password or raw nodemailer error |
| decrypted password (in-memory) → transport | Plaintext exists only transiently server-side at send time |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-2-CRED | Information disclosure | action return values / logs | mitigate | Typed ActionResult only; classified error carries a message string, never the raw error object or config; decrypted password never logged/returned; redaction test on JSON.stringify |
| T-2-IDOR | Information disclosure / Tampering | all three actions | mitigate | Each re-derives `userId` via `await auth()` and passes it to userId-scoped DAL; never trusts a client id (AUTH-02 defense-in-depth) |
| T-2-CSRF | Tampering | mutating actions | mitigate | Next.js Server Actions carry built-in origin checks; no state-changing GET endpoints introduced |
| T-2-SPAM | Elevation / abuse | verifyAndSave verify() | mitigate | Per-user in-process rate limit on verify attempts + short timeouts from 02-02 bound resource use |
| T-2-VERIFY | Tampering (integrity) | verifyAndSave persist path + sendTestEmail send path | mitigate | Persist happens ONLY on verify success; an unverified or alternate-mode-only config is never saved (D-04/D-05). sendTestEmail re-runs `verifyTransport` before any send (CLAUDE.md carry-forward) so a stale/broken config fails a classified verify instead of a blind sendOne |
</threat_model>
