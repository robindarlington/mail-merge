## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, badge padding |
| sm | 8px | Gaps between related form controls, inline button groups |
| md | 16px | Default gap between form fields; card internal padding unit |
| lg | 24px | Card padding, gap between form sections (server details vs sender identity) |
| xl | 32px | Page container padding, gap between page heading and content |
| 2xl | 48px | Gap between wizard stepper and step content; dashboard callout vertical padding |
| 3xl | 64px | Top offset for centered auth pages |

Exceptions: shadcn primitive internals (button heights, input padding) follow library defaults and are exempt. Wizard/form content column max-width: 640px (`max-w-2xl`). Sidebar width: shadcn sidebar default (16rem).

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px (`text-base`) | 400 | 1.5 |
| Label | 14px (`text-sm`) | 400 (form labels, sidebar nav, help text, footer) | 1.5 |
| Heading | 20px (`text-xl`) | 600 | 1.2 |
| Display | 28px (`text-[28px]`) | 600 | 1.2 |

- Exactly 4 sizes, 2 weights (400 regular, 600 semibold). No other sizes or weights in phase-authored code.
- shadcn primitive internals that ship `font-medium` (500) — e.g. Button, Label defaults — are accepted library defaults and are NOT overridden; do not introduce 500 in hand-written classes.
- Display (28px) is used once per page at most: dashboard greeting, wizard page title.
- Muted/secondary text uses `text-muted-foreground` at Body or Label size — never a new size.

## Color

All values are the existing `app/globals.css` tokens (oklch, light mode shown). Reference tokens by semantic name, never raw values.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--background` oklch(1 0 0) | Page background, main content surface |
| Secondary (30%) | `--sidebar` oklch(0.985 0 0) / `--muted` oklch(0.97 0 0) / `--card` + `--border` | Sidebar, cards, wizard step container, separators |
| Accent (10%) | `--primary` oklch(0.205 0 0) | See reserved list below |
| Destructive | `--destructive` oklch(0.577 0.245 27.325) | Field-anchored verify errors, invalid-input borders, error Alert variant. No destructive ACTIONS exist this phase (D-09: no delete) |
| Success (semantic, minimal) | `--success: oklch(0.627 0.17 149)` — NEW token, add to `globals.css` `:root` + `@theme inline` (dark: oklch(0.696 0.17 162)) | ONLY: "Verified" badge icon+text, step-2 success check icon, test-send success toast icon |

Accent (`--primary`) reserved for:
1. Primary CTA button of each screen (one per view): dashboard callout "Set up your SMTP server", wizard "Verify & continue", step-3 "Send test email"
2. Active sidebar nav item indicator
3. Wizard stepper: current/completed step markers
4. Focus rings (via `--ring`)

Everything else (secondary buttons, "Skip for now", "Back") uses `secondary`/`ghost`/`outline` variants. Never style all interactive elements with accent.

## Copywriting Contract

Sentence case everywhere (headings, buttons, labels). No exclamation marks. Errors always state the problem + what to do next.

| Element | Copy |
|---------|------|
| Primary CTA (wizard step 1→2) | "Verify & continue" (exact, D-04) |
| Primary CTA (dashboard soft-gate) | "Set up your SMTP server" (exact, D-02) |
| Primary CTA (wizard step 3) | "Send test email" |
| Step-3 skip affordance | "Skip for now" (ghost/link button — skippable per D-03) |
| Empty state heading (dashboard, no SMTP config) | "Connect your email server" |
| Empty state body | "Mail Merge sends through your own SMTP server — you stay in control of your sending. Set it up once and it's saved for every campaign." + CTA "Set up your SMTP server" |
| Error state — auth (`EAUTH`) | "Username or password rejected by the server. Double-check your SMTP username and password." — anchored to username + password fields (D-06) |
| Error state — connection (`EDNS`/`ECONNECTION`/`ETIMEDOUT`) | "Couldn't reach {host}:{port}. Check the host name and port, then try again." — anchored to host + port fields (D-06) |
| Error state — TLS | "Secure connection failed. This usually means the wrong TLS mode for this server." — anchored to the TLS mode control (D-06) |
| TLS auto-retry suggestion (D-05) | "Your server needs {STARTTLS / implicit SSL} — switch and continue?" with one-click button "Switch to {mode} & verify" |
| Raw error detail | Collapsible "Show technical details" revealing raw nodemailer message text (message only — never config/credentials) |
| Verify in-flight | Button label swaps to "Verifying…" with spinner; helper text "Connecting to your server — this can take up to 15 seconds." |
| Verify success (step 2) | "Connection verified" + success icon; auto-advance affordance "Continue" |
| Test-send success | Toast: "Test email sent — check {address}'s inbox." |
| Test-send failure | Alert: "Test email failed to send. {classified reason}. Your settings are still saved and verified." + "Show technical details" |
| Edit flow — password field | Placeholder/help: "Leave blank to keep your current password" (D-07) |
| Edit flow — connection fields note | "Changing these settings requires re-verifying your connection." (D-08) |
| Verified status badge | "Verified" (+ CheckCircle2 icon); unverified after edit: "Re-verify required" (neutral/outline badge, NOT destructive) |
| TLS mode control labels | Radio group "Connection security": "Implicit SSL/TLS (usually port 465)" / "STARTTLS (usually port 587)" — explicit choice, port hints only (SMTP-02) |
| Footer attribution (D-12) | "Built by Robin Darlington" + link "Hire me for custom tools" → `HIRE_ME_URL` constant in `lib/config.ts` (placeholder URL, Phase 9 flips it) |
| Destructive confirmation | None — no destructive actions this phase (D-09: no delete; edit/replace only) |

## Screen States

Every screen must handle these states — executor implements all, auditor checks all:

| Screen | States |
|--------|--------|
| Dashboard | 1) No SMTP config → soft-gate callout dominant. 2) Config saved + verified → summary card with "Verified" badge. 3) Config saved, `verified_at` cleared by edit → summary card with "Re-verify required" badge + CTA back into wizard step 2 |
| Wizard step 1 (details) | Default (empty form, TLS radio defaults by port hint but user must be able to choose freely) · client-side zod validation errors · edit mode (prefilled DTO values, blank password with keep-current help) |
| Wizard step 2 (verify) | In-flight (disabled form + spinner, ≤15s) · field-anchored error (auth / connection / TLS) · TLS switch suggestion alert with one-click action · success (check + Continue) |
| Wizard step 3 (test-send) | Default (recipient prefilled with Clerk primary email, editable) · sending · success toast · failure alert (settings remain saved) · skip |
| All shell pages | Loading: `Skeleton` for async config fetch — never layout shift or blank flash |

Interaction rules:
- Verify and test-send buttons disable while in flight; no double-submit.
- Errors never clear silently — they clear when the user edits an anchored field or retries.
- The wizard is also the edit flow (`/settings/smtp`): connection-field changes route through step 2 again (D-08); from-name/from-address-only edits save directly with a success toast.
- Password value never appears in any client-visible payload, including the edit form initial values (D-07, SMTP-04).
