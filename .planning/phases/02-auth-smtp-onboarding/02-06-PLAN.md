<objective>
Build the three-step SMTP onboarding wizard (and its edit flow) — the user-facing capability that makes the whole phase real. A user enters their server details with an explicit TLS-mode choice, clicks "Verify & continue" (which verifies then saves atomically via 02-05), sees actionable field-anchored errors or a one-click TLS switch on failure, and on success reaches a skippable test-send that proves delivery. The same page is the edit flow: it prefills the saved config (password always blank), routes connection-field changes back through verify, and saves from-field-only changes directly.

This is the final vertical slice that closes SMTP onboarding: after it, a fresh account can go from "no config" to a verified, encrypted, reusable SMTP config with a confirmed test delivery.

Purpose: SMTP-01 (details form), SMTP-02 (explicit TLS radio), SMTP-05 (onboarding completes only after verify; optional test-send), plus D-01/D-04/D-05/D-06/D-07/D-08 UX.
Output: `app/(app)/settings/smtp/page.tsx` + `components/smtp/{smtp-wizard,step-details,step-verify,step-test-send}.tsx`.
</objective>

## Phase Goal

**As a** signed-in user, **I want to** onboard and persist my own SMTP server proven functional by a live connection check, **so that** my credentials are stored encrypted at rest, reused across sessions, and never exposed to the client or logs.

## Acceptance Criteria (this plan)
- [ ] The wizard renders three labelled steps (Server details → Verify → Test send), each gating the next.
- [ ] A bad-credential verify shows the auth message anchored to username/password; a TLS mismatch offers "Switch to {mode} & verify".
- [ ] A successful verify advances to test-send; "Skip for now" completes onboarding without sending.
- [ ] Edit mode prefills host/port/from with a blank password; the password is absent from the page's client payload.
- [ ] `npx --no-install tsc --noEmit` exits 0.

| Boundary | Description |
|----------|-------------|
| browser form → Server Actions | Client input is untrusted; server re-validates (02-05) — client zod is convenience only |
| Server Action result → rendered UI | Error detail shown to the user must be message-only, never credentials/config |
| edit-mode initial values → client | The saved password must never be sent to the browser |
