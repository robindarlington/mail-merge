<objective>
Add the two defined operational routines (phase success criterion #4): a periodic idle-aware `wal_checkpoint(TRUNCATE)` and a periodic attachment-orphan sweep, both driven from the worker's own poll loop — no cron, no new dependency. Build the logic as a standalone, unit-tested module so the seam is proven against a temp DB without spinning the loop.

Purpose: With two long-lived processes (web + worker) each holding read snapshots, SQLite's PASSIVE auto-checkpoint cannot reset the WAL, so it grows unbounded between restarts (RESEARCH Finding 4). And pre-campaign attachment uploads that are never stamped (or abandoned in draft) accumulate files forever. Both are the worker's job because it already owns the idle-awareness the routines need.
Output: `lib/worker/maintenance.ts` + tests, wired into the worker loop.

<notes>
- RESEARCH Finding 4 is authoritative. `wal_checkpoint(TRUNCATE)` can return `busy:1` WITHOUT truncating if a reader holds a snapshot the whole time — inspect and log the returned row; never assume the WAL shrank.
- Orphan definition (CONTEXT + Finding 4): `campaign_id IS NULL` (never stamped) OR `campaign_id` points at a campaign whose `status = 'draft'`; AND `created_at < now - ATTACHMENT_ORPHAN_DAYS*86400`. Never touch attachments belonging to queued/running/completed/failed campaigns.
- Order of operations: delete the DB row inside a transaction FIRST (the DB is the source of truth for pending-bytes quota), THEN `unlink` the file. A failed unlink logs an `unlinkFailures` count and leaves a disk-only file (safe — phase-7 worker already tolerates missing attachment files).
- Log COUNTS only. Never log filenames, storage paths, or userIds.
- Keep the module free of the loop: pass in `connection`/`db`, a `now` (for deterministic tests), `orphanDays`, an `uploads`-unlink function or use node:fs directly, and a logger. The scheduler state (lastCheckpointAt/lastSweepAt) can live in the module or be passed in — make it injectable so tests don't depend on wall-clock timers.
</notes>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| maintenance routine → shared DB | Runs as the single writer only when idle; must not overlap a send tick |
| sweep → /data/uploads files | Deletes files only after their DB row is gone; degrades safely on unlink failure |
| routine logs → operator | Must emit counts only, never user filenames/paths/ids |
