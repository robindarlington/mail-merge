<notes>
- RESEARCH Pitfall 1 is the crux: `docker compose restart` ALWAYS uses a 10s timeout and ignores stop_grace_period — it tests the SIGKILL path unintentionally. The graceful assertion MUST use `docker compose stop` (honors the 5m worker grace) then `docker compose up -d`. Use `docker kill` SEPARATELY as the crash-path assertion.
- The worker sends over BYO SMTP from an encrypted smtp_config. Seed a config whose host is `host.docker.internal` (Docker Desktop 28.1.1, confirmed) and port = the stub's port, secure=false. The stub accepts all auth and logs every RCPT so duplicates are detectable.
- Seed via the harness run on the HOST against the same /data DB the containers use (set DATABASE_PATH to the mounted volume path, or exec the harness inside the web container). Prefer running the harness inside the web container (`docker compose exec`) so it uses the exact in-container DB + crypto key. Choose one approach and document it in the script header.
- Give the send enough rows and a large enough SEND_DELAY_MS that the interrupt reliably lands mid-batch (e.g. 10–15 recipients at SEND_DELAY_MS=1500), so some rows are sent pre-interrupt and the rest resume after.
- Assertions after resume: (1) campaign + send_records rows still present (data survived the volume across stop/up); (2) COUNT of terminal send_records == recipient count with each (campaign_id,to_addr) unique; (3) stub RCPT log has each recipient AT MOST once (no duplicate delivery). Fail loud with a nonzero exit on any violation.
- Mark clearly (comments + a printed banner) that the Coolify Stop-Grace-Period UI setting and the real VPS redeploy can only be proven at the plan 08-05 staging checkpoint — this local script proves the mechanism, not the platform.
- **Env preflight:** the hardened compose interpolates `${NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY}` (build.args), `${CLERK_SECRET_KEY}`, and `${CREDENTIAL_ENC_KEY}` (environment). If any is unset the build produces a broken image (Clerk throws) or the worker cannot decrypt the seeded smtp_config (wrong/empty enc key). The script MUST preflight these three vars up front and exit nonzero with a clear "set X in .env before running" message BEFORE `docker compose build`/`up` — never let a missing key surface as a confusing mid-run failure.
- **Grep-gate hygiene:** the `! grep … "compose restart"` gate in Task 2's verify must NOT trip on the script's own explanatory comments. Phrase any comment that mentions the forbidden command WITHOUT the literal string `compose restart` (e.g. write "bare compose-level restart" or "restart subcommand"), and the verify additionally strips comment lines before grepping so the gate only inspects executable lines.
</notes>
