<objective>
Build the Server Action seam that powers both preview and save, following the established `actions.ts` (`"use server"` auth wrappers) + `actions-core.ts` (testable, userId-injected, no directive) split verbatim. `previewCampaignCore` resolves a recipient set from a userId-scoped lookup, reads its CSV back via `readUpload`, re-parses server-side, and returns all rows + columns plus the two template-INDEPENDENT server-authoritative fields: the resolved email column and the invalid-email count. Template-DEPENDENT aggregates (unknown tokens, rows-with-empty-values) are deliberately NOT in the report â€” they change as the user types and are computed client-side in Plan 05 from the returned `rows`. `saveTemplateCore` validates subject/body and writes a standalone userId-scoped template.

Purpose: this is the security-critical seam â€” the client passes ONLY a `recipientSetId` (never a storage path), the server owns path resolution + tenancy + the template-INDEPENDENT authoritative values (the invalid-email count, and the resolved email column so the preview's "To:" render cannot diverge from the column used for `invalidEmailCount`). The template-DEPENDENT aggregates stay client-side so they can never go stale against the composed template. The standalone template write satisfies EDIT-04 without pre-empting Phase 5's campaign join.
Output: `lib/compose/actions-core.ts` (+ test), `lib/compose/actions.ts`, `lib/compose/index.ts`.
</objective>
