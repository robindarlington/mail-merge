<objective>
Build the Server Action seam that powers both preview and save, following the established `actions.ts` (`"use server"` auth wrappers) + `actions-core.ts` (testable, userId-injected, no directive) split verbatim. `previewCampaignCore` resolves a recipient set from a userId-scoped lookup, reads its CSV back via `readUpload`, re-parses server-side, and returns the rows plus the server-authoritative validation aggregate (invalid emails + unknown tokens + rows-with-empty-values) AND the server-resolved email column the client renders "To:" against. `saveTemplateCore` validates subject/body and writes a standalone userId-scoped template.

Purpose: this is the security-critical seam — the client passes ONLY a `recipientSetId` (never a storage path), the server owns path resolution + tenancy + the authoritative counts (PREV-03 anti-divergence) AND the resolved email column (so the preview's "To:" render cannot diverge from the column used for `invalidEmailCount`), and the standalone template write satisfies EDIT-04 without pre-empting Phase 5's campaign join.
Output: `lib/compose/actions-core.ts` (+ test), `lib/compose/actions.ts`, `lib/compose/index.ts`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → Server Action | Untrusted recipientSetId + subject + body cross into the server |
| Server Action → filesystem (readUpload) | storage_path must be server-resolved from a userId-scoped row |
| Server Action → DB | Template write + recipient-set read must be tenant-scoped |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-4-IDOR | Information Disclosure / Elevation | previewCampaignCore | mitigate | getRecipientSetForUser(userId, id); other tenants → not_found; client never sends a path (Task 1, tested) |
| T-4-TRAVERSAL | Tampering | readUpload call | mitigate | path comes only from the userId-scoped row's storage_path (relative UUID); readUpload prefix-check (Plan 01) |
| T-4-ENDPOINT | Spoofing / Elevation | actions-core seams | mitigate | only actions.ts carries "use server"; core seams have no directive → not wire-callable (Task 2) |
| T-4-TAMPER-OWNER | Tampering | saveTemplateCore | mitigate | createTemplate injects server-derived userId; values cannot carry it (Plan 02) |
| T-4-DIVERGE | Repudiation/Integrity | preview To: column vs invalidEmailCount | mitigate | emailColumn returned from the server = the exact column used for invalidEmailCount; client renders "To:" against it, never a re-detected guess (Task 1) |
| T-4-LOG | Information Disclosure | previewCampaignCore | mitigate | no console.* of subject/body/CSV cell values; ActionError.raw is always a string (Task 1, grep-gated) |
| T-4-DOS | Denial of Service | preview payload | mitigate | rows capped at upload (MAX_ROWS 5000); preview inherits the cap; aggregate is small |
| T-4-SC | Tampering | npm installs | accept | zero new npm dependencies |
