# Phase 4 â€” UI Design Contract

> Visual and interaction contract for the Compose editor + live merged Preview + pre-send validation report + template save. Generated by gsd-ui-researcher (autonomous mode), verified by gsd-ui-checker.

This phase adds ONE new authenticated surface â€” a **Compose** page (`app/(app)/compose/`, per 04-RESEARCH.md A3) â€” where a signed-in user picks one of their saved recipient lists, composes a plain-text subject + body with `{{merge-field}}` autocomplete (click-to-insert chips + a `{{`-triggered suggestion list), steps through a live merged preview against real CSV rows, reads a pre-send validation report, and saves the composed subject/body as a template. It inherits the design system, spacing scale, typography, and color contract established in **02-UI-SPEC.md** and carried verbatim through **03-UI-SPEC.md**. Consistency with the Phase-3 CSV uploader (`components/recipients/csv-uploader.tsx`) and the SMTP wizard (react-hook-form + zod, field-anchored errors, Card-based layout, sonner toasts, `text-[28px]` page headings, `gap-8` page column) is the primary design constraint. Only phase-specific additions (copy, new screen states, the editor/chips/popover, the preview stepper, and the validation report) are newly declared below.

## Spacing Scale

Inherited verbatim from 02-UI-SPEC.md (8-point scale, multiples of 4). No new spacing tokens this phase.

| Token | Value | Usage |

| lg | 24px | Card padding, gap between compose / preview / report cards (`gap-6` inside a card) |

- shadcn primitive internals (button heights, input/textarea/select padding, popover padding) follow library defaults and are exempt.

- **Icon-only stepper controls** (Previous / Next): if rendered icon-only, they carry a 44px minimum touch target (`min-h-11 min-w-11`). Preferred implementation is icon + visible label ("Previous" / "Next") ghost/outline buttons, which already clear 44px â€” no exception needed in that case.

## Typography

Inherited verbatim from 02-UI-SPEC.md. Exactly 4 sizes, 2 weights (400 regular, 600 semibold). No monospace â€” plain-text email stays in Geist sans; merge tokens are ordinary body text.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px (`text-base`) | 400 | 1.5 |
| Label | 14px (`text-sm`) | 400 (form labels, help text, chip labels, validation-report rows, stepper "Recipient i of n", saved-list rows) | 1.5 |
| Heading | 20px (`text-xl`) | 600 | 1.2 |
| Display | 28px (`text-[28px]`) | 600 | 1.2 |

- Page title "Compose" uses Display (28px), once per page (`h1`, matching `app/(app)/recipients/page.tsx`).
- Card titles ("Compose your email", "Preview", "Validation report") use Heading (20px, `text-xl`, `CardTitle`).
- Help text, merge-field chip labels, the stepper counter, and validation-report lines use Label (14px, `text-sm`); muted variants use `text-muted-foreground` at Label size â€” never a new size.
- The merged preview body renders at Body (16px, `text-base`), preserving the author's line breaks (`whitespace-pre-wrap`) â€” it is a faithful plain-text render, not a styled block.
- shadcn primitive internals that ship `font-medium` (500) â€” Button, Label, Popover items, Select â€” are accepted library defaults and are NOT overridden; do not introduce 500 in hand-written classes.

## Color

All values are existing `app/globals.css` tokens (oklch). Reference by semantic name, never raw values. No new palette tokens this phase.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--background` oklch(1 0 0) | Page background, main content surface, textarea/input fields |
| Secondary (30%) | `--card` / `--muted` / `--border` / `--sidebar` | Compose card, preview card, report card, merged-preview surface (`bg-muted`), field chips (secondary/outline), separators, sidebar |
| Accent (10%) | `--primary` oklch(0.205 0 0) | See reserved list below |
| Destructive | `--destructive` oklch(0.577 0.245 27.325) | Blocking save FAILURE only (destructive `Alert` on save error) + field-anchored "add a subject/message" validation on the save attempt. See note on why empty/unknown merge values are NOT destructive |
| Success (semantic, minimal) | `--success` oklch(0.627 0.17 149) â€” existing token, do NOT redeclare | ONLY: the "all rows valid / no missing values" positive report indicators (`CheckCircle2` + text) and the save-success toast icon |

Accent (`--primary`) reserved for:
1. The single primary CTA per view: **"Save template"** on the compose surface. In the empty state (no recipient lists yet), the single accent CTA is **"Go to recipients"** instead. One accent button visible per view.
2. Active sidebar nav indicator when "Compose" is the current route (shadcn `isActive` on `SidebarMenuButton` â€” the only nav accent use).
3. Focus rings (via `--ring`) on the recipient-list select, subject input, body textarea, merge-field chips, popover items, stepper controls, and buttons.

Everything else uses `secondary` / `ghost` / `outline` variants: the merge-field chips (click-to-insert), the `{{` suggestion-list items, the preview stepper's Previous/Next, and the "Choose a different list" style secondary actions. **Never style all interactive elements with accent** â€” the chip row especially must not read as a wall of accent buttons.

**Empty merge values and unknown tokens are NOT destructive.**
- An **empty merge value** (a row whose `{{name}}` is blank) is an expected, non-blocking outcome â€” that spot is simply blank in the sent email. Render it as neutral text (`text-muted-foreground`) with a lucide `AlertCircle` â€” the SAME informational treatment Phase 3 used for skipped invalid-email rows. Never destructive.
- An **unknown token** (a `{{typo}}` that is not a column in the recipient list) is a likely authoring mistake that would appear literally in every email. Surface it prominently â€” `text-foreground` (not muted) with a lucide `AlertTriangle` at the top of the validation report â€” but it is still NOT destructive and does NOT hard-block saving a template (the hard confirm-before-send gate is Phase 5). Destructive stays reserved for a save that cannot proceed.

## Screen States

Every state must be implemented by the executor and checked by the auditor.

**Focal point (Compose screen):** the Compose editor's Subject/Body fields are the primary visual anchor; the Preview stepper and Validation report are secondary, supporting surfaces. Visual weight (size, position, the single accent CTA) resolves to the editor first.

| Screen / Area | States |
|--------|--------|
| Compose page (no recipient lists) | Empty-state callout dominant: "Upload a recipient list to start composing" + "Go to recipients" accent CTA. Compose editor is NOT shown until a list exists |
| Compose page (lists exist, none selected) | Recipient-list picker present (unset, placeholder). Editor visible but merge-field chips + preview + report show their "choose a list" empty/idle copy |
| Recipient list â€” selecting | Rows/columns fetch for the chosen set: preview + report areas show `Skeleton`; picker + editor stay interactive |
| Editor â€” default | Subject `Input` + body `Textarea` + merge-field chip row (from the selected list's columns) + help text; "Save template" enabled only when subject AND body are non-empty |
| Editor â€” chip insert | Clicking a chip inserts `{{column}}` at the focused field's caret (subject or body); focus returns to that field |
| Editor â€” `{{` autocomplete | Typing `{{` opens a `Popover` suggestion list filtered by what follows; click / Enter inserts `{{column}}` and closes; Escape dismisses; "No matching fields." when the filter matches nothing |
| Preview â€” idle | "Choose a recipient list and write your message to see a preview." (no list selected, or subject+body both empty) |
| Preview â€” active row | "Recipient {i} of {total}", merged To / Subject / body for row i; Previous disabled at i=1, Next disabled at i=total |
| Preview â€” current row missing values | Neutral note + `AlertCircle`: "This recipient is missing a value for {fields}. That spot will be blank in their email." (PREV-02) |
| Preview â€” rows loading | `Skeleton` in place of the merged block; never a blank flash or layout shift |
| Validation report â€” unknown tokens | Prominent `AlertTriangle` line at the top: template uses a field that isn't a column (affects every row) â€” surfaced, never hidden (PREV-03; 04-RESEARCH.md Pitfall 1) |
| Validation report â€” invalid emails | Neutral `AlertCircle` count line, or `text-success` `CheckCircle2` when all valid (server-authoritative count) |
| Validation report â€” missing values | Neutral `AlertCircle` aggregate line, or `text-success` `CheckCircle2` when every row is complete (computed over ALL fetched rows) |
| Save â€” disabled | "Save template" disabled while subject or body is empty |
| Save â€” in-flight | "Savingâ€¦" + `Loader2`, controls disabled, no double-submit |
| Save â€” success | Success toast "Template saved."; the editor keeps its content (the user may continue toward the test-send step in Phase 5) |
| Save â€” field-invalid | Field-anchored `FormMessage`: "Add a subject before saving." / "Write a message before saving." (clears when the field becomes non-empty) |
| Save â€” server failure | Destructive `Alert`; the composed subject/body are NOT lost; the user can retry |

Interaction rules:
- The "Save template" button disables while empty and while in flight; no double-submit (mirrors the CSV uploader's save guard and the SMTP wizard's verify/test-send rule).
- Errors never clear silently â€” field-anchored save errors clear when the field becomes valid; a destructive save-failure Alert clears on the next save attempt.
- The merged preview and the per-row empty-value highlight run CLIENT-side via `fillMessage` / `analyzeMerge` over rows the SERVER fetched once (fetch-all-once, step client-side â€” 04-RESEARCH.md A4). papaparse never ships to the browser; only parsed row objects cross the wire.
- The validation-report aggregates are computed over the FULL fetched row set â€” never a 5-row sample â€” so the report can't say "0 missing" while a real gap exists (04-RESEARCH.md Pattern 3, Pitfall 4). The invalid-email count is the server's authoritative value.
- Merged output renders as escaped text (JSX / `whitespace-pre-wrap`); a CSV cell value is never injected as HTML (stored-XSS defense) and no CSV cell value is logged.
- Unknown tokens are surfaced as an authoring warning but do NOT hard-block Save â€” saving a template is recoverable and the hard confirm-before-send gate lives in Phase 5.
- The recipient set feeding the preview is resolved server-side from a userId-scoped lookup (the client sends only a `recipientSetId`, never a storage path) â€” IDOR + path-traversal defense (04-RESEARCH.md Pitfall 3).
