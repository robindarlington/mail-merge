## Phase Goal

**As a** signed-in user composing a merge email, **I want to** step through my message rendered against each real CSV row and read a pre-send validation report, **so that** I can prove the merge is correct — and catch blank values, typos, and invalid emails — before sending anything.

<objective>
Second vertical slice of Phase 4: add the live merged preview + pre-send validation report to the compose surface. Selecting a recipient list calls Plan 03's `previewCampaign` to fetch the parsed rows + the template-INDEPENDENT server-authoritative fields (invalid-email count + resolved email column) once; the new `preview-stepper.tsx` steps through rows client-side, rendering each row's merged subject/body via `fillMessage` and its "To:" line via the server-resolved `emailColumn`, highlighting rows with empty merge values via `analyzeMerge`, and computing the template-DEPENDENT report aggregates (unknown tokens, rows-with-empty-values) CLIENT-side and reactively over ALL fetched rows as the user types. A validation-report card surfaces unknown tokens (authoring warning), the invalid-email count, and the missing-values aggregate.

Purpose: PREV-01 (step merged rows vs real data), PREV-02 (highlight empty merge values), PREV-03 (aggregate validation report), and EDIT-03 (subject IS personalized in the render) — the phase's "prove it before you send" payoff. Computing the template-dependent aggregates client-side keeps them in sync with the composed template (no stale-at-fetch divergence).
Output: `components/compose/preview-stepper.tsx` (new) + `compose-editor.tsx` wiring (modified).
</objective>

import { previewCampaign } from "@/lib/compose/actions";
// previewCampaign(fd: FormData): Promise<{ ok:true; data: PreviewReport } | { ok:false; error: ActionError }>
type PreviewReport = { columns: string[]; rows: Record<string,string>[]; totalRows: number;
  emailColumn: string | null; invalidEmailCount: number; };
// fd carries: recipientSetId  (server resolves the path + invalidEmailCount + emailColumn — it never sees the composed template)
// emailColumn is the SERVER-resolved To: column (row.email_column ?? detectEmailColumn) — the SAME column used for invalidEmailCount.
// The client renders "To: {rows[i][report.emailColumn]}" from it; it NEVER re-detects the column. Null → render no "To:" line.
// unknownTokens + rowsWithEmptyValues are NOT in PreviewReport — they depend on the composed subject/body and are computed
// CLIENT-side (below) from report.rows via analyzeMerge, reactively as subject/body change, so they never go stale.

<!-- Browser-safe pure engine (Plan 01 + existing), NO deps — safe in a client component: -->
import { fillMessage, analyzeMerge } from "@/lib/core";
// fillMessage({ subject, body }, row) => { subject, body }
// analyzeMerge(subject+"\n"+body, row, columns) => { empty: string[]; unknown: string[] }
// unknownTokens (template-level, column-set dependent): union of analyzeMerge(...).unknown for the current subject+body.
// rowsWithEmptyValues: count of rows where analyzeMerge(subject+"\n"+body, row, columns).empty.length > 0, over ALL rows.
