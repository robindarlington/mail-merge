## Phase Goal

**As a** signed-in user composing a merge email, **I want to** step through my message rendered against each real CSV row and read a pre-send validation report, **so that** I can prove the merge is correct — and catch blank values, typos, and invalid emails — before sending anything.

<objective>
Second vertical slice of Phase 4: add the live merged preview + pre-send validation report to the compose surface. Selecting a recipient list calls Plan 03's `previewCampaign` to fetch the parsed rows + server-authoritative validation aggregate + the resolved email column once; the new `preview-stepper.tsx` steps through rows client-side, rendering each row's merged subject/body via `fillMessage` and its "To:" line via the server-resolved `emailColumn`, and highlighting rows with empty merge values via `analyzeMerge`. A validation-report card surfaces unknown tokens (authoring warning), the invalid-email count, and the missing-values aggregate.

Purpose: PREV-01 (step merged rows vs real data), PREV-02 (highlight empty merge values), PREV-03 (aggregate validation report), and EDIT-03 (subject IS personalized in the render) — the phase's "prove it before you send" payoff.
Output: `components/compose/preview-stepper.tsx` (new) + `compose-editor.tsx` wiring (modified).
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CSV cell values → merged preview render | Untrusted CSV content is displayed to the author |
| browser → previewCampaign action | Client sends recipientSetId (+ subject/body); server owns path + counts + emailColumn |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-4-XSS | Tampering | preview-stepper merged render | mitigate | merged output rendered as escaped JSX text in whitespace-pre-wrap; never dangerouslySetInnerHTML (Task 1, grep-gated) |
| T-4-IDOR | Information Disclosure | previewCampaign call | mitigate | client passes only recipientSetId; server resolves the path via userId-scoped lookup (Plan 03) — never a client path |
| T-4-DIVERGE | Repudiation/Integrity | validation report + To: column | mitigate | report aggregates AND emailColumn come from server props (computed/resolved over ALL rows); client never recomputes counts or re-detects the To: column from a sample (Task 1/2) |
| T-4-LOG | Information Disclosure | client preview | mitigate | no console logging of CSV cell values / merged content |
| T-4-SC | Tampering | npm installs | accept | zero new npm dependencies (fillMessage/analyzeMerge are in-repo, browser-safe) |
</threat_model>
