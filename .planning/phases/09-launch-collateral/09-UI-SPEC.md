# Phase 9 — UI Design Contract

> Visual and interaction contract for the four public signed-out routes, the site-wide attribution footer, and the repo-facing collateral (README + write-up). Generated by gsd-ui-researcher (autonomous mode), verified by gsd-ui-checker.

This phase is a **packaging + presentation** slice, not a feature build (09-RESEARCH.md). It adds four public, signed-out **marketing/docs routes** inside the *existing* Next.js 16 app — a niche-framed landing at `/`, `/docs`, `/self-host`, and `/agents` — under a new `(marketing)` route group with its own layout (header + `SiteFooter`). It flips `HIRE_ME_URL` from the `example.com` placeholder to the live destination (BRAND-01), and ensures the already-built `SiteFooter` renders on the public pages too. It also produces two repo-facing Markdown artifacts (root `README.md`, `docs/writeup.md`) and captures screenshots into `docs/screenshots/`. **No new runtime dependencies** (locked in CONTEXT); the work is composition and copy, not construction.

It inherits the design system, spacing, typography, and color contract established in **02-UI-SPEC.md** and carried verbatim through **03-/04-/05-/06-/07-UI-SPEC.md**. There is **no new palette, no new font, no new type scale.** The marketing pages are members of the *existing* design system — distinctiveness comes from layout and copy, never from new fonts or colors (09-RESEARCH.md Pitfall 5).

**This phase has NO destructive actions.** Every surface is a read-only public page or a repo Markdown file. There is no data entry, no deletion, no send. The lone state change is a single-value code constant flip (`HIRE_ME_URL`).

This is a deliberate, documented departure from the 640px app column (self-resolved decision D-1 below), justified because marketing/docs content is read at rest, not operated. Do not exceed `max-w-5xl`.

## Spacing Scale

Inherited verbatim from 02-UI-SPEC.md (8-point scale, multiples of 4). No new spacing tokens this phase.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps in feature rows and header nav |
| sm | 8px | Gap between a heading and its immediate sub-label; badge internal padding (library default) |
| md | 16px | Default gap between prose paragraphs, list items, and card internals (`gap-4`) |
| lg | 24px | Card padding (`p-6`, library default); gap between feature cards in a grid (`gap-6`); header horizontal padding (`px-6`) |
| xl | 32px | Gap between major page sections within a marketing page (`gap-8` / `space-y-8`); header height area |
| 2xl | 48px | Vertical rhythm between hero and the section beneath it (`py-12` hero block); section top/bottom padding on long pages |
| 3xl | 64px | Landing top hero vertical padding on desktop (`py-16` ≈ 64px) — the one page-level breathing space above the fold |

Exceptions:
- shadcn primitive internals (button heights, `Card p-6`, `Badge px-2 py-0.5`, `Separator`) follow library defaults and are exempt.
- Marketing header height: `h-16` (64px) matches the 09-RESEARCH.md Pattern 3 example and the app top bar — a fixed component dimension, not a new token.
- Marketing reading-column max-widths (`max-w-3xl` / `max-w-5xl`) are container constraints, not spacing tokens (see Design System → Layout width, D-1).

## Typography

Inherited verbatim from 02-UI-SPEC.md. Exactly **4 sizes, 2 weights** (400 regular, 600 semibold). No monospace face introduced — code snippets render in Geist sans inside a `bg-muted` block (self-resolved D-2: a mono font would be a new dependency; `font-mono` maps to the browser default stack, acceptable for code blocks only — see note). No new type scale (09-RESEARCH.md Pitfall 5).

| Role | Size | Weight | Line Height | Marketing usage |
|------|------|--------|-------------|-----------------|
| Body | 16px (`text-base`) | 400 | 1.5 | All prose, list items, card body, docs steps |
| Label | 14px (`text-sm`) | 400 | 1.5 | Footer text, captions, code-block contents, muted metadata, badge text (via primitive) |
| Heading | 20px (`text-xl`) | 600 | 1.2 | Section headings on every marketing page, card titles |
| Display | 28px (`text-[28px]`) | 600 | 1.2 | The landing **hero headline** — the single Display use on the marketing surface (mirrors the auth-page precedent that Display is reserved for one focal line per surface) |

- **Hero headline** uses Display (28px, weight 600). On large viewports it MAY scale up responsively to `text-4xl`/`text-5xl` via a `sm:`/`lg:` breakpoint (self-resolved D-3) — this is a *responsive scale of the single Display role*, not a new token; keep the mobile/base value at 28px so the declared scale holds. Do not introduce a second headline size at the same breakpoint.
- Page section headings use Heading (20px, `text-xl`, 600), matching every prior page heading in the app.
- **Code-block note:** snippets on `/agents`, `/self-host`, `/docs` render at Label size (14px, `text-sm`) inside `bg-muted rounded-md p-4`. `font-mono` (browser default monospace stack) is permitted **inside code blocks only** for legibility of commands/JSON — it is not a font *package* and adds no dependency. Prose never uses mono.
- **Accepted library-default exceptions:** `Badge` ships `text-xs font-medium` (12px / weight 500); `Button` ships `font-medium` (500). These are shadcn primitive defaults — accepted as-is exactly as recorded in prior UI-SPECs. Do NOT introduce 12px or weight-500 in hand-written classes.

## Color

All values are existing `app/globals.css` tokens (oklch). Reference by semantic name, never raw values. **No new palette tokens this phase.** Marketing pages are neutral-dominant with a single accent — the exact same one-accent discipline as the app.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--background` oklch(1 0 0) | Page background, hero surface, prose surface |
| Secondary (30%) | `--card` / `--muted` / `--border` / `--secondary` | Feature/niche `Card` surfaces, code-block backgrounds (`bg-muted`), section dividers (`border`/`Separator`), header bottom border, `secondary` badges |
| Accent (10%) | `--primary` oklch(0.205 0 0) | See reserved list below |
| Destructive | `--destructive` oklch(0.577 0.245 27.325) | **Not used** this phase — there are no destructive actions on any marketing page. Listed for completeness only |

Accent (`--primary`) reserved for:
1. **The single primary CTA per marketing page.** On the landing that is the **"Get started"** primary `Button` (leads to `/sign-up`). One accent button above the fold; a second identical CTA MAY repeat once at the page foot — still the same single action, not a new accent.
2. **The "Sign in" affordance** in the marketing header is `variant="outline"` (neutral), NOT accent — the header's sign-in is secondary to the page's "Get started" CTA (self-resolved D-4).
3. Focus rings (via `--ring`) on all links, buttons, and the header wordmark.
4. Inline text links within prose use the existing underline-on-neutral idiom (`underline underline-offset-4 hover:text-foreground`, exactly as `SiteFooter` does) — **links are neutral text, not accent-colored** (self-resolved D-5, matches the footer precedent).

Everything else uses `secondary` / `ghost` / `outline` / neutral text. **Never style all interactive elements with accent** — a docs page with 20 links must not render 20 accent-colored links; they are neutral underlined text. Feature-list icons are `text-muted-foreground` (decorative), never accent.

## Accessibility & Responsive

- **Contrast:** all text uses existing tokens (`foreground` on `background`, `muted-foreground` for secondary) which already meet the app's established contrast bar; no new low-contrast pairings.
- **Focus:** every link/button keeps the `--ring` focus indicator; the marketing header wordmark is a focusable `Link`.
- **Landmarks:** marketing layout uses `<header>`, `<main>`, `<footer>` (the last via `SiteFooter`) — one `<main>` per page.
- **Responsive:** single-column stack on mobile; niche cards and feature grid go 2-up at `sm:`/`md:`. Header collapses the wordmark + sign-in into a single row (no hamburger needed — only one nav action). Hero headline scales per D-3.
- **Images:** every screenshot/`img` needs descriptive `alt`; README images use relative paths so they render on GitHub (09-RESEARCH.md Pitfall 6).
